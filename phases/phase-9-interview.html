📅 Week 2 — .NET Theory Q&A Bank

7 C# / OOP Q&A (memorize these answers)

Q. Value vs Reference Type?

Value Type:

  • Hold actual data inside variable
  • Examples: int, double, bool, struct, DateTime
  • Stored on stack
  • Copied by value — change in one not affect other

Reference Type:

  • Hold pointer to data on heap
  • Examples: class, string, array
  • Both names point to same object
  • Change visible to all references

Note: string is reference type but immutable — behave like value.

Java parallel: Same like primitive (int) vs Object — but C# has struct.

Q. Boxing / Unboxing?

Boxing: Put value type (int) in object reference — wrap on heap.

Unboxing: Take value out of object back.

Why slow:

  • Heap allocation cost CPU and memory
  • GC has to clean later
  • Disaster in tight loops with millions iterations

Common Trap: Old ArrayList store everything as object — every int boxed.

Fix: Use generics — List<int> store ints natively, no boxing.

Java parallel: Same as Java auto-boxing (int → Integer).

Q. What is the CLR?

Definition: CLR = Common Language Runtime. Engine that run .NET code.

How it works:

  • C# compile to IL (Intermediate Language) — not machine code
  • JIT (Just-In-Time) compiler translate IL to machine code at run time

CLR handles:

  • Garbage Collection (auto clean memory)
  • Exception Handling (throw/catch flow)
  • Security (what code can do)

Other languages: F#, VB.NET also compile to IL — run on same CLR.

Java parallel: CLR = JVM. IL = Bytecode. Same idea.

Q. Stack vs Heap?

Stack:

  • Per-thread, fast, auto managed
  • Hold value types (local variables)
  • Hold method call frames (parameters, return address)
  • Hold reference variables themselves (pointer, not data)
  • Method return = stack frame wiped instantly

Heap:

  • Shared across threads, managed by GC
  • Hold reference types — every new allocate here
  • Slower than stack — find free space, bookkeeping
  • GC scan and reclaim memory periodically

Java parallel: Same Stack vs Heap concept as Java.

Q. Struct vs Class?

Class:

  • Reference type, live on heap
  • Support inheritance
  • Can be null
  • Copy by reference

Struct:

  • Value type, live on stack
  • NO inheritance (implicitly sealed)
  • Can implement interfaces
  • Copy by value

Use struct for:

  • Small immutable value-like data: Point(x,y), Money
  • Examples in BCL: int, DateTime, TimeSpan, Guid
  • Keep size under 16 bytes

Java parallel: Java has no struct. All custom types are class.

Note: For 99% of business code, use class.

Q. Abstract class vs Interface?

Interface:

  • Pure capability — "this type can do X"
  • Method/property signatures, no implementation (mostly)
  • Class can implement MANY interfaces
  • Examples: IDisposable, IEnumerable

Abstract Class:

  • Partly-built parent class
  • Can have fields, properties, constructors, real methods
  • Force children to fill abstract members
  • Class can inherit only ONE

Rule: Interface for contract. Abstract class for shared code/state.

Java parallel: Same exactly as Java abstract class vs interface.

Q. Explain SOLID?

S — Single Responsibility: One class, one reason to change. Don't combine user storage + email + log.

O — Open/Closed: Open for extension, closed for modification. Add behavior via inheritance, not editing.

L — Liskov Substitution: Child class work where parent expected. Don't make Penguin crash on Fly().

I — Interface Segregation: Many small interfaces beat one big. Don't force class to implement methods it not need.

D — Dependency Inversion: Depend on abstraction (interface), not concrete class. Why .NET DI exists.

Java parallel: Same SOLID rules apply in Java also.

Q. Delegate vs Event?

Delegate:

  • Type-safe function pointer
  • Hold method reference
  • Invoke like function
  • Built-in: Action, Func, Predicate

Event:

  • Delegate with extra safety
  • Only declaring class can INVOKE
  • Outside code can only SUBSCRIBE (+=) or UNSUBSCRIBE (-=)
  • Enforce publisher-subscriber pattern

Common Bug: Forget to unsubscribe — subscriber stay in memory (leak).

Java parallel: Like Observer pattern interface (PropertyChangeListener).

Q. Func vs Action vs Predicate?

Action:

  • Method that return void
  • Action = no args
  • Action<int> = one int
  • Up to 16 type params

Func:

  • Method that return value
  • LAST type param = return type
  • Func<int> = no args, return int
  • Func<int, int, bool> = 2 ints, return bool

Predicate<T>:

  • Shortcut for Func<T, bool>
  • Used in List<T>.Find, List<T>.RemoveAll

Example: list.Where(x => x > 5) — lambda is Func<int, bool>.

Q. What is a closure?

Definition:

  • Lambda capture variable from outer scope
  • Lambda carry access even after method return

Important: Captures by REFERENCE, not by value.

Classic Bug in for loop:

for (int i = 0; i < 5; i++)
    actions.Add(() => Console.WriteLine(i));
// All print 5 (final value of i)

Fix: Copy loop variable to local:

for (int i = 0; i < 5; i++) {
    var local = i;
    actions.Add(() => Console.WriteLine(local));
}

Java parallel: Same lambda capture issue in Java (variable must be effectively final).

Q. async/await internals?

Definition: Way to write async code that look sync.

Keywords:

  • async mark method as async
  • await pause until Task complete

Under the Hood:

  • Compiler rewrite method as state machine class
  • Each await = checkpoint
  • Method pause, capture local state, return
  • Thread released back to thread pool
  • When Task complete — continuation restore state and resume

Why Important:

  • ASP.NET Core serve thousands users with few threads
  • No thread blocked waiting for I/O

Rule: "async all the way" — caller must also be async.

Java parallel: Like CompletableFuture but cleaner syntax.

Q. Why never .Result on a Task in web?

Problem: Cause deadlock OR kill scalability.

Why Deadlock:

  • In classic ASP.NET, request run in sync context (one thread at time)
  • .Result block thread waiting for Task
  • Task continuation need same context to finish
  • You blocked the only thread — both wait forever

Even in ASP.NET Core:

  • Block thread that could serve another request
  • Under load — thread pool run out
  • Everything slow down

Fix: "async all the way" — make caller async, use await.

Q. IEnumerable vs IQueryable?

IEnumerable<T>:

  • In-memory sequence
  • Operations run in app process (LINQ to Objects)

IQueryable<T>:

  • Build expression tree
  • Provider (EF) translate to SQL
  • Filter happen at database

Performance Example:

  • Good: db.Products.Where(...).ToList() — SQL WHERE
  • Bad: db.Products.ToList().Where(...) — load all rows first

Rule: Stay on IQueryable as long as possible. Call .ToList() at very end.

Q. Deferred vs Immediate execution?

Deferred (build query, not run):

  • Where, Select, OrderBy, Take
  • Run only on enumeration (foreach, terminal operator)

Immediate (run now):

  • ToList, ToArray, ToDictionary
  • Count, Sum, First, Single, Any

Big Trap:

  • Re-enumerate same query = run again (DB hit again)
  • If iterate twice — call .ToList() first to cache

Note: Closures inside deferred queries capture by reference.

Q. throw vs throw ex?

throw; — re-throw current exception, KEEP original stack trace.

throw ex; — re-throw, RESET stack trace (lose original location).

Rule: Always use bare throw; alone.

To Wrap Exception:

throw new MyException("message", ex);
  • Original passed as inner exception — preserve chain

Quick Answer: "throw ex; destroy stack trace — almost always bug."

Java parallel: Java throw e; always keep stack — no this problem.

Q. What is IDisposable?

Definition:

  • Interface with one method Dispose()
  • Clean resources at right time (not wait for GC)

Why need:

  • For resources GC don't manage well — file handle, DB connection, socket, lock
  • Hit limits (connection pool full) — app stop working

How to Use — using statement:

using var stream = File.OpenRead(path);
  • Auto call Dispose at end of scope
  • Even if exception thrown
  • Cleaner than try/finally

Java parallel: IDisposable = AutoCloseable. using = try-with-resources.

Modern: IAsyncDisposable with await using.

Q. GC generations?

3 Generations:

  • Gen 0 — new objects (small, scanned often, fast)
  • Gen 1 — survived one GC (medium frequency)
  • Gen 2 — long-lived (static cache, singleton) — scanned rarely (expensive)

Why this design:

  • Most objects die young
  • Gen 0 small and quick to clean
  • Gen 2 large and slow — clean rarely

LOH (Large Object Heap):

  • Objects ≥ 85KB
  • Treated like Gen 2
  • Not compacted by default — can fragment

Java parallel: Gen 0/1 = Young Gen (Eden + Survivor). Gen 2 = Old Gen (Tenured).

Note: Never call GC.Collect() in production.

Q. readonly vs const?

const:

  • Compile-time constant
  • Value must be known at compile time
  • Only primitive type or string
  • Baked into every assembly that reference it
  • Catch: change in library = need recompile users

readonly:

  • Run-time constant
  • Set in field declaration OR constructor
  • Any type (including reference type)
  • Field reference cannot be reassigned

Note: readonly not make OBJECT immutable. readonly List — can still list.Add(x).

Java parallel: const = Java final primitive. readonly = Java final object reference.

Q. What is a record?

Definition:

  • Special class for immutable data (C# 9+)
  • Concise syntax: public record User(int Id, string Name);

Built-in features:

  • Value-based equality — equal if all properties equal
  • Auto ToString() print all properties
  • Deconstructor: var (id, name) = user;
  • with expression: var modified = user with { Name = "New" }
  • Default immutable (init-only properties)

Use for:

  • DTOs, value objects, data-transfer scenarios

Note: record struct (C# 10) for value-type records.

Java parallel: Java 14+ also has record with same idea.

Q. Generics with constraints?

Definition:

  • Generics let code work with any type
  • Constraints restrict T to types that meet rules
  • Use where keyword

Common Constraints:

  • where T : class — must be reference type
  • where T : struct — must be value type
  • where T : new() — must have parameterless constructor
  • where T : IComparable<T> — must implement interface
  • where T : SomeBaseClass — must inherit from class
  • where T : notnull — non-nullable

Combine multiple:

where T : class, IEntity, new()

Java parallel: Java <T extends Comparable> — same idea, different syntax.

8 ASP.NET / EF Core Q&A

Q. Middleware order matters — example?

Definition: Pipeline run in exact order of app.UseXxx() calls.

Standard Correct Order:

  1. UseRouting — endpoint matching first
  2. UseCors — CORS check
  3. UseAuthentication — WHO is user?
  4. UseAuthorization — WHAT they can do?
  5. MapControllers — call controller

Critical Rule:

  • UseAuthentication MUST come before UseAuthorization
  • Need to know WHO before checking WHAT
  • Flip them = authorization see no user, reject everything

Note: Custom exception middleware near start. Forget UseStaticFiles = wwwroot files don't serve.

Q. Singleton, Scoped, Transient — when?

Compare Table:

LifetimeCreatedUse For
SingletonOnce for appConfig, cache, logger, HttpClient
ScopedOnce per requestDbContext, request-bound services
TransientEvery injectionLightweight stateless helpers

Captive Dependency Bug (interview trap):

  • Don't inject Scoped (DbContext) into Singleton
  • Scoped get captured forever
  • Break request isolation, not thread-safe
  • Use ValidateScopes in dev to catch
Q. What does [ApiController] do?

Definition: Attribute on controller class — enable modern Web API conventions automatically.

Features:

  • Auto model validation — invalid JSON return 400 before action run
  • Binding source inference — complex type = [FromBody], simple = [FromRoute]/[FromQuery]
  • ProblemDetails errors — RFC 7807 standard JSON
  • Force attribute routing — no conventional routing

Note: Almost every new ASP.NET Core API controller should have it.

Q. How to do JWT auth in ASP.NET?

3 Steps:

Step 1 — Register in Program.cs:

  • builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
  • .AddJwtBearer(opts => { opts.TokenValidationParameters = ... })
  • Set: ValidateIssuer, ValidateAudience, ValidateLifetime, IssuerSigningKey
  • builder.Services.AddAuthorization()

Step 2 — Add to Pipeline:

  • app.UseAuthentication();
  • app.UseAuthorization();
  • In that order, before MapControllers

Step 3 — Token Generation:

  • Create Claims (sub, email, role)
  • Make JwtSecurityToken with issuer/audience/expiry/signing
  • Use JwtSecurityTokenHandler.WriteToken()

Note: Add refresh token in DB for revocation.

Q. Authentication vs Authorization?

Authentication: WHO are you? (verify identity).

Authorization: WHAT can you do? (check permission).

AspectAuthenticationAuthorization
QuestionWho are you?What can you do?
MethodUsername/password, JWTRoles, policies, claims
Fail status401 Unauthorized403 Forbidden
MiddlewareUseAuthentication()UseAuthorization()

Real Example: Show ID at gate = auth. Gate check 5th floor access = authz.

Note: UseAuthentication MUST come before UseAuthorization.

Q. How to handle exceptions globally?

Wrong Way: try/catch in every controller — repetitive, easy miss.

Right Way — Custom Middleware:

  • Write ExceptionHandlingMiddleware
  • Wrap call to next() in try/catch
  • Catch unhandled exceptions
  • Log with full context (path, user, correlation ID)
  • Map exception to HTTP status (NotFoundException → 404, ValidationException → 400, else → 500)
  • Return ProblemDetails JSON (RFC 7807)

.NET 8 Cleaner Way:

  • Implement IExceptionHandler
  • AddExceptionHandler<T>()
  • app.UseExceptionHandler()
  • builder.Services.AddProblemDetails()
Q. What is CORS?

Definition: CORS = Cross-Origin Resource Sharing. Browser security blocking JS calling API on different origin.

When needed:

  • Frontend on different origin than backend
  • Common: Angular on localhost:4200 calling .NET on localhost:5000

Configure (.NET):

  • In services: AddCors with policy
  • Specify WithOrigins, AllowAnyHeader, AllowAnyMethod
  • Optional: AllowCredentials for cookies
  • In pipeline: app.UseCors("PolicyName")

Note: Cannot combine AllowAnyOrigin() + AllowCredentials() — forbidden by spec.

Q. IActionResult vs ActionResult<T>?

IActionResult:

  • Base interface — Ok(), NotFound(), BadRequest()
  • Flexible
  • Swagger don't know data shape

ActionResult<T>:

  • Generic version (newer .NET)
  • Return either status code OR typed data
  • Better Swagger documentation
  • Stronger typing

Best Practice:

  • Use ActionResult<T> when action return data
  • Use IActionResult when only status (204 NoContent on delete)
Q. What is filter and types?

Definition: Cross-cutting code inside MVC pipeline. Run AFTER routing identify action, AROUND action execution.

Different from Middleware: Middleware run in broader HTTP pipeline.

5 Filter Types (execution order):

  1. Authorization — first, allow or deny
  2. Resource — wrap everything else, used for caching
  3. Action — before and after action method
  4. Exception — catch exceptions from actions or filters
  5. Result — wrap result execution

When to use:

  • Filter — controller-specific behavior ([ServiceFilter] for action logging)
  • Middleware — apply to all requests including static files

Note: Filter MVC-aware (access controller context). Middleware see only HttpContext.

Q. Explain N+1 in EF?

Definition: 1 query for parent + N queries for navigation = 1+N queries.

Example:

  • Load 100 orders = 1 query
  • Loop and access order.Customer.Name = 100 queries
  • Total = 101 queries
  • 50 requests/sec = 5050 queries — DB fall over

Fix 1 — Eager Loading:

db.Orders.Include(o => o.Customer).ToListAsync()

Fix 2 — Projection (BEST):

.Select(o => new { o.Id, CustomerName = o.Customer.Name })

Note: Always check generated SQL with .ToQueryString().

Q. Tracking vs NoTracking?

Tracking (default):

  • EF track every entity for change detection
  • Cost: extra memory + CPU
  • Worth for UPDATE scenarios

NoTracking:

  • db.Products.AsNoTracking().ToListAsync()
  • Skip change tracker
  • 20-50% faster, less memory
  • For read-only / GET endpoints

Rule: AsNoTracking on every GET. Tracking when modify and save.

Global default: UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking)

Q. EF Core vs Dapper?

EF Core for: 80% of CRUD work, productivity matter more than 5% perf.

Dapper for: Hot read paths, heavy reporting queries, complex SQL LINQ can't express.

Note: Many real projects use BOTH — EF for writes, Dapper for read-optimized.

Java parallel: EF Core = Hibernate. Dapper = JDBC with light wrapper.

Q. Eager vs Lazy vs Explicit loading?

Eager (BEST):

  • Use .Include() — fetch related data in same query (JOIN)
  • You control what loads

Lazy (AVOID in web):

  • Load on first access
  • Need Microsoft.EntityFrameworkCore.Proxies + virtual nav properties
  • Cause N+1 in loops
  • Break JSON serialization (infinite recursion)

Explicit:

  • Manual: db.Entry(order).Reference(o => o.Customer).LoadAsync()
  • You control timing

Best Practice: Eager with .Include() OR .Select() projection into DTOs.

Q. How does EF Concurrency work?

Approach: Optimistic Concurrency (assume conflicts rare).

Setup:

  • Add RowVersion column with [Timestamp]
  • SQL Server auto-increment on every UPDATE
  • EF include in WHERE: WHERE Id = @id AND RowVersion = @original

If row changed in between:

  • UPDATE affect 0 rows
  • EF throw DbUpdateConcurrencyException

How to Handle:

  • Retry operation
  • Merge conflict by re-fetching
  • Show user "modified by someone else"
  • Pick winner (last-write-wins)

Note: Optimistic > pessimistic for scale (no DB locks).

Q. What is CQRS?

Definition: CQRS = Command Query Responsibility Segregation.

  • Separate code for writes (commands) from code for reads (queries)

Commands (Write):

  • CreateOrder, UpdateProduct, DeleteUser
  • Dedicated handler classes
  • Validation + business rules
  • Don't return data, just success/failure

Queries (Read):

  • GetOrder, ListProducts
  • Own handlers
  • Read from optimized projections (denormalized or cached)

Benefits:

  • Reads and writes scale independently
  • Different complexity profiles

.NET Tool: MediatR library — mediator.Send(command)

Note: CQRS not always separate DBs — can be just code pattern.

Q. Why use DTOs?

Definition: DTO = Data Transfer Object. Separate class for data sent over wire.

4 Big Reasons:

  • Security — User entity has PasswordHash, don't leak it
  • Over-posting protection — client cannot send {"isAdmin": true} sneakily
  • Decoupling — DB schema change without breaking API
  • Payload size — include only needed fields

Tools: AutoMapper — map Entity to DTO automatically.

Java parallel: Same DTO pattern. Java POJO = .NET POCO.

Q. How to make API stateless?

Definition: Stateless = no per-user data in memory between requests.

Benefits:

  • ANY server can handle ANY request
  • Trivial horizontal scaling
  • Server crash = no user state lost

How to achieve:

  • JWT for auth (no server sessions)
  • Token contain user identity + claims
  • Shared state in Redis or DB (not in-memory)
  • Don't store in static fields
  • Build idempotent operations (safe to retry)

Result: Spin up 10 more instances behind load balancer with zero code change.

Q. What is gRPC and when to use?

Definition:

  • High-performance RPC framework by Google
  • HTTP/2 transport
  • Protobuf binary serialization (not JSON)

Advantages over REST:

  • Faster (binary, smaller payloads)
  • Strongly typed (no JSON guessing)
  • Streaming support (server, client, bidirectional)

Trade-offs:

  • Not browser-friendly (need proxy)
  • Harder to debug than JSON
  • Need .proto file shared client/server

Use for: Internal microservice-to-microservice communication.

NOT for: Public APIs facing browsers — stick with REST or GraphQL.

Q. What is SignalR?

Definition: .NET's real-time communication framework.

Transport (auto pick best):

  • WebSockets (preferred)
  • Server-Sent Events
  • Long polling (fallback)

Use Cases:

  • Chat applications
  • Live notifications
  • Real-time dashboards (live order tracking)
  • Collaborative editing
  • Multiplayer games
  • Sports/stock tickers

Setup:

  • Server: create Hub class
  • Angular: @microsoft/signalr + HubConnection
  • Auth: JWT via accessTokenFactory

Scale: Use Redis backplane — message from one server reach clients on other servers.

Note: Easy "wow factor" feature for portfolio projects.

Q. Health checks — why and how?

Definition: Endpoint like /healthz that report if app + dependencies are working.

Why needed:

  • Load balancers / orchestrators ping endpoint
  • If fail — stop routing traffic to that instance
  • Prevent users hitting dead server

Setup (one line):

app.MapHealthChecks("/healthz")

Add Dependency Checks:

builder.Services.AddHealthChecks()
    .AddDbContextCheck<AppDbContext>()
    .AddRedis(...)
    .AddUrlGroup(new Uri("https://api.payment.com/status"))

Best Practice — TWO Endpoints:

  • /healthz/live — liveness (is process alive?). Failure = restart pod
  • /healthz/ready — readiness (dependencies ready?). Failure = stop traffic, no restart

📅 Week 3 — System Design + Behavioral + Resume

9 System Design Round Framework

For 12-15 LPA, expect 1 system design round. They want to see structure, tradeoffs, and conversation — not perfection.

🎯 The 6-Step Framework (use every time)

  1. Clarify requirements (2 min) — functional + non-functional. Ask about scale, latency, availability.
  2. Estimate scale (2 min) — QPS, storage, bandwidth (back-of-envelope).
  3. High-level design (5 min) — boxes and arrows. Client → LB → API → Cache → DB.
  4. Deep dive (10 min) — pick 1-2 components: schema, sharding, caching strategy, queue.
  5. Identify bottlenecks (5 min) — what fails first under load? How to fix?
  6. Wrap up (2 min) — tradeoffs, what you'd do differently with more time.

🎯 Practice These Designs

  • URL Shortener (bit.ly)
  • Pastebin
  • Twitter feed (timeline)
  • Instagram (basic photo sharing)
  • WhatsApp / Chat app
  • Uber dispatch
  • Rate Limiter
  • Notification service
  • YouTube (video upload + streaming)
  • E-commerce checkout (your Project 1 — be the expert!)

10 Behavioral / HR Round (STAR)

Underrated phase. 50% of rejections at 12-15 LPA are behavioral, not technical. Prepare 6-8 stories using STAR.

🎯 STAR Format

  • Situation — context (1 sentence)
  • Task — what you needed to do (1 sentence)
  • Action — what YOU did, specifically (3-4 sentences, the meat)
  • Result — measurable outcome (1-2 sentences)

🎯 Stories You MUST Have Ready

  1. Tell me about yourself — 90 sec max: present role → biggest project → why .NET → what you're seeking
  2. Why are you switching? — pull (growth, full-stack ownership, .NET excitement), not push
  3. Toughest bug you fixed — investigative process, not just the fix
  4. Conflict with a teammate — disagreement → discussion → resolution → outcome
  5. Time you missed a deadline — own it, what you learned, what you do differently now
  6. Best feedback you received — specific, what you changed
  7. Time you took initiative — proactive ownership of something nobody asked for
  8. Why this company? — research the company, mention 2-3 specifics

🎯 Questions to Ask Them

  • What does success look like for this role in 6 months?
  • What's the team's biggest technical challenge right now?
  • How is the engineering culture — code reviews, on-call, learning time?
  • What does the deployment cadence look like?
  • What does career growth look like beyond this role?

Avoid: "How is work-life balance?" (perceived as low effort), salary in first round, anything you could've Googled.

11 Resume & LinkedIn Polish

🎯 Resume Rules

  • 1 page for < 5 years experience. Period.
  • Plain ATS-friendly format (no fancy graphics). Save as PDF.
  • Top: Name, role headline, contact, GitHub, LinkedIn, location, portfolio URL
  • Skills section: group as Languages / Frameworks / Cloud / Databases / Tools
  • Experience bullets: STAR-compressed with numbers
  • Projects section linking to live deployments + repos
  • Education at the bottom

🎯 LinkedIn Tweaks

  • Headline: "Full Stack Developer · .NET / Angular / Azure" (keywords for recruiters)

  • Featured section: pin your best project
  • Skills: add the .NET stack, Azure, EF Core etc. (recruiter search)
  • Open to work: enable for recruiters only
  • Post weekly: "Built X this week — here's what I learned" (algorithm rewards recency)

12 Application & Negotiation Strategy

🎯 Where to Apply (Indian market for .NET 12-15 LPA)

  • LinkedIn Jobs — primary. Set alerts for ".NET Full Stack" / "Angular .NET"
  • Naukri — large Indian database, set premium for visibility
  • Instahyre — quality-curated, recruiter-driven
  • Hirect / Cutshort — startup-heavy
  • Company careers pages — direct application has higher conversion
  • Referrals — message ex-colleagues, college alumni at target companies (best ROI)

🎯 Target Companies (mid-tier, .NET-friendly)

  • Product mid: Zoho, Freshworks, Microsoft IDC (EAS), ThoughtWorks, Persistent Systems, Walmart Labs, Atlassian, GE Digital
  • Service top: TCS Digital, Infosys (specialist track), EPAM, Capgemini Sogeti, Cognizant Digital, Wipro Digital
  • Startups using .NET (less common but exist; check their stack on LinkedIn engineering pages)

🎯 Application Cadence

  • Apply 5-10 jobs/day — quantity matters initially
  • Track in spreadsheet (company, role, applied date, status, contact)
  • Customize 1-line opener per company (mention specific product/blog)
  • Follow up after 5 days if no response
  • Start applying from Month 7 — don't wait for "ready"

🎯 Negotiation

  • Always ask for more. Companies expect a 10-20% counter.
  • Know your number: research on AmbitionBox, Glassdoor, Levels.fyi for Indian market
  • Multiple offers = leverage. Time them within 2 weeks of each other.
  • Negotiate fixed first, then variable, then joining bonus, then ESOPs
  • Never accept on the spot — "Let me discuss with family, I'll respond in 2 days"
  • Get the offer in writing before resigning
