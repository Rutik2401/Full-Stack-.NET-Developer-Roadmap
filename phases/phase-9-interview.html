📅 Week 2 — .NET Theory Q&A Bank

7 C# / OOP Q&A (memorize these answers)

Value vs Reference Type

Value types like int, double, bool, struct, and DateTime hold the actual data inside the variable, usually on the stack. When you assign or pass them, you get a copy — changes in one place don't affect the other. Reference types like class, string, and arrays hold a pointer to data on the managed heap. When you assign or pass them, both names point to the same object — a change is visible everywhere. The interesting twist is that string is a reference type but it behaves like a value because strings are immutable. The interview bonus point is boxing: when a value type gets stored in an object variable, the runtime wraps it on the heap, which is slow. So always use generic collections like List<int> instead of non-generic ArrayList to avoid boxing in hot paths.

Boxing / Unboxing

Boxing happens when you take a value type (like an int) and store it in a reference type variable like object. The runtime allocates a new object on the heap and copies the int into it — slow because of the heap allocation and the work it creates for the garbage collector. Unboxing is the reverse — extracting the value back out, which involves a type check and another copy. The most common place this used to happen was old non-generic collections like ArrayList that store everything as object — every primitive added gets boxed. The fix is generics: List<int> stores ints natively with zero boxing. In a normal piece of code you won't notice the cost, but in tight loops running millions of iterations, boxing kills performance and triggers extra GC pressure. Modern .NET code rarely sees boxing thanks to generics, but it's a popular interview question.

What is the CLR?

The CLR (Common Language Runtime) is the engine that actually runs .NET code. Your C# code first compiles to IL — Intermediate Language — not directly to machine code. At runtime, the CLR's JIT (Just-In-Time) compiler translates IL into native machine code just before each method runs. The CLR also takes care of three important things you don't have to manage: garbage collection (it cleans up unused memory automatically), exception handling (it manages the throw/catch flow), and security (it enforces what code is allowed to do). Because the CLR handles all this, C# developers focus on business logic instead of memory management. Other languages like F# and VB.NET also compile to IL and run on the same CLR — that's how they all interoperate. The bonus point: the CLR enables features like reflection, generics, async/await, and modern garbage collection.

Stack vs Heap?

The stack and heap are two different memory areas your .NET app uses. The stack is per-thread, fast, and managed automatically. It holds value types (when they're local variables), method call frames (parameters, return addresses), and reference variables themselves (the pointer, not the data they point to). When a method returns, its stack frame is wiped instantly. The heap is shared across threads and managed by the garbage collector. It holds reference types — every new ClassName() allocates here. Heap allocation is slower than stack because it involves finding free space and bookkeeping. The GC periodically scans for objects with no references and reclaims that memory. As a developer you don't manually manage either, but understanding this helps you reason about performance — for example, why creating millions of small classes pressures the GC, while structs (value types) avoid heap allocation entirely.

What's a struct vs a class?

Both are user-defined types that hold data and methods, but they have very different behavior. A class is a reference type — it lives on the heap, supports inheritance, can be null, and is copied by reference (both names point to the same object). A struct is a value type — it lives on the stack (when local) or inline inside its containing object, does NOT support inheritance (it's implicitly sealed), can implement interfaces, and is copied by value. Use a struct only for small, immutable, value-like data — examples are Point(x, y), Money(amount, currency), or coordinates. The .NET BCL uses structs for things like int, DateTime, TimeSpan, Guid. The rules of thumb: keep struct size under 16 bytes, prefer immutability (use readonly struct), and don't use them for anything with reference semantics. C# 10 added the ability for structs to have parameterless constructors. For 99% of business code, you should use class.

Abstract class vs Interface

Both are ways to define a contract that other classes must follow, but they have different roles. An interface is pure capability — it says "this type can do X". It contains method/property signatures with no implementation (since C# 8, it can have default implementations though). A class can implement many interfaces. Examples: IDisposable for things that can be disposed, IEnumerable for things you can loop over. An abstract class is a partly-built parent class — it can have fields, properties, real constructors, and some method implementations, but it forces children to fill in the abstract members. A class can only inherit from one abstract class. The simple rule: use an interface when you only need a contract; use an abstract class when you also need shared state or shared code that all children inherit. Modern .NET tends to prefer interfaces with default implementations because they're more flexible and align with the Dependency Inversion principle.

SOLID

SOLID is five design rules that make code easier to maintain and test. S — Single Responsibility: a class should have one reason to change. If a class handles user storage AND email sending AND logging, split it into three. O — Open/Closed: code should be open for extension but closed for modification. Add new behavior through inheritance or composition rather than editing existing working code. L — Liskov Substitution: subclasses must work everywhere their parent type is used without breaking expectations. Don't make a Penguin that crashes when you call Fly() — that violates the contract. I — Interface Segregation: many small focused interfaces beat one giant interface. A class should not be forced to implement methods it doesn't need. D — Dependency Inversion: depend on abstractions (interfaces), not concrete classes. This is exactly why .NET dependency injection exists — controllers depend on IUserService, not on UserService directly, so you can swap implementations and unit-test with mocks.

Delegate vs Event

A delegate is a type-safe function pointer in C#. You define a delegate type that says "any method matching this signature", then create a variable that holds a method reference and invoke it like a function. Delegates are the basis for callbacks, LINQ, and async patterns. The built-in delegates Action, Func, and Predicate cover most cases. An event is a delegate wrapped with extra safety — only the declaring class can invoke the event, while outside code can only subscribe with += or unsubscribe with -=. This enforces the publisher-subscriber (observer) pattern at the language level. So if class A exposes an OnOrderPlaced event, only A can fire it; other classes can listen but cannot accidentally trigger it. The classic bug with events is forgetting to unsubscribe, which keeps the subscriber alive in memory — a memory leak.

Func vs Action vs Predicate

These are the three built-in generic delegate types you'll see everywhere in .NET — they cover almost every callback need. Action is for methods that return void: Action takes no arguments, Action<int> takes one int, up to 16 type parameters. Func is for methods that return a value — the LAST type parameter is always the return type: Func<int> returns an int with no args, Func<int, int, bool> takes two ints and returns a bool. Predicate<T> is a shortcut for Func<T, bool> — used in methods like List<T>.Find and List<T>.RemoveAll. Knowing these saves you from defining custom delegate types most of the time. Example: list.Where(x => x > 5) — that lambda is implicitly a Func<int, bool>.

What's a closure?

A closure happens when a lambda or anonymous method captures a variable from the surrounding scope. The lambda then "carries" access to that variable even after the original method has returned. Captures are by REFERENCE, not by value — the lambda sees the current value of the variable, not the value at capture time. The classic bug is in loops: for (int i = 0; i < 5; i++) actions.Add(() => Console.WriteLine(i)); — every lambda captures the SAME i, so when you invoke them all, they all print 5 (the value when the loop ended). The fix is to copy the loop variable into a local: var local = i; inside the loop, then capture local. Modern foreach in C# 5+ has its own scoped variable per iteration, so this trap mostly happens with for. Closures are powerful for callbacks and LINQ but be aware they keep referenced objects alive longer than expected.

async/await internals

Async/await is C#'s way to write asynchronous code that reads like synchronous code. You mark a method async and put await in front of any operation returning a Task. A Task represents work that will finish later. Under the hood, the compiler rewrites your async method into a hidden state machine class. Each await becomes a checkpoint — the method pauses, captures its local state, and returns. The thread is released back to the thread pool. When the awaited Task completes, a continuation runs that restores state and resumes from the next line. This is why async scales so well — ASP.NET Core can serve thousands of concurrent users with very few threads, because no thread sits blocked waiting for I/O. The most important rule is "async all the way" — once a method is async, every caller above should also be async and await it. Calling .Result blocks the thread and can deadlock.

Why never .Result on a Task in web?

Calling .Result or .Wait() on a Task in web code can cause a deadlock — and even when it doesn't deadlock, it kills your scalability by blocking a thread. Here's why deadlock happens: in classic ASP.NET, every request runs in a synchronization context that allows only one thread at a time. When you call .Result, the thread blocks waiting for the Task to complete. Inside the Task, the awaited continuation needs to come back to that same context to finish — but you've blocked the only available thread, so the continuation can never run. Both wait forever. Even in ASP.NET Core where the sync context issue is less severe, blocking on .Result wastes a thread that could serve another request — under load, your thread pool runs out and everything slows down. The fix is "async all the way" — make the calling method async too and use await instead. The pattern only takes one async method to ripple all the way up.

IEnumerable vs IQueryable

This is one of the most asked LINQ interview questions. IEnumerable<T> represents a sequence that lives in memory — operations on it run inside your application's process using LINQ to Objects. IQueryable<T> represents a query that builds an expression tree, which a provider (like Entity Framework) translates into the target language, usually SQL. The huge practical difference is where filtering happens. db.Products.Where(p => p.Price > 100).ToList() with EF translates to a SQL WHERE clause and only matching rows return — fast. But db.Products.ToList().Where(p => p.Price > 100) pulls EVERY product into memory then filters in C# — slow on big tables. The rule: stay on IQueryable as long as possible (Where, Select, OrderBy, Take, Skip), then call .ToList() at the very end. Calling .AsEnumerable() too early has the same trap.

Deferred vs Immediate execution

LINQ operators come in two flavors. Deferred operators like Where, Select, OrderBy, and Take don't actually run when you call them — they just build up the query. The query executes only when you start enumerating it, like in a foreach loop or by calling a terminal operator. Immediate operators force the query to run right away and return a concrete result. Examples: ToList, ToArray, ToDictionary, Count, Sum, First, Single, Any. The big trap with deferred execution: if you re-enumerate the same query, it runs again — calling the database again, fetching all data again. If you'll iterate twice, materialize first with .ToList() to cache the results. Also be careful about closures inside deferred queries — they capture variables by reference, so changing the variable later changes the query result.

throw vs throw ex

This looks tiny but is a popular interview trick question. throw; with no name re-throws the current exception while keeping the original stack trace intact — your logs still show where the error really happened. throw ex; re-throws but RESETS the stack trace, making it look like the error started at the line where you wrote throw ex;. This hides the real source of the bug and makes debugging painful. So in 99% of cases, when you catch an exception and want to bubble it up, write a bare throw;. The only legitimate case for throw ex; is when you intentionally want to discard context, which is almost never. If you want to wrap the exception in a new one, use throw new MyException("message", ex); — passing the original as inner exception preserves the chain. Quick interview answer: "Always use throw; alone. throw ex; destroys the stack trace and is almost always a bug."

What is IDisposable?

IDisposable is an interface with one method, Dispose(), that lets your class clean up resources deterministically — right when you're done with it, not whenever the garbage collector feels like running. This matters for resources the GC doesn't manage well: file handles, database connections, network sockets, lock handles. If you don't release them quickly, you can hit limits like the database connection pool maximum and your app stops working. The easiest way to use IDisposable correctly is the using statement: using var stream = File.OpenRead(path);. The compiler generates code that calls Dispose() automatically when the variable goes out of scope, even if an exception is thrown — much cleaner than try/finally. You should implement IDisposable yourself whenever your class wraps an unmanaged resource or holds another IDisposable. Modern .NET also has IAsyncDisposable with DisposeAsync() for async cleanup, used with await using.

GC generations?

The .NET garbage collector groups heap objects by age into three generations to make cleanup efficient. Gen 0 holds new objects you just created. Gen 1 holds objects that survived one GC cleanup. Gen 2 holds long-lived objects like static caches or singletons that survived many cleanups. The reason for this design is that most objects die young — they get created, used briefly, and become garbage quickly. So the GC scans Gen 0 very often, but Gen 0 is small so it's quick. Gen 1 is scanned less often, and Gen 2 is scanned rarely because it's the largest set and most expensive to clean. There's also a special area called the Large Object Heap (LOH) for objects 85KB or bigger. LOH is treated like Gen 2 and is not compacted by default, so it can fragment over time. As a developer you don't manage this — you just trust the GC. The interview bonus point: never call GC.Collect() manually in production, it interferes with the GC's tuning.

readonly vs const

Both create values that cannot be changed, but they work differently. const is a compile-time constant — its value must be known at compile time, can only be a primitive type or string, and gets baked directly into every assembly that references it. The catch: if you change a const in a library and don't recompile the apps that use it, they keep the old value. readonly is a runtime constant — its value can be set in the field declaration OR in the constructor (so it can be different per instance), and it can be any type including reference types. Once set, the field reference cannot be reassigned. Note that readonly doesn't make the OBJECT immutable — for a List, you still can't reassign the field, but you can still list.Add(x). Prefer readonly for most cases — safer for library evolution and supports any type. There's also static readonly for class-level constants of complex types. C# 10 added const for interpolated strings.

What's a record?

A record (added in C# 9) is a special kind of class designed for immutable data, with several conveniences built in. The biggest difference from a normal class is value-based equality — two records are equal if all their properties are equal, not just if they're the same instance. Other built-in goodies: a ToString() that prints all properties, a deconstructor (so you can do var (id, name) = user;), and a with expression that creates a copy with some properties changed (var modified = user with { Name = "New" }). The syntax is concise — public record User(int Id, string Name); declares a record with two properties in one line. Records default to immutable (init-only properties). They're perfect for DTOs, value objects, and any data-transfer scenarios. There are also record struct (C# 10) for value-type records when you want stack allocation.

Generics with constraints

Generics let you write code that works with any type while keeping compile-time type safety — like List<T> works for List of int, List of string, etc. Sometimes you need to restrict T to types that meet certain rules. That's what constraints do, declared with the where keyword. Common constraints: where T : class (must be a reference type), where T : struct (must be a value type), where T : new() (must have a parameterless constructor so you can do new T()), where T : IComparable<T> (must implement that interface), where T : SomeBaseClass (must inherit from that class), and where T : notnull. You can combine multiple: where T : class, IEntity, new(). Constraints unlock methods specific to that type — for example with the IComparable constraint, you can call a.CompareTo(b) inside the generic method. Generics + constraints are the foundation for type-safe libraries like LINQ, EF Core, and the entire collections framework.

8 ASP.NET / EF Core Q&A

Middleware order matters — example?

The middleware pipeline runs in the exact order you call app.UseXxx() in Program.cs, and order is critical because each middleware depends on what previous ones did. The standard correct order is UseRouting → UseCors → UseAuthentication → UseAuthorization → MapControllers. UseRouting must come first so that endpoint matching happens — middleware after it knows which endpoint is targeted. UseAuthentication MUST come before UseAuthorization because you need to know WHO the user is (authentication) before you can check WHAT they're allowed to do (authorization). If you flip them, authorization sees no user and rejects everything. Custom middleware like exception handling typically goes near the start (so it catches errors from everything below), while logging can go anywhere. Forgetting UseStaticFiles means your wwwroot files don't serve. Rule: when something doesn't work, check the middleware order first.

Singleton, Scoped, Transient — when?

These are the three service lifetimes in .NET dependency injection. Singleton — one instance for the entire app lifetime. Use for stateless services like configuration, in-memory cache, loggers, HttpClient factory. Scoped — one instance per HTTP request, reused within that request, then disposed. Use for DbContext (so all parts of one request share the same context and transaction) and request-bound services. Transient — a new instance every single time the service is injected, even within the same request. Use for lightweight stateless helpers. The classic interview trap is the captive dependency bug: never inject a Scoped service like DbContext into a Singleton, because the Scoped service gets "captured" forever, lives as long as the Singleton, breaks request isolation, and is not thread-safe. .NET has ValidateScopes in development mode to catch this.

What does [ApiController] do?

[ApiController] is an attribute on a controller class that turns on several modern Web API conventions automatically. First, automatic model validation — if the incoming JSON fails any [Required] or [EmailAddress] check, the framework returns a 400 Bad Request with a clean ProblemDetails JSON before your action even runs. Second, binding source inference — for complex types it assumes [FromBody], for simple types it assumes [FromRoute] or [FromQuery], so you don't have to write attributes everywhere. Third, errors come back as standard RFC 7807 ProblemDetails JSON, the modern industry standard. Fourth, it requires attribute routing instead of conventional routing. Together these make controllers cleaner and consistent. Almost every new ASP.NET Core API controller should have [ApiController].

How do you do JWT auth in ASP.NET?

Setup happens in three steps. First, in Program.cs, register JWT services: builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme).AddJwtBearer(opts => { opts.TokenValidationParameters = new() { ValidateIssuer = true, ValidateAudience = true, ValidateLifetime = true, ValidateIssuerSigningKey = true, ValidIssuer = ..., ValidAudience = ..., IssuerSigningKey = new SymmetricSecurityKey(...) }; }); and builder.Services.AddAuthorization();. Second, add to the pipeline: app.UseAuthentication(); app.UseAuthorization(); in that order, before MapControllers. Third, write a token generation method that creates Claims (sub, email, role), instantiates a JwtSecurityToken with issuer/audience/expiry/signing credentials, and uses JwtSecurityTokenHandler.WriteToken() to produce the string. The login endpoint validates credentials against the DB and returns this token. The client sends it in Authorization: Bearer ... on every request, the middleware validates the signature and populates HttpContext.User. For production, add refresh tokens stored in the DB so you can revoke.

Difference between Authentication and Authorization?

People often confuse these two but they answer different questions. Authentication answers "Who are you?" — verifying user identity through credentials, JWT validation, or another mechanism. After authentication, the server knows who the user is and what claims they carry. Failure returns HTTP 401 Unauthorized (which actually means "not authenticated"). Authorization answers "What are you allowed to do?" — checking whether the authenticated user has permission for the requested resource based on roles, policies, or claims. Failure returns HTTP 403 Forbidden, meaning "we know who you are, but you can't do this". A simple analogy: showing your ID at the office gate is authentication; the system checking whether your ID grants access to the 5th floor is authorization. In .NET, UseAuthentication() middleware handles the first, UseAuthorization() handles the second, and they MUST be called in that order.

How to handle exceptions globally?

The wrong way is to wrap every controller action in try/catch — repetitive and easy to miss. The right way is centralized via middleware. Write a custom ExceptionHandlingMiddleware that wraps the call to next() in a try/catch. When any unhandled exception bubbles up, it catches it, logs with full context (request path, user, correlation ID), maps the exception type to an HTTP status code (NotFoundException → 404, ValidationException → 400, anything else → 500), and writes a standardized ProblemDetails JSON response (RFC 7807). This gives clients consistent error shapes with title, status, detail. In .NET 8 there's an even cleaner built-in approach: implement IExceptionHandler, register it with AddExceptionHandler<T>(), add app.UseExceptionHandler(), and call builder.Services.AddProblemDetails(). Either way, you get one place to handle errors, consistent responses, and clean controllers.

What is CORS?

CORS stands for Cross-Origin Resource Sharing. It's a browser security mechanism that blocks JavaScript from one origin (domain + port + protocol) from calling APIs on a different origin by default. This protects users from malicious sites making calls to their banking app while they're logged in. You need CORS configuration whenever frontend and backend are on different origins — most commonly Angular at http://localhost:4200 calling .NET at http://localhost:5000. To allow this, configure CORS on the SERVER side (because the server is the one trusting the origin). In .NET: AddCors in services with a policy specifying WithOrigins, AllowAnyHeader, AllowAnyMethod, and optionally AllowCredentials; then app.UseCors("PolicyName") in the pipeline before UseAuthentication. Be careful: combining AllowAnyOrigin() with AllowCredentials() is forbidden by the spec — list specific origins when sending cookies or auth.

Difference between IActionResult and ActionResult<T>?

Both are return types for controller actions but serve slightly different needs. IActionResult is the base interface that any action result implements — Ok(), NotFound(), BadRequest(), CreatedAtAction(), etc. It's flexible: you can return any of these. The downside is the return type doesn't tell anyone (Swagger, callers, IDE) what data shape comes back on success. ActionResult<T> is the generic version added in newer .NET. It lets you return either a status code result like NotFound() OR a typed value of T directly with return user;. The advantage is much better Swagger documentation (Swagger sees that on success the response is a User object) and stronger typing for callers. Modern best practice: use ActionResult<T> for actions returning data on success, plain IActionResult for actions returning only status codes (like 204 NoContent on delete). Either way, the Ok/NotFound/etc helpers still work the same.

What's a filter? Types?

Filters are cross-cutting code that runs INSIDE the MVC pipeline, after routing has identified which action to call but around the action execution itself. Different from middleware (which runs in the broader HTTP pipeline). There are five filter types in execution order. Authorization filters run first to allow or deny access. Resource filters wrap everything else — used for caching or short-circuiting. Action filters run before and after the action method. Exception filters catch exceptions thrown by actions or other filters. Result filters wrap the result execution. Use filters when you need behavior that's specific to controllers (like [ServiceFilter] to log every action call). Use middleware when behavior should apply to all requests including static files. Filters are MVC-aware so they have access to the controller and action context, while middleware just sees raw HttpContext.

Explain N+1 in EF

N+1 is the most common EF performance bug and a guaranteed interview question. It happens when you fetch a list of parent rows with one query, then loop through them accessing a navigation property — which triggers one extra query per parent. So loading 100 orders and accessing order.Customer.Name in a loop runs 1 + 100 = 101 queries. With 50 such requests per second, you slam the DB with 5050 queries. Fix it two ways. First, eager loading with .Include(): db.Orders.Include(o => o.Customer).ToListAsync() generates a JOIN. Second and often best, projection: shape your query into a DTO with .Select(o => new { o.Id, CustomerName = o.Customer.Name }) — fetches only the columns you need. Always check generated SQL with .ToQueryString() during development to catch N+1 before it hits production.

Tracking vs NoTracking

EF Core tracks every entity it returns from a query by default, so it can detect changes you make and translate them to UPDATE statements when you call SaveChanges. Change-tracking has costs — extra memory (EF stores a snapshot of every entity) and extra CPU (it compares snapshots to detect changes). For UPDATE scenarios this cost is worth it. But for read-only queries — like a GET endpoint just returning data — tracking is wasted work. The fix is .AsNoTracking(): var products = await db.Products.AsNoTracking().ToListAsync();. This skips the change tracker entirely. Result is typically 20-50% faster and noticeably less memory, especially on large result sets. Rule of thumb: AsNoTracking on every GET / list endpoint, normal tracking when you'll modify and save. You can also set the default globally with UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking) for read-heavy apps.

EF Core vs Dapper

Both let you talk to a SQL database from .NET, but they sit at different points on the productivity vs raw-speed spectrum. EF Core is a full ORM — write LINQ, EF generates SQL, manages change tracking, supports migrations, handles relationships, integrates with the rest of .NET (DI, async). The cost is an abstraction layer that's slightly slower than hand-written SQL. Dapper is a "micro-ORM" — a thin layer over ADO.NET that maps SQL results to C# objects. You write SQL yourself: conn.QueryAsync<Product>("SELECT * FROM Products WHERE CategoryId = @cat", new { cat = 5 }). Performance is essentially the same as raw ADO.NET. The decision: EF Core for 80% of CRUD work because productivity matters more than 5% of perf. Dapper for hot read paths (heavy reporting queries, read-mostly endpoints with millions of hits) where every millisecond counts, or for complex SQL that LINQ can't express well. Many serious .NET projects use BOTH — EF for writes and standard CRUD, Dapper for read-optimized queries.

Eager vs Lazy vs Explicit loading

Three EF strategies to fetch related data. Eager loading uses .Include() to fetch related data in the same query as the parent — EF generates a JOIN. db.Orders.Include(o => o.Customer). This is the default and best choice for most cases — you control exactly what loads. Lazy loading loads related data on first access — read order.Customer and EF transparently fires a query at that moment. Requires the Microsoft.EntityFrameworkCore.Proxies package and virtual on navigation properties. The big problem is N+1 — accessing the navigation in a loop triggers one query per iteration. Lazy also breaks JSON serialization (infinite recursion). Avoid lazy loading in web apps. Explicit loading means manually calling db.Entry(order).Reference(o => o.Customer).LoadAsync() after the main query. Useful when you sometimes need related data and sometimes don't, but you control the timing. Modern best practice: eager loading with .Include() for full entities, or even better, .Select() projection into DTOs.

How does EF Concurrency work?

EF supports optimistic concurrency, which assumes conflicts are rare and detects them at save time instead of locking rows pessimistically. The standard pattern in SQL Server: add a RowVersion column (a byte array marked with [Timestamp]) to your entity. SQL Server auto-increments this column on every UPDATE. When EF generates an UPDATE statement, it includes WHERE Id = @id AND RowVersion = @originalRowVersion. If another user updated the row between your fetch and your save, the rowversion changed, the UPDATE affects 0 rows, and EF throws DbUpdateConcurrencyException. You catch this and decide how to handle: retry the operation, merge the conflict by re-fetching and re-applying, show the user a "this was modified by someone else" message, or pick a winner (last-write-wins). This is much more scalable than pessimistic locking because no DB locks are held while users think. For pessimistic scenarios you'd use raw SQL with SELECT ... FOR UPDATE, but in real .NET apps optimistic is what you want 95% of the time.

What's CQRS?

CQRS stands for Command Query Responsibility Segregation. The core idea is to separate the code (and often the data model) for writes from the code for reads. Commands are write operations — CreateOrder, UpdateProduct, DeleteUser. They go through dedicated handler classes containing validation, business rules, calls to repositories. They typically don't return data, just success/failure. Queries are read operations — GetOrder, ListProducts. They go through their own handlers reading from optimized projections, often denormalized or pulled from cache. Benefits: reads and writes scale independently and have different complexity profiles. Reads can be denormalized for speed; writes can be richly modeled with full business rules. In .NET, CQRS is usually implemented with the MediatR library — your controllers just call mediator.Send(command) and the right handler runs. CQRS doesn't always mean separate databases — it can be just a code pattern. When paired with Event Sourcing, you get truly separate read/write storage.

Why use DTOs?

DTO stands for Data Transfer Object — a separate class representing the shape of data sent over the wire, separate from your DB entity. There are several strong reasons to use them. First, security — your User entity might have a PasswordHash field; if you return it directly, that hash leaks. With a DTO you only include safe fields. Second, over-posting protection — if your API accepts the User entity directly, a malicious client could send { "isAdmin": true } and EF would set that property when binding. With a CreateUserDto that only has Email and Password, IsAdmin is invisible. Third, decoupling — DB schema can change without breaking API consumers. Fourth, payload size — only include fields the client needs, keeping responses small. Use a tool like AutoMapper to map between Entity and DTO without writing tedious property-by-property code. This is one of the easiest "senior-level" practices to adopt and interviewers always notice it.

How to make API stateless?

A stateless API doesn't store any per-user data in its own memory between requests. Every request carries everything needed to process it. The huge benefit: ANY server instance can handle ANY request — they're interchangeable, so horizontal scaling is trivial. To achieve statelessness: use JWT for authentication instead of server-side sessions — the token contains user identity and claims. For shared state across requests (like a shopping cart for a logged-in user), use Redis or another distributed store, never in-memory dictionaries. Don't store anything in static fields you'd need across requests. Build idempotent operations so they can be safely retried. The result: you can spin up 10 more instances behind a load balancer with zero code changes. If a server crashes, no user state is lost. This is the foundation for cloud-native scaling on Azure App Service, Kubernetes, anywhere.

What is gRPC and when to use it?

gRPC is a high-performance Remote Procedure Call framework built by Google, using HTTP/2 for transport and Protobuf (Protocol Buffers) for serialization instead of JSON. It's strongly-typed end-to-end via .proto schema files that generate matching client and server code. The advantages over REST: significantly faster (binary format, smaller payloads), strongly typed contracts (no JSON guesswork), and built-in support for streaming (server-streaming, client-streaming, bidirectional). The trade-offs: not browser-friendly without a proxy (browsers can't speak HTTP/2 with custom framing easily), harder to debug than JSON over HTTP, requires the .proto file to be shared between client and server. The right use case is internal microservice-to-microservice communication within your infrastructure — fast, type-safe, efficient. For public APIs facing browsers and third parties, stick with REST or GraphQL. .NET has excellent gRPC support — Microsoft.AspNetCore.Grpc packages — and many companies use it for backend service meshes.

SignalR — what is it?

SignalR is .NET's real-time communication framework. It abstracts away the transport so your code just calls "broadcast a message to all clients" or "send to a specific user", and SignalR picks the best transport: WebSockets (preferred), Server-Sent Events, or long polling as fallback. Use cases: chat applications, live notifications, real-time dashboards (e.g., live order tracking), collaborative editing, multiplayer games, live sports/stock tickers. On the server, you create a Hub class with methods that clients can invoke and that can call back to clients. On the Angular side, install @microsoft/signalr and create a HubConnection. Authentication integrates with JWT — pass the token via accessTokenFactory. For scaling across multiple servers, use the Redis backplane so that a message published from one server reaches clients connected to other servers. SignalR is one of the easiest "wow factor" features to add to a portfolio project for interviews.

Health checks — why and how?

A health check endpoint is a URL like /healthz that reports whether your app is alive and its dependencies are working. The reason: load balancers and orchestrators (like Azure App Service or Kubernetes) periodically ping this endpoint. If it returns failure, they stop routing traffic to that instance, preventing users from hitting a dead server. In .NET, the setup is one line: app.MapHealthChecks("/healthz"). To check more than just "the app is responding", chain in dependency checks: builder.Services.AddHealthChecks().AddDbContextCheck<AppDbContext>().AddRedis(...).AddUrlGroup(new Uri("https://api.payment.com/status")). This way the endpoint reports unhealthy if DB, Redis, or downstream APIs are down. Best practice: have TWO endpoints — /healthz/live for liveness (is the process alive?) and /healthz/ready for readiness (are dependencies ready to serve traffic?). Liveness failure means restart the pod; readiness failure means stop sending traffic but don't restart.

9 System Design Round Framework

For 12-15 LPA, expect 1 system design round. They want to see structure, tradeoffs, and conversation — not perfection.

🎯 The 6-Step Framework (use every time)

  1. Clarify requirements (2 min) — functional + non-functional. Ask about scale, latency, availability.
  2. Estimate scale (2 min) — QPS, storage, bandwidth (back-of-envelope).
  3. High-level design (5 min) — boxes and arrows. Client → LB → API → Cache → DB.
  4. Deep dive (10 min) — pick 1-2 components: schema, sharding, caching strategy, queue.
  5. Identify bottlenecks (5 min) — what fails first under load? How to fix?
  6. Wrap up (2 min) — tradeoffs, what you'd do differently with more time.
