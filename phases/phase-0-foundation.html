Phase 0 — Foundation & TS → C# Bridge

You're already a developer. This phase isn't about learning to code — it's about installing the .NET stack and translating your TypeScript instincts into C# syntax.

⏱ 1 Week 🎯 Setup + Bridge 💪 Lighter for you

📋 Prerequisites

  • Working laptop, 8GB+ RAM, ~20 GB free disk
  • Existing JS/TS knowledge (you have this ✓)
  • Comfortable with Git/GitHub (you have this ✓)

1 Tools & Environment Setup

The .NET ecosystem has a primary IDE (Visual Studio) and supporting tools. Get this right once and you won't fight your tools later.

🎯 What to Install

  • Visual Studio 2022 Community — the main .NET IDE (free). Pick workloads: ASP.NET and web development, .NET desktop development, Data storage and processing.
  • .NET 8 SDK — long-term support runtime. Verify: dotnet --version
  • VS Code + extensions: C# Dev Kit, C#, NuGet Gallery, REST Client
  • SQL Server Express + SSMS (or Azure Data Studio)
  • Postman — API testing
  • Docker Desktop — install now, used in Phase 7
  • Git — already configured
# verify everything is installed
dotnet --version              # should print 8.x.x
dotnet --list-sdks
git --version
docker --version
⚠️ Gotcha: Don't install old .NET Framework 4.8 confused with .NET 8. .NET Framework = legacy Windows-only. .NET (5/6/7/8) = modern cross-platform. You want the modern one.

2 The .NET Ecosystem

Definition: .NET is a big platform with many parts. You should know each part for interview.

Important Parts

  • .NET Framework (1.0 → 4.8) — old, Windows only, in maintenance mode
  • .NET Core (1, 2, 3.x) — new cross-platform version. Renamed to just ".NET" from v5
  • .NET 5/6/7/8 — modern unified version. .NET 8 is current LTS
  • C# — language we write code in
  • .NET — runtime + libraries
  • ASP.NET Core — web framework that run on .NET
  • CLR (Common Language Runtime) — engine that run IL code
  • BCL (Base Class Library) — built-in types like String, List, File

Common Terms

TermMeaning
SDKTools to build apps (compiler, CLI) — developer install this
RuntimeWhat is needed to run apps — server install this
NuGetPackage manager — like npm for .NET
MSBuildBuild engine — like webpack for .NET
LTSLong-Term Support release — pick for production (.NET 6, 8)

Java parallel

  • .NET SDK = JDK
  • .NET Runtime = JRE
  • CLR = JVM
  • NuGet = Maven
  • BCL = java.lang + java.util packages

Note: Always pick latest LTS version (.NET 8) for new projects.

3 TypeScript → C# Syntax Bridge

You already know strong typing, classes, generics, async/await, and decorators in TS. C# uses the same ideas with slightly different syntax. Learn the diff, not the concept.

ConceptTypeScriptC#
Variablelet x: number = 5int x = 5; or var x = 5;
Stringlet s: string = "hi"string s = "hi";
Arraynumber[] / Array<number>int[] or List<int>
MapMap<string, number>Dictionary<string, int>
Classclass User { name: string; }class User { public string Name { get; set; } }
Interfaceinterface IUser { id: number }interface IUser { int Id { get; set; } }
Constructorconstructor(name: string) {}public User(string name) {}
Asyncasync fn(): Promise<User>async Task<User> Fn()
Null safetystring \| null, ?.string?, ?., ??
Genericfunction f<T>(x: T): TT F<T>(T x)
Decorator@Injectable()[Authorize] (called attribute)
Spread[...arr]arr.ToList() or [..arr] (C# 12)
Destructureconst {name} = uservar (name, _) = user; (records)
String interp`Hello ${name}`$"Hello {name}"
Optional paramfunction f(x?: number)void F(int x = 0)

🎯 Mental Model Differences

  • Properties vs fields: C# uses properties ({ get; set; }) instead of TS's plain class fields. Same role, different syntax.
  • Naming: C# uses PascalCase for methods/properties (GetUser), camelCase only for local variables and parameters.
  • Statements end with semicolons, just like TS.
  • No truthy/falsy: if (x) only works on bool in C#. You must write if (x != null) or if (str.Length > 0).
// Same logic — TS first, then C#

// TS:
interface IUser { id: number; name: string; }
class UserService {
  async getUser(id: number): Promise<IUser> {
    return { id, name: "Rutik" };
  }
}

// C#:
public interface IUser { int Id { get; set; } string Name { get; set; } }
public class UserService {
  public async Task<IUser> GetUser(int id) {
    return new User { Id = id, Name = "Rutik" };
  }
}
🔗 Angular Parallel: 80% of the syntax is muscle-memory swap. The concepts are identical — types, classes, interfaces, generics, async. The only true new thing is properties with get/set.

4 dotnet CLI — your daily driver

The dotnet CLI creates, builds, runs, and tests projects. You'll use it constantly. Learn the 10 commands that cover 95% of work.

# project lifecycle
dotnet new list                       # show all templates
dotnet new console -n MyApp           # console app
dotnet new webapi -n MyApi            # Web API
dotnet new classlib -n MyLib          # class library
dotnet new sln -n MySolution          # solution file
dotnet sln add MyApi/MyApi.csproj     # add proj to solution

# build / run / watch
dotnet restore                        # restore NuGet packages
dotnet build                          # compile
dotnet run                            # compile + run
dotnet watch run                      # hot reload (like ng serve)

# testing
dotnet new xunit -n MyApp.Tests
dotnet test

# packages (NuGet)
dotnet add package Serilog
dotnet remove package Serilog
dotnet list package --outdated

# publish for production
dotnet publish -c Release -o ./publish
🔗 Angular Parallel:
dotnet new webapi ≈ ng new my-app
dotnet watch run ≈ ng serve
dotnet add package ≈ npm install
dotnet build ≈ ng build
dotnet test ≈ ng test

5 NuGet — Package Manager

NuGet is to .NET what npm is to Node. Packages live at nuget.org and dependencies are tracked in your .csproj file (the .NET equivalent of package.json).

🎯 Things You'll Use Daily

  • .csproj file lists <PackageReference Include="Serilog" Version="3.0.0" />
  • No node_modules! Packages cached globally at ~/.nuget/packages/
  • Install via CLI (dotnet add package X) or VS Package Manager UI
  • Lock file: packages.lock.json (opt-in, recommended for CI)

📦 Packages You'll See Constantly

  • Microsoft.EntityFrameworkCore — ORM (Phase 4)
  • Microsoft.AspNetCore.Authentication.JwtBearer — JWT (Phase 3)
  • Serilog.AspNetCore — structured logging
  • AutoMapper — DTO mapping
  • FluentValidation — input validation
  • MediatR — CQRS handler dispatcher
  • xUnit / Moq / FluentAssertions — testing trinity
  • Polly — retry/circuit-breaker resilience
  • Dapper — micro-ORM for raw SQL speed
⚠️ Gotcha: Packages have version compatibility tied to .NET versions (e.g. EF Core 8 needs .NET 8). Always check the package's "Frameworks" section on nuget.org.

6 Visual Studio Debugging — Skill that Pays Forever

VS has the best debugger in the industry. Master these from day one — interviewers respect a dev who debugs systematically instead of Console.WriteLine everywhere.

🎯 Must-Know Debugger Features

  • Breakpoints — F9 to set, F5 to run-to-next, F10 step over, F11 step into, Shift+F11 step out
  • Conditional breakpoints — break only when x > 100 (right-click breakpoint → Conditions)
  • Tracepoints — log without stopping (replaces Console.WriteLine)
  • Watch / QuickWatch (Shift+F9) — inspect any expression while paused
  • Immediate window — execute C# code mid-debug
  • Call Stack — see how you got here
  • Locals / Autos — inspect current scope variables
  • Edit and Continue — modify code while paused, keep running
  • Exception Settings — break on thrown exceptions (Debug → Windows → Exception Settings)

⌨️ Productivity Shortcuts

🎤 Interview Q&A — Phase 0

Q. What is difference between .NET Framework and .NET (Core)?

Definition:

  • .NET Framework = old version, only run on Windows
  • .NET (Core) = new version, run on Windows, Linux, Mac

Key Points:

  • .NET Framework last version is 4.8 — Microsoft not adding new features
  • .NET Core was renamed to just ".NET" from version 5
  • Current LTS version is .NET 8 — support for 3 years
  • .NET Framework is legacy, only for old enterprise apps
  • For all new projects use .NET 8 (or latest LTS)
  • .NET (Core) is open source and faster

Java parallel:

  • Same like JDK 1.8 vs JDK 17 — old vs modern version
  • .NET 8 is like JDK LTS — 3 year support

Note: In interview just say — "Framework is old Windows-only, .NET 8 is modern cross-platform. I always use .NET 8."

Q. What is the CLR?

Definition:

  • CLR = Common Language Runtime
  • It is the engine that run C# code at run time

Key Points:

  • C# code first compile to IL (Intermediate Language) — not machine code
  • CLR has JIT (Just-In-Time) compiler that convert IL to machine code at run time
  • CLR handle Garbage Collection (clean unused memory)
  • CLR handle Exception (throw/catch flow)
  • CLR handle Security (check what code can do)
  • Other languages like F#, VB.NET also run on CLR

Java parallel:

  • CLR is same like JVM in Java
  • IL is same like Java Bytecode
  • JIT in C# = JIT in Java — same idea

Note: CLR enable features like Reflection, Generics, async/await.

Q. What is NuGet and how it relate to npm?

Definition:

  • NuGet is package manager for .NET
  • Same like npm for Node.js

Key Points:

  • Packages hosted at nuget.org
  • Install command: dotnet add package PackageName
  • Package listed in .csproj file (same like package.json)
  • Packages stored in one global folder — not per project
  • No node_modules like big folder problem
  • Lock file: packages.lock.json for CI/CD

Java parallel:

  • Same like Maven (pom.xml) or Gradle (build.gradle)
  • NuGet ≈ Maven Central

Common packages:

  • Serilog — logging
  • AutoMapper — object mapping
  • EF Core — database ORM
  • FluentValidation — input validation
Q. SDK vs Runtime — when to install which?

Definition:

  • SDK = Software Development Kit (for building apps)
  • Runtime = only for running already built apps

Key Points:

  • SDK contain — compiler + dotnet CLI + build tools + runtime
  • Runtime contain only — what is needed to run app
  • Developer install SDK on machine
  • Production server install only Runtime
  • Runtime is small in size — save space on server

Java parallel:

  • Same like JDK vs JRE
  • SDK = JDK (for development)
  • Runtime = JRE (for run only)

Real example:

  • In Docker — first stage use SDK image to compile
  • Second stage use small Runtime image to run
  • Final image is small and more secure
Q. Why C# is strongly typed?

Definition:

  • Strongly typed = compiler check type of every variable
  • If int x = 5, you cannot assign "hello" to x

Key Points:

  • Type check happen at compile time — error shown before run
  • Most silly bugs caught in build, not in production
  • Refactoring is safe — rename method, compiler show all places
  • IntelliSense show correct method suggestion
  • Big code base easy to maintain
  • C# check type at both compile time AND run time

Java parallel:

  • Same like Java — both are strongly typed
  • Same compile time type check
  • Same IntelliSense / IDE support benefit

Note: TypeScript also strongly typed but type disappear at run time. C# keep types at run time also — extra safety.

⚠️ Common Mistakes in Phase 0

  • Installing .NET Framework 4.8 instead of .NET 8 SDK.
  • Skipping Visual Studio in favor of VS Code only — VS is the platform standard for .NET; learn it even if VS Code stays your daily driver.
  • Treating C# as "Java but worse." It's the most modern enterprise language alongside Kotlin — appreciate features like records, pattern matching, and source generators.
  • Trying to learn C# basics from a "for absolute beginners" course. You're already a dev — read "C# in Depth" or Microsoft Docs directly.
