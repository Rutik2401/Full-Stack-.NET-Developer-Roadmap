📅 Week 1-2 — C# Basics

1 Program Structure & Syntax

Every C# program is organized into namespaces (file groups), classes (containers), and methods (functions). Execution starts at Main — or in modern C# 9+, at the top of Program.cs directly (top-level statements).

🎯 Interview Points

  • using imports namespaces (like TS import)
  • namespace groups related types (like TS modules)
  • Main is the entry point; signature is static void Main(string[] args)
  • Top-level statements (C# 9+) let you skip Main in Program.cs
  • Global usings (C# 10) let you import once for the whole project
// classic
using System;

namespace MyApp;

public class Program {
  public static void Main(string[] args) {
    Console.WriteLine("Hello");
  }
}

// modern (C# 9+) — top-level statements
using System;
Console.WriteLine("Hello");
🔗 Angular Parallel: namespace ≈ Angular module, using ≈ import statement. Top-level statements = like writing logic in main.ts directly.

2 Value Types vs Reference Types

The single most important concept in C# memory model. Value types hold the data directly (stack). Reference types hold a pointer to data on the heap. This affects copying, equality, and performance.

🎯 Interview Points

  • Value types: int, double, bool, char, struct, enum, decimal, DateTime — copied by value
  • Reference types: class, string, array, interface, delegate — copied by reference
  • Passing a value type to a method copies it — caller's variable unchanged
  • Passing a reference type passes the reference — both names point to same object, mutations visible to all
  • String is reference type but immutable — behaves like value for assignment
  • Boxing: wrapping a value type in an object reference (heap allocation, slow). Unboxing: reverse.
// Value type
int a = 5; int b = a; b = 10;
// a is still 5 — copied by value

// Reference type
var list1 = new List<int> { 1, 2 };
var list2 = list1; list2.Add(3);
// list1 also has [1,2,3] — same object

// Boxing — avoid in hot paths
int n = 42;
object boxed = n;        // allocates on heap
int unboxed = (int)boxed;
⚠️ Gotcha: Boxing in tight loops kills performance. Use generics (List<int>) instead of ArrayList to avoid boxing.
🔗 Angular Parallel: Same as JS — primitives (number, boolean) are by-value, objects/arrays are by-reference. C# just makes the rule explicit and adds struct for custom value types.

3 Nullable Types & Null Operators

C# 8+ added nullable reference types — like TS strict null checks. The compiler tracks nullability and warns on potential null dereferences.

🎯 Interview Points

  • int? = nullable value type (alias for Nullable<int>)
  • string? = nullable reference type (post-C# 8 with <Nullable>enable</Nullable>)
  • ?. — null-conditional access (same as TS optional chaining)
  • ?? — null-coalescing (returns right if left is null)
  • ??= — assign only if null
  • ! — null-forgiving operator (tells compiler "trust me, not null")
  • .HasValue / .Value — explicit access on nullable value types
int? age = null;
if (age.HasValue) Console.WriteLine(age.Value);

string? name = GetName();
int len = name?.Length ?? 0;          // safe access

string greeting = name ?? "Guest";    // fallback
config.Timeout ??= 30;                  // assign if null

// null-forgiving — use sparingly
var upper = name!.ToUpper();          // I promise it's not null
🔗 Angular Parallel: Identical to TS strict mode — ?. and ?? are the same. Enable <Nullable>enable</Nullable> in .csproj just like strict: true in tsconfig.

4 Type Conversion

C# is strict about type conversion. Implicit happens automatically when safe (no data loss). Explicit (cast) requires (type)value. Parsing strings has dedicated APIs.

🎯 Conversion Methods

  • Implicit: int → long → double (no loss)
  • Explicit (cast): (int)3.14 = 3 (data loss → must be explicit)
  • Convert.ToXxx: handles null safely, returns 0 for null
  • Parse: throws FormatException if invalid
  • TryParse ⭐: returns bool, never throws — preferred in production
  • as: safe reference cast, returns null if fails
  • is: type-check, returns bool (with optional pattern)
// Always prefer TryParse for user input
if (int.TryParse(input, out var num))
    Console.WriteLine(num);
else
    Console.WriteLine("Invalid");

// is + pattern (modern)
if (obj is User u) Console.WriteLine(u.Name);

// as cast
var user = obj as User;     // null if not User
⚠️ Gotcha: Using Parse on user input is a common bug source — always use TryParse in real apps.

5 Strings & StringBuilder

Strings in C# are immutable reference types. Every modification creates a new string. For heavy concatenation use StringBuilder (mutable, single allocation).

🎯 Interview Points

  • String pooling (interning): identical literals share memory
  • String interpolation: $"Hello {name}" — preferred over concat
  • Verbatim strings: @"C:\path\file" — no escape processing
  • Raw strings (C# 11): """multiline content"""
  • String.Equals with StringComparison.OrdinalIgnoreCase for case-insensitive
  • StringBuilder in loops with 4+ appends, otherwise + is fine
  • Common methods: Split, Join, Trim, Replace, Substring, Contains, IndexOf, ToUpper, ToLower
// Interpolation + format
var price = 99.5; var name = "Pro";
Console.WriteLine($"{name}: ₹{price:F2}");     // Pro: ₹99.50

// StringBuilder for heavy concat
var sb = new StringBuilder();
for (int i = 0; i < 1000; i++) sb.Append(i).Append(", ");
var result = sb.ToString();

// Case-insensitive compare (correct way)
if (str.Equals("Admin", StringComparison.OrdinalIgnoreCase)) ...
⚠️ Gotcha: str1 == str2 compares values in C# (operator overload), unlike Java where it compares references. So you usually don't need .Equals() for value comparison — but DO use it with StringComparison for case rules.

📅 Week 3 — OOP Deep Dive

6 Classes, Objects, Constructors

A class is a blueprint; an object is an instance. Constructors initialize state. C# supports default, parameterized, static, and copy constructors plus the modern primary constructor (C# 12).

🎯 Constructor Types

  • Default (parameterless) — auto-generated if no constructor defined
  • Parameterized — typical use case
  • Static — runs ONCE before any access; no params, no access modifier
  • Copy constructor — manual: takes same-type object, clones state
  • Primary constructor (C# 12) — declare params on the class itself
  • Constructor chaining: : this(...) calls another ctor; : base(...) calls parent's
public class Order {
    public int Id { get; }
    public DateTime CreatedAt { get; }
    public static int TotalCreated;

    // static ctor — runs once before any Order is created
    static Order() { TotalCreated = 0; }

    // parameterized
    public Order(int id) {
        Id = id;
        CreatedAt = DateTime.UtcNow;
        TotalCreated++;
    }

    // chaining
    public Order() : this(0) { }
}

// Primary constructor (C# 12)
public class Money(decimal amount, string currency) {
    public decimal Amount => amount;
    public string Currency => currency;
}
🔗 Angular Parallel: Constructors work the same way as in TypeScript classes. C# 12's primary constructor is similar to TS shorthand constructor(public name: string) {}.

7 Encapsulation & Access Modifiers

Hide implementation details behind a public interface. C# offers fine-grained visibility control with 6 access modifiers.

🎯 Best Practices

  • Default to private for fields, expose via properties
  • Use internal for library implementation details — invisible to consumers
  • protected for extensibility points designed for inheritance

8 Inheritance

A class can extend one base class with : syntax. C# supports single inheritance for classes, multiple for interfaces.

🎯 Interview Points

  • Single inheritance only for classes; multiple for interfaces
  • base keyword accesses parent members; : base(...) calls parent constructor
  • sealed class cannot be inherited (perf optimization, finality)
  • sealed override on a method prevents further override
  • All classes implicitly inherit from System.Object
public class Animal {
    protected string Name;
    public Animal(string name) { Name = name; }
    public virtual void Speak() => Console.WriteLine("...");
}

public class Dog : Animal {
    public Dog(string name) : base(name) { }
    public override void Speak() {
        base.Speak();
        Console.WriteLine($"{Name}: Woof!");
    }
}

public sealed class Puppy : Dog { ... }   // can't inherit further

9 Polymorphism (Overloading + Overriding)

Two flavors. Compile-time (overloading): same method name, different signatures. Runtime (overriding): derived class replaces base class behavior using virtual + override.

🎯 virtual / override / new

  • virtual on base method allows override
  • override in derived class replaces virtual implementation (runtime dispatch)
  • new hides base method (compile-time, dispatch depends on declared type — usually a code smell)
  • abstract method has no body and MUST be overridden
  • Method overloading: same name, different parameter types/count (compile-time)
class Shape {
    public virtual double Area() => 0;
}
class Circle : Shape {
    public double Radius { get; set; }
    public override double Area() => Math.PI * Radius * Radius;
}

// runtime polymorphism
Shape s = new Circle { Radius = 5 };
Console.WriteLine(s.Area());   // 78.5 — Circle's version called

// overloading
class Calculator {
    public int Add(int a, int b) => a + b;
    public double Add(double a, double b) => a + b;
}
⚠️ Gotcha: If you forget virtual on the base, the derived override won't compile. Use new only if you mean to shadow, not override.

10 Abstraction — Abstract Class vs Interface

Both define contracts; pick based on intent. Use interface for capability ("can do X"). Use abstract class when you have shared partial implementation + state.

AspectAbstract ClassInterface
Multiple inheritanceNo (single)Yes (many)
State (fields)YesNo (until C# 8 default impls)
ConstructorYesNo
Access modifiersAnypublic (default)
Default method bodyYesYes (C# 8+)
Use whenShared code + "is-a"Capability + "can-do"
 
public interface IRepository<T> {
    Task<T?> GetById(int id);
    Task Add(T entity);
}

public abstract class RepositoryBase<T> : IRepository<T> {
    protected readonly DbContext _ctx;
    protected RepositoryBase(DbContext ctx) => _ctx = ctx;

    public abstract Task<T?> GetById(int id);     // must override
    public virtual Task Add(T e) { _ctx.Add(e); return _ctx.SaveChangesAsync(); }
}
🔗 Angular Parallel: Same as TS interfaces vs abstract classes. The "interface for capability, abstract for shared code" rule applies identically.

11 Properties (the C# way of fields)

Properties expose data with optional logic — they're methods masquerading as fields. Always use properties, never public fields, even for simple cases.

🎯 Property Variants

  • Auto-property: public string Name { get; set; }
  • Read-only auto: public int Id { get; } (set in constructor)
  • Init-only (C# 9): public string Email { get; init; } (immutable after construction)
  • Full property with backing field for logic
  • Computed (expression-bodied): public int Sum => A + B;
  • Required (C# 11): public required string Name { get; set; }
public class User {
    public required string Name { get; init; }
    public DateTime BirthDate { get; init; }

    // computed
    public int Age => DateTime.Today.Year - BirthDate.Year;

    // full property with validation
    private string _email = "";
    public string Email {
        get => _email;
        set {
            if (!value.Contains("@")) throw new ArgumentException();
            _email = value;
        }
    }
}

var u = new User { Name = "Rutik", BirthDate = new(1998, 1, 1) };
⚠️ Gotcha: Public fields in C# break encapsulation AND ABI compatibility (changing a field to a property is a binary-breaking change). Always start with properties.

All in System.Collections.Generic. Pick based on access pattern: ordered indexed (List), key→value (Dictionary), unique items (HashSet), FIFO (Queue), LIFO (Stack).

var users = new List<User>();

if (ages.TryGetValue("Rutik", out var age)) ...

13 Generics & Constraints

Write code that works for any type while keeping compile-time type safety. Constraints (where) restrict T to types matching certain criteria.

🎯 Constraints

  • where T : class — reference type
  • where T : struct — value type
  • where T : new() — has parameterless constructor
  • where T : IComparable — implements interface
  • where T : BaseClass — inherits from class
  • where T : notnull — non-nullable
public class Repository<T> where T : class, IEntity, new() {
    private List<T> _items = new();
    public T Create() => new T();
    public T? Find(int id) => _items.FirstOrDefault(x => x.Id == id);
}

// generic method
T Max<T>(T a, T b) where T : IComparable<T> =>
    a.CompareTo(b) > 0 ? a : b;

14 Exception Handling

Use try/catch/finally. Throw Exception subclasses to signal errors. Don't use exceptions for control flow.

🎯 Best Practices

  • Catch specific exceptions first, generic Exception last
  • finally always runs (even on return) — for cleanup
  • Use using statement for IDisposable auto-cleanup (preferred over try/finally)
  • Custom exceptions: inherit from Exception, end name with "Exception"
  • throw vs throw ex: throw; preserves stack trace, throw ex; resets it (almost always wrong)
  • In Web API, use global exception middleware instead of try/catch in every controller
  • Log exceptions with stack trace using ILogger, never swallow silently
try {
    var data = File.ReadAllText(path);
} catch (FileNotFoundException ex) {
    _logger.LogWarning("Missing file: {Path}", path);
    throw new ConfigMissingException("Config not found", ex);
} catch (Exception ex) when (ex.Message.Contains("timeout")) {
    // exception filter — newer feature
    _logger.LogError(ex, "Timeout");
    throw;     // preserves stack trace
} finally {
    cleanup();
}

// using = automatic Dispose, no try/finally needed
using var stream = File.OpenRead(path);
⚠️ Gotcha: throw ex; resets the stack trace and hides the original error location. Always use bare throw;.

15 File Handling

The System.IO namespace has APIs for files, directories, and streams. File.ReadAllText for small files, streams for large.

// quick file ops
string text = File.ReadAllText("data.txt");
File.WriteAllText("out.txt", text);
var lines = File.ReadAllLines("log.txt");

// async (preferred for large files / web)
var json = await File.ReadAllTextAsync("data.json");
await File.WriteAllTextAsync("out.json", json);

// stream-based for big files (don't load all into memory)
using var sr = new StreamReader("big.csv");
string? line;
while ((line = await sr.ReadLineAsync()) != null) {
    Process(line);
}

// directories
Directory.CreateDirectory("logs");
var files = Directory.EnumerateFiles(".", "*.cs", SearchOption.AllDirectories);

// safe path construction
var path = Path.Combine("data", "users.json");
⚠️ Gotcha: Always use Path.Combine instead of string concat with slashes — it handles platform separators (Windows \ vs Unix /) correctly.

16 LINQ — Language Integrated Query ⭐

LINQ unifies querying over collections, databases, XML — same syntax everywhere. Two flavors: method syntax (chained methods) and query syntax (SQL-like). Both compile to identical IL.

🎯 Most-Used Operators

  • Filter: Where
  • Project: Select, SelectMany
  • Order: OrderBy, OrderByDescending, ThenBy
  • Group: GroupBy
  • Join: Join, GroupJoin
  • Aggregate: Count, Sum, Min, Max, Average, Aggregate
  • Single element: First, FirstOrDefault, Single, SingleOrDefault, Last
  • Any element: Any, All, Contains
  • Set ops: Distinct, Union, Intersect, Except
  • Materialize: ToList, ToArray, ToDictionary
// method syntax (preferred)
var top5 = orders
    .Where(o => o.Total > 100)
    .OrderByDescending(o => o.Total)
    .Select(o => new { o.Id, o.Total })
    .Take(5)
    .ToList();

// query syntax
var q = from o in orders
        where o.Total > 100
        orderby o.Total descending
        select new { o.Id, o.Total };

// group + aggregate
var byUser = orders
    .GroupBy(o => o.UserId)
    .Select(g => new { UserId = g.Key, Total = g.Sum(o => o.Total) });

🎯 IEnumerable vs IQueryable (interview favorite)

  • IEnumerable<T> — in-memory, executes locally (LINQ to Objects). Filter happens after fetch.
  • IQueryable<T> — translates expression tree to backend (SQL via EF). Filter happens at DB.
  • Calling .AsEnumerable() too early fetches everything then filters in memory — performance trap.

🎯 Deferred vs Immediate Execution

  • Deferred: Where, Select, OrderBy — query runs on enumeration (foreach, ToList)
  • Immediate: ToList, ToArray, Count, First, Sum — query runs immediately
⚠️ Gotcha: Re-enumerating a deferred query re-runs it. If you'll iterate twice, call .ToList() first.
🔗 Angular Parallel: RxJS operators map perfectly: filter ≈ Where, map ≈ Select, reduce ≈ Aggregate. The functional pipeline is identical.

📅 Week 5 — Advanced

17 Delegates, Func, Action, Predicate

A delegate is a type-safe function pointer. Func, Action, Predicate are built-in generic delegates that cover 99% of use cases.

🎯 The Trio

  • Action — returns void: Action<int, string>
  • Func — last type param is return: Func<int, int, bool> (two ints → bool)
  • Predicate<T> — shortcut for Func<T, bool>
  • Multicast delegates can chain multiple methods (+=, -=)
  • Delegates are the basis of events, LINQ, and async callbacks
Action<string> log = msg => Console.WriteLine(msg);
Func<int, int, int> add = (a, b) => a + b;
Predicate<int> isEven = n => n % 2 == 0;

log("hello");
Console.WriteLine(add(2, 3));
Console.WriteLine(isEven(4));

// passing delegates as params (LINQ does this internally)
List<int> nums = [1, 2, 3, 4, 5];
var evens = nums.FindAll(isEven);
🔗 Angular Parallel: Like callback functions in TS. Func<X, Y> ≈ (x: X) => Y. C# just gives the function-type a name.

18 Events

An event is a delegate wrapped with safety — only the declaring class can raise it; outside code can only subscribe/unsubscribe. The publisher-subscriber (observer) pattern made first-class.

public class OrderService {
    public event EventHandler<Order>? OrderPlaced;

    public void Place(Order o) {
        Save(o);
        OrderPlaced?.Invoke(this, o);    // raise
    }
}

// subscribe
var svc = new OrderService();
svc.OrderPlaced += (sender, order) => Console.WriteLine($"New: {order.Id}");
svc.OrderPlaced += SendEmailHandler;

// unsubscribe — important to prevent memory leaks!
svc.OrderPlaced -= SendEmailHandler;
⚠️ Gotcha: Forgetting to unsubscribe creates memory leaks — the event holds a reference to the handler's owner.
🔗 Angular Parallel: Same as RxJS Subjects — emit/subscribe/unsubscribe. Or like Angular's @Output() EventEmitter.

19 Lambdas, Anonymous Methods, Extension Methods

Lambdas are inline functions: (x) => x * 2. Extension methods let you add methods to existing types without inheriting — the magic behind LINQ.

// lambda variants
Func<int, int> sq = x => x * x;
Func<int, int, int> add = (a, b) => a + b;
Action greet = () => Console.WriteLine("hi");

// multi-statement lambda
Func<int, int> complex = x => {
    var y = x * 2;
    return y + 1;
};

// closure — captures outer variable
int counter = 0;
Action inc = () => counter++;

// EXTENSION METHOD — must be in static class, first param has 'this'
public static class StringExtensions {
    public static bool IsValidEmail(this string s) => s.Contains("@");
}

// usage:
if ("x@y.com".IsValidEmail()) ...    // looks like instance method!

🎯 Closure Gotcha (interview classic)

  • Lambda captures variable by reference, not value
  • In a loop, all lambdas may capture the same variable — use a local copy

📅 Week 6 — Async, Threading, Memory + SOLID

20 Async / Await ⭐

Asynchronous code without callback hell. async marks a method as awaitable; await pauses until the Task completes — without blocking the calling thread.

🎯 Interview-Critical Points

  • Task = a future result (like Promise). Task<T> = future of T.
  • async void only for event handlers — never for normal methods (exceptions get swallowed)
  • Naming convention: async methods end with Async (e.g. GetUserAsync)
  • ConfigureAwait(false) — in libraries, avoid capturing sync context (perf, deadlock prevention)
  • Don't .Result or .Wait() on a Task — causes deadlocks. Always async all the way.
  • CancellationToken — propagate to enable cooperative cancellation
  • Task.WhenAll — run multiple tasks in parallel; WhenAny — first to finish
  • ValueTask — perf optimization for hot paths returning sync result often
public async Task<User> GetUserAsync(int id, CancellationToken ct = default) {
    var response = await _http.GetAsync($"/users/{id}", ct);
    response.EnsureSuccessStatusCode();
    return await response.Content.ReadFromJsonAsync<User>(ct);
}

// parallel calls
var [user, orders, profile] = await Task.WhenAll(
    GetUserAsync(1),
    GetOrdersAsync(1),
    GetProfileAsync(1)
);
⚠️ Gotcha: var x = SomeAsync().Result; in ASP.NET Web API → deadlock. Always await from top to bottom.
🔗 Angular Parallel: Same as TS async/await + Promise. Task ≈ Promise, Task.WhenAll ≈ Promise.all.

21 Multithreading & Parallel Programming

Modern .NET prefers Task-based async over manual Thread management. But you must understand thread safety, locking, and the parallel APIs for CPU-bound work.

🎯 Concepts

  • Thread — OS-level unit of execution (heavyweight)
  • ThreadPool — managed pool, .NET reuses threads (lightweight)
  • Task — abstraction over ThreadPool for async work (preferred)
  • Parallel.For / ForEach — for CPU-bound loops over data
  • PLINQ — .AsParallel() on LINQ for data parallelism
  • lock — mutual exclusion: lock(_obj) { ... }
  • Monitor.Enter/Exit — what lock compiles to
  • Interlocked.Increment — atomic operations
  • SemaphoreSlim — async-friendly semaphore
  • Race condition, deadlock, livelock — be ready to define
  • Concurrent collections: ConcurrentDictionary, ConcurrentQueue, ConcurrentBag
// Parallel CPU-bound work
Parallel.ForEach(images, img => ResizeImage(img));

// PLINQ
var primes = numbers.AsParallel().Where(IsPrime).ToList();

// thread-safe counter
private static int _count = 0;
private static readonly object _lock = new();

void Inc() {
    lock (_lock) { _count++; }      // classic
    Interlocked.Increment(ref _count);  // faster (atomic)
}
⚠️ Gotcha: Don't use lock in async methods (can't await inside). Use SemaphoreSlim.WaitAsync().

22 Memory Management, Garbage Collection & SOLID

.NET has automatic garbage collection but you should understand stack vs heap, generations, and the IDisposable pattern. SOLID principles guide all your class designs.

🎯 Memory Model

  • Stack: value types, method frames, fast, automatic cleanup
  • Heap: reference types, GC managed
  • GC Generations: Gen 0 (new), Gen 1 (survived 1 GC), Gen 2 (long-lived). Higher gens collected less often.
  • Large Object Heap (LOH): objects ≥ 85KB live here, not compacted by default → fragmentation risk
  • Finalizer (~destructor) — runs before GC, slow, avoid unless wrapping unmanaged resources
  • IDisposable — release resources deterministically with using
public class FileLogger : IDisposable {
    private StreamWriter _writer;
    public FileLogger(string path) => _writer = new StreamWriter(path);
    public void Log(string msg) => _writer.WriteLine(msg);
    public void Dispose() => _writer?.Dispose();
}

using (var log = new FileLogger("app.log")) {
    log.Log("hi");
}   // Dispose called automatically

🎯 SOLID Principles ⭐ (memorize one-line definitions)

LetterPrincipleOne-liner
SSingle ResponsibilityOne class, one reason to change.
OOpen/ClosedOpen for extension, closed for modification.
LLiskov SubstitutionSubtypes must be substitutable for their base type.
IInterface SegregationMany small interfaces > one fat interface.
DDependency InversionDepend on abstractions, not concretions.
⚠️ Gotcha: Calling GC.Collect() manually is almost always wrong — interferes with the GC's optimization. Trust the GC.

🎤 Interview Q&A — Phase 1

Q. What is difference between Value Type and Reference Type?

Definition:

  • Value type — hold actual data inside variable
  • Reference type — hold pointer that point to data on heap

Value Type Examples:

  • int, double, bool, char, struct, enum, DateTime
  • Stored on stack (fast, auto cleanup)
  • Copied by value — changes in one not affect other

Reference Type Examples:

  • class, string, array, interface, delegate
  • Stored on heap (managed by GC)
  • Copied by reference — both names point to same object
  • Change visible to all references

Java parallel:

  • Same like Java — primitive (int, boolean) vs Object (class)
  • C# add struct for custom value type — Java has no struct

Note: string is reference type but it is immutable — behave like value type.

Special case — Boxing:

  • When value type stored in object variable — wrapped on heap (slow)
  • Always use generics like List<int> not ArrayList to avoid boxing
Q. Abstract class vs Interface — when to use which?

Definition:

  • Interface — only contract (capability)
  • Abstract class — partial built parent class with shared code + state

Interface — Use when:

  • Define capability ("can do X")
  • Need multiple inheritance — class can implement many interfaces
  • No shared state needed
  • Examples: IDisposable, IEnumerable, IComparable

Abstract class — Use when:

  • Need shared code + shared state for child classes
  • Need constructor
  • Need fields (state)
  • Want "is-a" relationship — single inheritance only

State (fields)YesNo

Java parallel:

  • Same like Java abstract class vs interface
  • Same rules apply — class can implement many interfaces, only one abstract class
Q. What is difference between IEnumerable and IQueryable?

Definition:

  • IEnumerable — sequence in memory (LINQ to Objects)
  • IQueryable — build expression tree, run at data source (database)

Key Points:

  • IEnumerable — operations run in your app memory
  • IQueryable — operations translated to SQL by EF Core
  • IQueryable filter happen AT database (only matching rows come back)
  • IEnumerable filter happen IN memory (all data first loaded)

Performance Example:

  • db.Products.Where(p => p.Price > 100).ToList() — fast (SQL WHERE)
  • db.Products.ToList().Where(p => p.Price > 100) — slow (load all rows first)

Java parallel:

  • IEnumerable = Java Stream API (in-memory)
  • IQueryable = something like JPA Criteria API (translated to SQL)

Rule:

  • Stay on IQueryable as long as possible
  • Call .ToList() at very end
  • Never call .AsEnumerable() too early
Q. What does async/await do under the hood?

Definition:

  • async/await = way to write async code in simple way
  • async keyword mark method as async
  • await keyword pause method till Task complete

How it works:

  • Compiler convert async method into hidden state machine
  • Each await become a checkpoint
  • At await, method pause and thread released back to thread pool
  • Thread can serve other requests
  • When Task complete, method resume from where it paused

Why important:

  • Web app can handle thousands of users with few threads
  • No thread sit blocked waiting for database
  • Better performance and scalability

Java parallel:

  • Task = Java's Future or CompletableFuture
  • Java need ExecutorService + Runnable — C# is simpler
  • Same idea as Java's CompletableFuture but cleaner syntax

Note:

  • Rule = "async all the way" — caller must also be async
  • Never call .Result or .Wait() on Task in web — cause deadlock
Q. What is deadlock and how to prevent?

Definition:

  • Deadlock = two threads wait for each other forever
  • Nothing moves, app freeze

Classic Example:

  • Thread A hold lock X, want lock Y
  • Thread B hold lock Y, want lock X
  • Both wait forever

How to prevent:

  • Always take locks in same order in whole code
  • Use timeout with Monitor.TryEnter — give up after few seconds
  • Use lock-free types like Interlocked.Increment for counters
  • Use ConcurrentDictionary instead of lock + Dictionary
  • Never call .Result on Task in async code

Java parallel:

  • Same like Java deadlock — you already know from Java MultiThreading
  • C# lock(obj){} = Java synchronized(obj){}
  • C# Interlocked.Increment = Java AtomicInteger.incrementAndGet()
Q. Explain SOLID principles?

Definition:

  • SOLID = 5 design rules to make code easy to maintain and test

S — Single Responsibility:

  • One class should do one thing only
  • If UserService handle user + email + log — split it

O — Open/Closed:

  • Open for extension, closed for modification
  • Add new feature by inheritance — not by editing working code

L — Liskov Substitution:

  • Child class must work where parent expected
  • Don't make Penguin (child of Bird) crash when you call Fly()

I — Interface Segregation:

  • Many small interfaces better than one big interface
  • Class should not implement methods it not need

D — Dependency Inversion:

  • Depend on abstraction (interface), not on concrete class
  • This is why .NET DI exist — service depend on IRepository not SqlRepository

Java parallel:

  • Same SOLID principles apply in Java also — same rules
Q. What is boxing and why it is slow?

Definition:

  • Boxing = put value type (like int) in object reference
  • Unboxing = take value out of object back

Why it slow:

  • Box allocate new object on heap
  • Heap allocation cost CPU and memory
  • GC has to clean up the box later
  • In tight loop with millions calls — performance disaster

Classic Trap:

  • Old ArrayList store everything as object
  • Every int added — get boxed

Fix:

  • Use generic collections — List<int>, Dictionary<K,V>, HashSet<T>
  • Generics avoid boxing — store ints directly

Java parallel:

  • Same as Java auto-boxing — int → Integer
  • Java List<Integer> also auto-box
  • C# generics avoid boxing — Java generics use erasure (always box)
Q. What is difference between throw and throw ex?

Definition:

  • throw; — re-throw current exception (keep original stack trace)
  • throw ex; — re-throw but RESET stack trace (lose original error location)

Key Points:

  • Use throw; always — keep stack trace
  • throw ex; hide real source of bug
  • throw ex; make debugging painful
  • throw new MyException("msg", ex) — wrap with inner exception (OK)

Java parallel:

  • Java does not have this problem — Java throw e; always keep stack
  • This is C# specific gotcha

Quick interview answer:

  • "Always use throw; alone"
  • "throw ex; destroy stack trace — almost always bug"
Q. Explain GC generations?

Definition:

  • GC = Garbage Collector — clean unused memory automatically
  • GC group objects by age into 3 generations for efficiency

Generations:

  • Gen 0 — new objects just created (small, scanned often, fast)
  • Gen 1 — survived one GC (medium frequency)
  • Gen 2 — long-lived objects (static cache, singleton) — scanned rarely (expensive)

Why this design:

  • Most objects die young — used briefly, become garbage
  • Gen 0 small and quick to clean
  • Gen 2 large and slow — clean rarely

Special area — LOH (Large Object Heap):

  • Objects ≥ 85KB stored here
  • Treated like Gen 2
  • Not compacted by default — can fragment

Java parallel:

  • Same like Java GC — Young Generation, Old Generation
  • Gen 0/1 = Young Gen (Eden + Survivor)
  • Gen 2 = Old Gen (Tenured)

Note: Never call GC.Collect() in production — interfere with GC tuning.

Q. What is purpose of IDisposable?

Definition:

  • IDisposable = interface with one method Dispose()
  • Used to clean up resources at right time (not wait for GC)

Why need it:

  • For resources GC don't manage well — file handle, DB connection, socket, lock
  • If not released quick — can hit limit (DB connection pool full)
  • App stop working when limits reached

How to use:

  • using statement — auto call Dispose at end of scope
  • Example: using var stream = File.OpenRead(path);
  • Dispose called even if exception thrown
  • Cleaner than try/finally

When to implement:

  • When your class wrap unmanaged resource
  • When your class hold another IDisposable as field

Java parallel:

  • IDisposable = Java AutoCloseable interface
  • using statement = Java try-with-resources
  • Dispose() = Java close()

Modern: IAsyncDisposable with DisposeAsync() — used with await using.

⚠️ Common Mistakes in Phase 1

  • Calling .Result or .Wait() on async tasks → deadlock in web code.
  • Using throw ex; instead of throw; → loses stack trace.
  • Using public fields instead of properties → breaks encapsulation and binary compatibility.
  • Forgetting to dispose IDisposable resources → file/socket leaks.
  • Catching Exception generically and swallowing it.
  • Mutating a List while iterating it (InvalidOperationException).
  • Not understanding closure capture in loops (capturing loop variable instead of value).
  • Boxing in hot loops (using object or non-generic collections).
