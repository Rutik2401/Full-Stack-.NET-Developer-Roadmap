Phase 6 — System Design

The single biggest skill that separates 8 LPA from 15 LPA. You don't need to design YouTube — you need to discuss tradeoffs intelligently and apply patterns to real problems.

⏱ 3 Weeks 14 Topics ⭐ Highest signal for senior roles

📋 Prerequisites

  • Working backend skill (Phase 3)
  • DB experience (Phase 2 + 4)
  • Understanding of HTTP, REST, JSON
  • One real production-style API in your portfolio

📅 Week 1 — Fundamentals

1 Client-Server & HTTP/HTTPS

Every web app is client-server. Client (Angular) requests, server (.NET) responds. HTTPS = HTTP over TLS — encrypted in transit, also authenticates the server.

🎯 Senior-Level Knowledge

  • HTTP/1.1 vs HTTP/2 vs HTTP/3: H2 multiplexes many requests over one TCP connection; H3 uses QUIC over UDP (faster startup, better mobile)
  • TCP vs UDP: TCP reliable + ordered (web, email); UDP fast + lossy (gaming, streaming, DNS)
  • DNS: domain → IP. TTL controls cache. Use SRV/CNAME for service discovery.
  • TLS handshake: cert validation, key exchange, symmetric encryption setup
  • Headers that matter: Cache-Control, ETag, Authorization, Content-Type, CORS headers

2 REST API Design Principles

REST is a style, not a protocol. A "RESTful" API uses HTTP correctly — resources, verbs, status codes — and is stateless.

🎯 The 6 Constraints (Roy Fielding)

  1. Client-Server — separation of concerns
  2. Stateless — server stores no client context
  3. Cacheable — responses must declare cacheability
  4. Layered system — client doesn't see beyond the immediate server
  5. Uniform interface — resources, URIs, verbs, hypermedia
  6. Code on demand (optional) — server can send executable code

🎯 Design Checklist

  • URLs are nouns, not verbs (/users/5/orders not /getOrdersByUser)
  • Plural resource names (/products)
  • Hierarchical for relationships (/users/5/orders/10)
  • Use query params for filter/sort/page (?status=paid&page=2)
  • Versioning: /api/v1/products from day one
  • Return proper status codes — 200/201/204/400/401/403/404/409/500
  • Use ProblemDetails (RFC 7807) for errors
  • Pagination metadata in response envelope
  • Consistent naming (camelCase JSON)
  • HATEOAS (hypermedia links) — optional, rare in practice

3 Stateless vs Stateful — Why It Matters

Stateless servers can be scaled horizontally trivially (add more instances behind a load balancer). Stateful servers need sticky sessions or shared session stores.

🎯 Make Your APIs Stateless

  • Use JWT (token holds claims) instead of server-side sessions
  • Store session/cart in Redis (shared) if you must
  • Never store data in memory that next request needs (any node should serve any request)
  • Idempotent operations (POST/PUT/DELETE retried produce same effect)

4 Monolith vs Microservices

Monolith = single deployable unit. Microservices = many small services. Most successful systems start monolith, split later when pain demands it.

ScalingScale whole appScale specific services

Failure modeAll or nothingPartial degradation
Best forMost products day 1Big org, scale problems

Modular Monolith is often the best of both: single deployment, but well-bounded internal modules ready to extract later.

5 Clean Architecture ⭐ (must-know for 15 LPA)

Layered architecture where dependencies point INWARD. Inner layers (Domain) know nothing about outer (Infrastructure). Enables testability, framework swap, and clear boundaries.

┌─────────────────────────────────────────────┐
│   Web API / UI    (controllers, DTOs)        │  ← outermost
├─────────────────────────────────────────────┤
│   Infrastructure  (EF, file IO, email)       │
├─────────────────────────────────────────────┤
│   Application     (use cases, MediatR, DI)   │
├─────────────────────────────────────────────┤
│   Domain          (entities, business rules) │  ← innermost, no deps
└─────────────────────────────────────────────┘

Dependencies always point INWARD →

🎯 Project Structure (typical)

  • Shop.Domain — Entities, ValueObjects, DomainEvents (zero NuGet refs)
  • Shop.Application — Interfaces, Use Cases (CQRS handlers), DTOs
  • Shop.Infrastructure — EF DbContext, repositories, email/storage adapters
  • Shop.Api — Controllers, middleware, Program.cs (composition root)

🎯 Why Senior Devs Love It

  • Domain logic is independent of frameworks (test without DB or HTTP)
  • Swap EF for Dapper without touching business code
  • Forces dependency inversion (D in SOLID)
  • Aligns with DDD principles

Related Patterns

Onion Architecture — same idea, different naming. Hexagonal (Ports & Adapters) — same intent. DDD — methodology that pairs naturally.

7 Load Balancing

Distributes traffic across multiple servers. Sits between client and your servers.

Algorithms

  • Round-Robin — rotate through servers
  • Least Connections — to least-busy
  • Least Response Time — fastest responding
  • IP Hash — same client → same server (sticky)
  • Weighted — heavier servers get more

L4 vs L7

  • L4 — TCP-level (e.g., AWS NLB, Azure Load Balancer); fast, no header inspection
  • L7 — HTTP-level (e.g., NGINX, Azure Application Gateway, Cloudflare); can route by URL/host/header, do TLS termination, WAF

Health Checks

Load balancer periodically pings /healthz on each server; removes failing nodes. .NET 8: app.MapHealthChecks("/healthz").

8 Caching & Redis ⭐

Caching is the cheapest performance win. Multiple layers — browser, CDN, gateway, app, database.

Cache Layers

  • Browser — Cache-Control, ETag
  • CDN — static assets / cacheable APIs
  • API gateway — output caching
  • App / process — IMemoryCache (in-process, single node)
  • Distributed (Redis) — shared cache across nodes
  • DB query cache — built into RDBMS

Common Strategies

  • Cache-aside (lazy) — check cache, miss → DB → populate cache. Most common.
  • Write-through — every write goes to cache + DB
  • Write-behind — write to cache, async to DB (risk: data loss)
  • Read-through — cache itself fetches from DB on miss

🎯 Cache Invalidation (the hard problem)

  • TTL — set expiry, simplest
  • Manual eviction on writes
  • Pub/sub invalidation across nodes
  • Versioned keys (product:5:v3) — increment on update
// Redis with .NET
builder.Services.AddStackExchangeRedisCache(o => o.Configuration = "localhost:6379");

public class ProductService(IDistributedCache cache, AppDbContext db) {
    public async Task<Product?> GetAsync(int id) {
        var key = $"product:{id}";
        var cached = await cache.GetStringAsync(key);
        if (cached != null) return JsonSerializer.Deserialize<Product>(cached);

        var p = await db.Products.FindAsync(id);
        if (p != null) {
            await cache.SetStringAsync(key, JsonSerializer.Serialize(p),
                new DistributedCacheEntryOptions { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) });
        }
        return p;
    }
}

9 DB Sharding & Replication

Replication (read scaling)

  • One primary handles writes, multiple read replicas handle reads
  • Replication can be sync (consistent, slower) or async (eventually consistent)
  • App routes reads to replicas, writes to primary
  • Provides HA — promote replica if primary fails

Sharding (write scaling)

  • Partition data across multiple databases by a shard key (e.g., user_id % 4)
  • Each shard holds a subset; queries that span shards are expensive
  • Common when single DB can't hold the data or handle writes
  • Adds complexity — only when truly needed (Postgres can scale to TBs vertically first)

Other DB Concepts

  • Connection pooling — reuse connections (built into ADO.NET)
  • Read/write splitting — separate connection strings, route appropriately
  • Materialized views — pre-computed, refresh periodically
  • Partitioning — split a single big table into chunks (by date, range, hash)

🎯 CAP Theorem (interview classic)

Distributed system can guarantee only 2 of: Consistency, Availability, Partition tolerance. Network partitions are inevitable, so you actually choose between CP and AP. Most NoSQL = AP, traditional RDBMS = CP.

🎯 ACID vs BASE

  • ACID (SQL): Atomic, Consistent, Isolated, Durable
  • BASE (NoSQL): Basically Available, Soft state, Eventually consistent

📅 Week 3 — Microservices, Patterns

11 Microservices Architecture

Many small services, each with its own database, communicating via HTTP/gRPC/messages. Powerful — and operationally complex.

Patterns You Must Know

  • API Gateway — single entry point, routes/auth/rate-limit (e.g., Ocelot, YARP, Azure API Management)
  • BFF (Backend for Frontend) — gateway tailored per client (web, mobile)
  • Service Discovery — services find each other (Consul, Eureka, K8s built-in)
  • Service Mesh — sidecar proxies handle traffic/auth/observability (Istio, Linkerd)
  • Database-per-service — each service owns its data; never share DB tables
  • Saga — distributed transaction via choreography or orchestration
  • Circuit Breaker — stop calling a failing service (Polly library)
  • Bulkhead — isolate failures, don't let one service exhaust threads
  • Retry with backoff — Polly handles this
  • Outbox pattern — guarantee message + DB consistency
// Polly resilience policies in .NET
builder.Services.AddHttpClient<OrderClient>()
    .AddPolicyHandler(HttpPolicyExtensions.HandleTransientHttpError()
        .WaitAndRetryAsync(3, attempt => TimeSpan.FromSeconds(Math.Pow(2, attempt))))
    .AddPolicyHandler(HttpPolicyExtensions.HandleTransientHttpError()
        .CircuitBreakerAsync(5, TimeSpan.FromMinutes(1)));

When to NOT Go Microservices

  • Small team (< 20 devs) — coordination cost outweighs benefit
  • Early product (still finding market fit) — iteration speed matters more
  • Tightly coupled domain (every change touches multiple services)

12 Message Queues & Event-Driven

Async communication between services via a broker. Producer puts message, consumer takes — decoupled in time and space.

Brokers

  • RabbitMQ — classic, mature, easy
  • Azure Service Bus — managed, enterprise features (sessions, dedup)
  • Kafka — high-throughput streaming, log-based, used for event sourcing
  • Redis Streams / SQS — lighter alternatives

Patterns

  • Queue (point-to-point) — one consumer processes each message
  • Topic / Pub-Sub — every subscriber gets a copy
  • Dead Letter Queue — failed messages go here for inspection
  • Idempotent consumers — handle duplicates safely

Why Use a Queue

  • Decouple services (producer doesn't wait for consumer)
  • Smooth out traffic spikes (queue absorbs bursts)
  • Retry / DLQ for resilience
  • Fan-out (one event → many handlers)
  • Cross-language interop

In .NET, use MassTransit or NServiceBus as broker abstraction libraries.

13 CQRS & Event Sourcing

CQRS = separate read and write models. Reads use optimized projections, writes go through command handlers. Event Sourcing = store events instead of current state — replay to reconstruct.

CQRS Benefits

  • Reads can use denormalized DB / cache for speed
  • Writes encapsulate business rules in command handlers
  • Independent scaling of reads vs writes
  • Cleaner code — different shapes for different needs
// MediatR-based CQRS in .NET
public record CreateOrderCommand(int UserId, List<int> ProductIds) : IRequest<int>;

public class CreateOrderHandler(AppDbContext db) : IRequestHandler<CreateOrderCommand, int> {
    public async Task<int> Handle(CreateOrderCommand cmd, CancellationToken ct) {
        // business logic, validation, persist
    }
}

// query side
public record GetOrderQuery(int Id) : IRequest<OrderDto?>;

Event Sourcing — when to use

  • Audit trail is a hard requirement (banking, compliance)
  • Need to replay history / fix past data via new projection
  • Ready to invest in operational complexity

Saga Pattern

Coordinates a long-running transaction across services. Two flavors: orchestration (central coordinator like a state machine) and choreography (each service reacts to events).

14 GoF Design Patterns (the ones interviewers ask)

Creational

  • Singleton — single instance (DI container handles this)
  • Factory — encapsulate object creation
  • Builder — fluent API for complex construction
  • Prototype — clone existing object

Structural

  • Adapter — make incompatible interfaces work
  • Decorator — wrap to add behavior (LINQ/HTTP delegating handler are decorators)
  • Facade — simple interface over complex subsystem
  • Proxy — placeholder/wrapper (e.g., EF lazy loading proxy)
  • Composite — tree structures, treat groups as single

Behavioral

  • Strategy — pluggable algorithms (sort comparer, payment provider)
  • Observer — events, pub/sub
  • Command — encapsulate operation as object (CQRS commands)
  • Mediator — central hub (MediatR library)
  • Chain of Responsibility — middleware pipeline
  • Template Method — base defines skeleton, subclass fills steps

🎯 Modern .NET Already Uses These

  • DI container = Service Locator (and IoC)
  • Middleware = Chain of Responsibility
  • EF DbContext = Unit of Work + Repository
  • MediatR = Mediator + Command
  • HttpClient handlers = Decorator

🎤 Interview Q&A — Phase 6

Q. Monolith vs Microservices — which to pick for new product?

Answer: Start with modular monolith, not microservices.

Why NOT Microservices for new product:

  • Microservices solve ORG problems (many teams, independent deploys)
  • Cost = huge operational complexity
  • Service discovery, distributed transactions
  • Observability across many processes
  • Deployment pipeline per service
  • Network debugging hell
  • Small team get crushed by overhead

Why Modular Monolith:

  • One deployment unit — simple
  • Most architectural cleanliness
  • Clear internal boundaries (separate projects, interfaces)
  • No cross-module direct DB access
  • Can extract module to service later when needed
  • Shopify, Stack Overflow, Basecamp run on modular monolith at huge scale

Right Time to Split (microservices):

  • Multiple teams stepping on each other
  • Different scaling profiles for different parts
  • One bad module crashes whole app often

Note: Don't go microservices because it sounds modern.

Q. Explain Clean Architecture?

Definition:

  • Way to organize code in layers
  • Dependencies always point INWARD, never outward
  • Made famous by Uncle Bob

Layers (inner to outer):

  • Domain — pure business entities, value objects, domain rules. No dependencies.
  • Application — use cases (MediatR handlers), DTOs, interfaces (IEmailSender, IRepository)
  • Infrastructure — concrete implementations using EF Core, external APIs, file systems, email
  • Web/API — controllers, middleware, Program.cs

Big Benefits:

  • Testability — Domain + Application have no DB / HTTP dependency, easy unit test
  • Can swap EF for Dapper without touching domain code
  • Enforce SOLID Dependency Inversion principle
  • High-level policy don't depend on low-level details

Java parallel:

  • Same layered approach as Java enterprise apps
  • POJO + Service + Repository = similar layering

Note: Onion Architecture and Hexagonal Architecture = same idea, different names.

Q. How to scale a read-heavy API?

Approach: Attack in layers, cheapest first.

Step 1 — Caching (solve 80% of problems):

  • In-process cache (IMemoryCache) for hot static data (product categories)
  • Distributed cache (Redis) for cross-instance data (product details)
  • Cache-aside pattern with reasonable TTL

Step 2 — Read Replicas:

  • Route reads to replicas, writes to primary
  • EF support multiple connection strings

Step 3 — Denormalize Hot Reads:

  • Materialized views, pre-aggregated tables
  • Or separate read model in CQRS style

Step 4 — CDN for Cacheable Endpoints:

  • Set Cache-Control headers correctly
  • Even APIs can be CDN-cached

Step 5 — Horizontal Scale:

  • Multiple stateless app instances behind load balancer
  • Autoscale on CPU or queue depth

Step 6 — Search Engine:

  • Elasticsearch for read-heavy filtering
  • Event-driven projections

Note: Order matters — caching solve 80% of problems with 20% of effort.

Q. What is CAP theorem?

Definition:

  • In distributed system — can guarantee at most 2 of 3 properties

3 Properties:

  • C — Consistency — every read see most recent write
  • A — Availability — every request get response (even if not latest)
  • P — Partition tolerance — system work when network between nodes break

The Trick:

  • Network partitions inevitable in any distributed system
  • Cannot drop P
  • Real choice = CP or AP

CP — Consistency + Partition tolerance:

  • Refuse some requests to keep data consistent
  • Example: traditional SQL DBs in cluster mode
  • Use for banking transactions

AP — Availability + Partition tolerance:

  • Keep responding with possibly stale data
  • Example: Cassandra, DynamoDB, most NoSQL
  • Use for social media feeds

BASE Model (NoSQL counterpart to ACID):

  • Basically Available, Soft state, Eventual consistency
Q. When to use message queue?

Definition:

  • Decouple producer from consumer in time, scale, technology
  • Examples: RabbitMQ, Azure Service Bus, Kafka

5 Scenarios Where Queues Shine:

  1. Async background work — send emails, generate PDFs, process images
  2. Traffic smoothing — queue absorb spikes, consumer process at steady rate
  3. Fan-out — one event ("OrderPlaced") trigger many handlers (email, analytics, inventory)
  4. Resilience — if consumer down, messages wait safely. Dead Letter Queue for failed
  5. Cross-language interop — .NET service talk to Python ML service via JSON

When NOT to use:

  • Synchronous request/response
  • When you need immediate answer

Java parallel:

  • Same as JMS, ActiveMQ in Java world
  • Same producer-consumer pattern

Note: Queue add complexity — use only when async makes sense.

Q. What is Circuit Breaker pattern?

Definition:

  • Resilience pattern to prevent cascading failures
  • Like electrical breaker in your house
  • Stop hammering failing dependency

3 States:

  • Closed — normal operation, calls go through. If failures cross threshold (5 errors in 30 sec) — trip to Open
  • Open — every call fail fast without trying. Save resources, give dependency time to recover. After cooldown (1 min) — go Half-Open
  • Half-Open — next call allowed as test. Success → Closed. Fail → back to Open

Big Benefit:

  • Instead of 1000 threads waiting on dead API for 30 sec each
  • Calls fail in milliseconds
  • App stay responsive

.NET Implementation:

  • Polly library — easy integration with HttpClient

Java parallel:

  • Same as Hystrix or Resilience4j in Java

Note: Always combine with retry (exponential backoff) and timeout for full resilience.

Q. Explain CQRS in simple words?

Definition:

  • CQRS = Command Query Responsibility Segregation
  • Separate code for writes from code for reads

Commands (Write):

  • Operations like CreateOrder, UpdateProduct, DeleteUser
  • Go through dedicated handler classes
  • Contain validation, business rules
  • Don't return data, just success/failure

Queries (Read):

  • Operations like GetOrder, ListProducts
  • Own handlers
  • Read from optimized projections (denormalized or cached)

Benefits:

  • Reads and writes scale independently
  • Different complexity profiles
  • Reads can be denormalized for speed
  • Writes can be richly modeled with business rules

.NET Tool:

  • MediatR library — controllers call mediator.Send(command)

Note: CQRS doesn't always mean separate databases — can be just code pattern. When paired with Event Sourcing, get truly separate read/write storage.

Q. What is N+1 problem at system level?

Definition:

  • Same N+1 problem you see in EF Core, but at service boundaries

Pattern:

  • One API call return list of items
  • For each item — client make follow-up call
  • List of 50 products = 51 HTTP requests total
  • Network latency stack up
  • Page feel slow, server pounded

4 Fixes:

  1. Batch endpoints — return all needed data in one call: GET /products?include=reviews,inventory
  2. GraphQL — client specify exact fields, resolver batch DB calls
  3. BFF (Backend for Frontend) — server aggregator calls microservices and pre-assemble data
  4. DataLoader pattern — server-side batching of similar lookups within request

Goal: Fewer round trips between client and server.

Q. How to design URL shortener?

Step 1 — Clarify Requirements:

  • How many URLs created per day?
  • How many redirects per day?
  • Custom aliases? Expiry? Click analytics?
  • Assume: 10M new URLs/day, 100M redirects/day (read-heavy)

Step 2 — Core Endpoints:

  • POST /shorten — take long URL, generate code, store, return short URL
  • GET /:code — lookup long URL, return 301 redirect

Step 3 — Short Code Generation:

  • Base62 encoding (a-z, A-Z, 0-9 = 62 chars)
  • Of auto-incrementing ID
  • 7-char codes give 3.5 trillion URLs

Step 4 — Storage:

  • Write to SQL DB for durability
  • Populate Redis cache for fast reads
  • Redirect path: check Redis first, fall back to DB on miss

Step 5 — Performance:

  • CDN for hot codes at edge
  • Queue write traffic if spikes are bursty

Step 6 — Click Analytics:

  • Async events into queue
  • Worker process into separate analytics DB
  • Don't slow down redirects

Step 7 — Abuse Prevention:

  • Rate limit by IP
  • Validate URLs against malware blocklists
Q. Why statelessness important for scaling?

Definition:

  • Stateless server don't store per-user data in memory between requests
  • Every request carry everything needed (JWT with user identity)

Benefits:

  • ANY server instance can handle ANY request — interchangeable
  • Horizontal scaling trivial — spin up 10 more instances
  • No special configuration needed
  • If server crash — no user state lost
  • Next request go to another instance

vs Stateful Server:

  • Store session data in memory (cart contents)
  • Load balancer need "sticky sessions"
  • Same user always go to same server
  • Limit scaling, problem if server die

How to make API Stateless:

  • Push state to client — JWT for auth, browser localStorage for cart
  • Push state to shared store — Redis, database

Result: Architecture scale from 10 to 10,000 servers without code change.

⚠️ Common Mistakes

  • Choosing microservices for a small product — operational pain >> benefit
  • Sharing a DB across microservices (defeats the entire point)
  • Caching without an eviction strategy → stale data forever
  • Synchronous HTTP everywhere → cascading failures, latency stacks
  • Ignoring CAP — promising both consistency AND availability across regions
  • Adding NoSQL because it's "modern" when SQL would do the job better
  • Designing for scale you don't have (premature optimization)
