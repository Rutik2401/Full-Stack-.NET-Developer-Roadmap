2 Networking Basics

  • IP / Subnet — IPv4 (192.168.1.10/24), IPv6, public vs private
  • DNS — domain → IP. A record, CNAME, MX, TXT, SRV
  • Ports — 80 HTTP, 443 HTTPS, 22 SSH, 5432 Postgres, 1433 SQL Server, 6379 Redis
  • Firewall — allow/deny rules; cloud SGs / NSGs
  • NAT — many private IPs → one public
  • VPN / VNet — private network in cloud
  • TLS / SSL — cert from Let's Encrypt or Azure-managed
  • HTTP status codes + headers (Cache-Control, ETag, CORS)
  • WebSockets — full-duplex over HTTP upgrade (SignalR uses this)
  • CDN — Azure CDN / Cloudflare for static asset edge caching

  • Squash-merge for clean history; rebase for shared branches

Docker Building Blocks

  • Image — read-only template (your code + runtime)

5 Dockerfile (multi-stage build for .NET)

# --- BUILD STAGE ---
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# copy csproj first to leverage layer cache for restore
COPY *.sln .
COPY Shop.Api/*.csproj Shop.Api/
COPY Shop.Application/*.csproj Shop.Application/
COPY Shop.Domain/*.csproj Shop.Domain/
COPY Shop.Infrastructure/*.csproj Shop.Infrastructure/
RUN dotnet restore

# copy source & publish
COPY . .
RUN dotnet publish Shop.Api/Shop.Api.csproj -c Release -o /app/publish --no-restore

# --- RUNTIME STAGE (small, no SDK) ---
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS runtime
WORKDIR /app
COPY --from=build /app/publish .
EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080
ENTRYPOINT ["dotnet", "Shop.Api.dll"]
# build & run
docker build -t shop-api:latest .
docker run -d -p 8080:8080 --name api shop-api:latest

# inspect
docker ps                       # running
docker logs -f api
docker exec -it api bash        # shell inside
docker stop api && docker rm api

# push to registry
docker tag shop-api:latest myacr.azurecr.io/shop-api:1.0
docker push myacr.azurecr.io/shop-api:1.0

🎯 Best Practices

  • Multi-stage builds — small final image (no SDK in prod)
  • Layer caching — copy .csproj + restore BEFORE copying source
  • Use .dockerignore (like .gitignore)
  • Don't run as root in production (USER app)
  • Pin image versions (aspnet:8.0 not aspnet:latest)
  • Health checks: HEALTHCHECK instruction

6 Docker Compose — Multi-Container Apps

Define multi-container app declaratively. docker-compose up brings everything up; down tears it all down.

# docker-compose.yml — API + SQL Server + Redis
version: '3.9'
services:
  api:
    build: ./Shop.Api
    ports: ["8080:8080"]
    environment:
      ConnectionStrings__Default: "Server=db;Database=Shop;User=sa;Password=Pass@word1;TrustServerCertificate=true"
      Redis__Connection: "redis:6379"
    depends_on: [db, redis]

  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      ACCEPT_EULA: Y
      SA_PASSWORD: Pass@word1
    ports: ["1433:1433"]
    volumes: [sqldata:/var/opt/mssql]

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]

volumes:
  sqldata:
docker compose up -d
docker compose logs -f api
docker compose down -v       # -v removes volumes

9 Azure App Service — host your .NET API

PaaS for web apps. Push code or container, Azure runs it. Auto-scale, slots, custom domains, free SSL.

# create plan + web app
az appservice plan create -g rg-shop -n plan-shop --sku B1 --is-linux
az webapp create -g rg-shop -p plan-shop -n shop-api-rutik --runtime "DOTNETCORE:8.0"

# deploy zip
dotnet publish -c Release -o ./publish
cd publish && zip -r ../app.zip . && cd ..
az webapp deploy -g rg-shop -n shop-api-rutik --src-path app.zip --type zip

# OR deploy from container
az webapp config container set -g rg-shop -n shop-api-rutik \
  --container-image-name myacr.azurecr.io/shop-api:1.0

# app settings (env vars)
az webapp config appsettings set -g rg-shop -n shop-api-rutik \
  --settings ConnectionStrings__Default="Server=..." Jwt__Key="..."

Killer Features

  • Deployment slots — staging slot for blue/green deploys, swap with zero downtime
  • Auto-scale rules (CPU, queue length, schedule)
  • Custom domains + free Managed Certificates
  • Always On setting (don't unload after idle)
  • Application Insights integration in one click

10 Azure SQL Database

Managed SQL Server in Azure — backups, HA, scaling all handled. Connect via standard connection string.

az sql server create -g rg-shop -n sql-shop-rutik -l centralindia \
  -u sqladmin -p "Pass@word1234"

# firewall rule (allow your IP)
az sql server firewall-rule create -g rg-shop -s sql-shop-rutik \
  -n MyIp --start-ip-address 0.0.0.0 --end-ip-address 0.0.0.0   # 0.0.0.0 = allow Azure svc

az sql db create -g rg-shop -s sql-shop-rutik -n shop-db --service-objective S0

Tiers (high-level)

  • DTU model — Basic, Standard (S0–S12), Premium
  • vCore model — General Purpose, Business Critical, Hyperscale
  • Serverless — auto-pause, pay per use

Connection string is just SQL Server format — your EF Core code works unchanged. Use Managed Identity + Azure AD auth instead of passwords in prod.

12 Key Vault & Managed Identity ⭐

Store secrets (DB passwords, JWT keys, API keys) in Key Vault. Use Managed Identity so your App Service authenticates to Key Vault without storing credentials anywhere.

az keyvault create -g rg-shop -n kv-shop-rutik
az keyvault secret set --vault-name kv-shop-rutik -n SqlConnString --value "..."

# enable managed identity on App Service
az webapp identity assign -g rg-shop -n shop-api-rutik

# grant access
az keyvault set-policy -n kv-shop-rutik --object-id <app-identity> --secret-permissions get list
// .NET reads Key Vault as configuration
builder.Configuration.AddAzureKeyVault(
    new Uri("https://kv-shop-rutik.vault.azure.net/"),
    new DefaultAzureCredential());

// then use as normal
var conn = builder.Configuration["SqlConnString"];

This is the gold standard for secret management — no credentials in code, config, or env vars.

13 Azure Functions & Service Bus

Serverless compute — pay per execution, scale to zero. Triggers: HTTP, Timer, Queue, Blob, Service Bus, etc.

[Function("ProcessOrder")]
public async Task Run(
    [ServiceBusTrigger("orders", Connection = "SbConn")] Order order,
    FunctionContext ctx) {

Service Bus Patterns

  • Queue — point-to-point messages
  • Topic + Subscriptions — pub/sub with filtering

14 Application Insights — Observability

Auto-collects requests, dependencies, exceptions, traces, custom metrics. Powerful query language (Kusto/KQL).

builder.Services.AddApplicationInsightsTelemetry();

// custom event
_telemetry.TrackEvent("OrderPlaced", new() { ["OrderId"] = order.Id.ToString() });
// KQL query — top errors last 24h
exceptions
| where timestamp > ago(24h)
| summarize count() by problemId
| order by count_ desc
| take 10

What to Monitor

  • RED metrics: Rate, Errors, Duration (per endpoint)
  • USE metrics: Utilization, Saturation, Errors (per resource)
  • Dependency call latency (DB, external APIs)
  • Exceptions with stack traces
  • Custom business events (signups, conversions)
  • Set alerts on critical thresholds (5xx rate, p99 latency, queue depth)

15 GitHub Actions — Automate Everything

# .github/workflows/ci-cd.yml
name: CI/CD
on:
  push: { branches: [main] }
  pull_request: { branches: [main] }

jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with: { dotnet-version: '8.0' }
      - run: dotnet restore
      - run: dotnet build --no-restore -c Release
      - run: dotnet test --no-build -c Release --logger trx
      - uses: actions/upload-artifact@v4
        with: { name: testresults, path: '**/TestResults/*.trx' }

  deploy:
    needs: build-test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with: { dotnet-version: '8.0' }
      - run: dotnet publish Shop.Api -c Release -o ./publish
      - uses: azure/login@v2
        with: { creds: ${{ secrets.AZURE_CREDENTIALS }} }
      - uses: azure/webapps-deploy@v3
        with:
          app-name: shop-api-rutik
          package: ./publish

🎤 Interview Q&A — Phase 7

Q. Container vs VM?

Definition: Both isolate and run software but work differently.

Compare Table:

AspectVMContainer
IncludesFull guest OSJust process on host OS
KernelOwn kernelShare host kernel
SizeGBsMBs
BootMinutesSeconds
Density per hostFew dozensHundreds
IsolationStrongSlightly weaker

VM (Virtual Machine):

  • Run on hypervisor (Hyper-V, VMware)
  • Has own kernel, drivers, system services

Container:

  • Just a process on host OS
  • Isolated using Linux namespaces and cgroups
  • No second OS inside

Container Benefits:

  • Same environment from laptop to production
  • Easy horizontal scaling
  • Consistent deployments

Note: VMs still used for full OS isolation or different OS than host.

Q. What is multi-stage Docker build?

Definition:

  • Multi-stage = use multiple FROM in one Dockerfile
  • Produce smaller and more secure final image
  • Essential for compiled languages like C#

How it works:

  1. First stage use .NET SDK image (compiler, MSBuild, tools)
  2. Restore packages and publish app
  3. Second stage use smaller ASP.NET runtime image
  4. Copy ONLY compiled output from first stage
  5. Build tools, source code never in final image

Size Difference:

  • SDK image — around 800 MB
  • Runtime image — around 200 MB
  • Final container = 4x smaller

Benefits:

  • Faster pulls in CI/CD
  • Faster deploys
  • Lower bandwidth costs
  • Lower attack surface

Layer Caching Tip:

  • Copy .csproj first, run dotnet restore
  • Then copy source
  • Docker cache restore layer — skip on rebuild when only source change

Note: Standard pattern for production .NET Dockerfiles.

Q. Singleton, Scoped, Transient at Azure App Service level?

Basic Rule:

  • Same .NET DI lifetime rules apply
  • BUT important catch when scale out on Azure App Service

Catch:

  • App Service can run app on multiple instances behind load balancer
  • Singleton = one instance PER PROCESS (not whole app)
  • 5 App Service instances = 5 separate Singleton objects
  • In-memory Singleton cache only visible on one instance

For Cross-Instance Sharing:

  • Redis — cache and SignalR backplane
  • Azure Service Bus — messaging
  • Database — persistent state

Scoped:

  • Still once per HTTP request
  • Works fine — request lands on one instance

Transient:

  • Same — new every injection

Mental Shift: Think Singleton = "one per process", not "one for whole app".

Q. How to secure secrets in Azure?

Gold Standard: Azure Key Vault + Managed Identity.

4 Setup Steps:

  1. Create Key Vault, store all secrets (DB conn, JWT keys, API keys)
  2. Enable System-Assigned Managed Identity on App Service
  3. Grant Managed Identity Get + List permission on Key Vault
  4. In .NET: builder.Configuration.AddAzureKeyVault(vaultUri, new DefaultAzureCredential())

How it works:

  • Managed Identity gives App Service Azure AD identity automatically
  • No credentials to handle
  • DefaultAzureCredential auto use Managed Identity on Azure

Result:

  • Zero credentials in source code
  • Zero credentials in app settings
  • Zero credentials in env variables
  • Secret lifecycle (rotation, audit) managed by Key Vault

Local Development:

  • DefaultAzureCredential fall back to Visual Studio / Azure CLI login
  • Same code work everywhere
Q. What is CI/CD in plain English?

CI — Continuous Integration:

  • Every code push trigger build and run all tests
  • Catch problems early (minutes not weeks)
  • Developers push small changes often
  • Bugs surface while code fresh in head

CD — Continuous Deployment:

  • When CI pass on main branch — auto deploy
  • Deploy to staging environment
  • Run more tests there
  • Push to production with no human

Continuous Delivery vs Deployment:

  • Delivery = manual approval before prod
  • Deployment = full auto, no approval

Typical CI/CD Pipeline Steps:

  1. Lint / format check
  2. Restore packages
  3. Build
  4. Run unit tests
  5. Static security analysis
  6. Build Docker image
  7. Push to registry
  8. Deploy to staging
  9. Smoke tests
  10. Swap to production
  11. Final smoke tests
  12. Notify team (Slack)

Tools: GitHub Actions, Azure DevOps Pipelines, GitLab CI, Jenkins.

Result: Ship multiple times a day with confidence.

Q. How to achieve zero-downtime deployment in App Service?

Answer: Use deployment slots.

What is a Slot:

  • Separate web app sharing App Service Plan
  • Has own URL and configuration

Flow Steps:

  1. Create "staging" slot alongside production
  2. Deploy new build to staging slot
  3. Slot warm up — load app, JIT-compile, fill caches, open DB connections
  4. Run smoke tests against staging URL
  5. Perform swap — Azure switch routing atomically
  6. Production traffic go to new slot (already warm)
  7. Old production become new staging

Benefits:

  • No downtime — no failed requests, no spinner of death
  • Fast and seamless (new instances already warm)
  • Instant rollback — swap back if issue
  • Within seconds back on old version

Note: Configure slot-specific settings (like connection strings) for things that should not swap.

Q. What is Application Insights?

Definition:

  • Azure's APM (Application Performance Monitoring) service
  • Auto-instrument .NET app

What it Collects:

  • Requests — every HTTP request with duration, status, URL, user
  • Dependencies — every outgoing call (DB, Redis, API) with timing
  • Exceptions — every unhandled exception with stack trace and context
  • Traces — your ILogger messages
  • Custom events — events you fire from code

Query Language:

  • KQL (Kusto Query Language) — powerful SQL-like for log data

Features:

  • Build dashboards
  • Set alerts ("5xx error rate over 1% for 5 minutes")
  • Live metrics streaming
  • Track end-to-end transactions across services

Setup (one line):

builder.Services.AddApplicationInsightsTelemetry()

Without APM:

  • Flying blind in production
  • Only hear problems from angry users

With App Insights:

  • See latency p99 on every endpoint
  • Slowest DB queries
  • Most common errors
  • Impact of each deploy
Q. Azure Functions vs App Service?

Both run code on Azure but very different billing/lifecycle.

Compare Table:

AspectApp ServiceFunctions
TypeLong-running web appServerless function
BillingHourly rate (always)Per execution
Always onYes (24/7)Scale to zero
TriggerHTTP requestsHTTP, queue, blob, timer
Use forWeb APIs, websitesShort tasks, cron, webhooks

App Service:

  • Provision a Plan (B1, S1, P1v2 — VM tier)
  • Pay even with no traffic
  • Always on, ready to serve

Azure Functions:

  • Code = function trigger by event
  • Consumption plan — pay per execution + request count
  • Auto scale 0 to thousands
  • No server to manage
  • Trade-off: cold start latency on first call after idle
  • Limit on max execution time

Real Pattern: Use BOTH — App Service for main API, Functions for background processors and event handlers.

Q. What is Managed Identity?

Definition:

  • Azure AD identity auto created for Azure resource
  • Removes need to store credentials in code/config

2 Flavors:

  • System-Assigned — tied to one resource, delete when resource deleted. Easiest — toggle on in portal
  • User-Assigned — standalone, attach to multiple resources

How to Use:

  1. Enable Managed Identity on App Service
  2. Go to other resources (Key Vault, Storage, SQL DB)
  3. Grant identity permissions through their access controls
  4. In code: use DefaultAzureCredential from Azure.Identity NuGet

Result:

  • No connection strings with passwords
  • No API keys checked into git
  • No credentials anywhere
  • Auditing built in — Azure log every action

Note: Modern best practice for any Azure-hosted .NET app.

Q. Docker Compose vs Kubernetes?

Both orchestrate containers but very different scales.

Docker Compose:

  • For local development and single-host
  • Write docker-compose.yml file
  • Describe services (API, DB, Redis) with image, ports, env, dependencies
  • docker compose up — bring stack up
  • docker compose down — tear down
  • Perfect for local dev environment

Kubernetes (K8s):

  • Production-grade orchestration
  • Across cluster of many machines
  • Schedule pods (groups of containers) onto nodes
  • Auto restart when containers crash
  • Rolling updates with zero downtime
  • Auto scale based on CPU/memory/custom metrics
  • Service discovery, load balancing
  • Secret management, persistent volumes

Concepts to Know:

  • Pods, Deployments, Services, Ingress, ConfigMaps
  • Steep learning curve

Azure Options:

  • AKS — managed Kubernetes
  • Azure Container Apps — simpler serverless container option

Note: For 12-15 LPA .NET roles you don't need to operate K8s, but understand concepts.

⚠️ Common Mistakes

  • Pinning :latest tag → unpredictable rebuilds
  • Single-stage Dockerfile → shipping the SDK to prod (huge, insecure)
  • Storing secrets in appsettings.json in source control
  • Running container as root
  • Not pinning .NET runtime version → image rebuilds break
  • Calling db.Database.Migrate() from multiple App Service instances at startup → race
  • Forgetting Always On in App Service → first request after idle is super slow
  • Not setting up health checks → load balancer routes traffic to dead instances
  • No App Insights / log aggregation → blind in production
