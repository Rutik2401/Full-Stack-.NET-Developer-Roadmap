📅 Week 1 — Fundamentals

1 Program.cs & The Hosting Model

In .NET 6+, Program.cs is the single entry point. The WebApplicationBuilder configures services + middleware in two clear sections (Build phase + Run phase).

var builder = WebApplication.CreateBuilder(args);

// 1️⃣ REGISTER SERVICES (DI container)
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(builder.Configuration.GetConnectionString("Default")));
builder.Services.AddScoped<IUserService, UserService>();

var app = builder.Build();

// 2️⃣ CONFIGURE MIDDLEWARE PIPELINE (order matters!)
if (app.Environment.IsDevelopment()) {
    app.UseSwagger();
    app.UseSwaggerUI();
}
app.UseHttpsRedirection();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();

app.Run();

🎯 Hosting Model Facts

  • Kestrel — built-in cross-platform web server (default)
  • IIS / Nginx — reverse proxy in front of Kestrel for production
  • WebApplication = simplified host (.NET 6+); replaces older Startup.cs pattern
  • Configuration auto-loads appsettings.json, appsettings.{Environment}.json, env vars, command line
🔗 Angular Parallel: builder.Services.AddXxx() ≈ Angular's NgModule.providers. Middleware pipeline ≈ Angular HTTP interceptor chain.

2 Configuration & Options Pattern

Settings live in appsettings.json (per-environment). Read via IConfiguration directly, or — better — bind to a strongly-typed class via the Options pattern.

// appsettings.json
{
  "ConnectionStrings": { "Default": "Server=...;Database=Shop;..." },
  "Jwt": { "Issuer": "shop-api", "Key": "super-secret", "ExpiryMinutes": 60 }
}

// strongly typed
public class JwtOptions {
    public string Issuer { get; set; } = "";
    public string Key { get; set; } = "";
    public int ExpiryMinutes { get; set; }
}

// register
builder.Services.Configure<JwtOptions>(builder.Configuration.GetSection("Jwt"));

// inject
public class TokenService(IOptions<JwtOptions> opts) {
    private readonly JwtOptions _jwt = opts.Value;
}

🎯 IOptions vs IOptionsSnapshot vs IOptionsMonitor

  • IOptions<T> — singleton, read once at startup
  • IOptionsSnapshot<T> — scoped, re-binds per request (config can change)
  • IOptionsMonitor<T> — singleton + listens for changes (notifications)
⚠️ Gotcha: Never store secrets (passwords, JWT keys) in appsettings.json in source control. Use User Secrets (dev) and Azure Key Vault (prod).

📅 Week 2 — Web API Core

3 Controllers, Actions & Routing

Controllers handle HTTP requests. Each public method (action) maps to a route. Modern Web API uses attribute routing.

[ApiController]
[Route("api/[controller]")]   // → /api/products
public class ProductsController : ControllerBase {
    private readonly IProductService _svc;
    public ProductsController(IProductService svc) => _svc = svc;

    [HttpGet]                              // GET /api/products
    public async Task<IActionResult> GetAll() => Ok(await _svc.GetAll());

    [HttpGet("{id:int}")]               // GET /api/products/5  (constraint)
    public async Task<IActionResult> Get(int id) {
        var p = await _svc.Get(id);
        return p is null ? NotFound() : Ok(p);
    }

    [HttpPost]
    public async Task<IActionResult> Create([FromBody] CreateProductDto dto) {
        var created = await _svc.Create(dto);
        return CreatedAtAction(nameof(Get), new { id = created.Id }, created);
    }

    [HttpPut("{id:int}")]
    public async Task<IActionResult> Update(int id, [FromBody] UpdateProductDto dto) {
        await _svc.Update(id, dto);
        return NoContent();
    }

    [HttpDelete("{id:int}")]
    public async Task<IActionResult> Delete(int id) {
        await _svc.Delete(id);
        return NoContent();
    }
}

🎯 [ApiController] gives you

  • Auto model validation (returns 400 on invalid)
  • Binding source inference ([FromBody] for complex types, [FromRoute] for route params)
  • Problem-Details responses (RFC 7807) for errors
  • Required attribute routing

Route Constraints

  • {id:int}, {guid:guid}, {slug:alpha}, {date:datetime}, {name:minlength(3)}
🔗 Angular Parallel: [Route] ≈ Angular routing config. [HttpGet] attribute ≈ Angular's route definition. Constructor injection is identical.

4 Model Binding & Validation

Model Binding maps HTTP request data (URL, body, headers) to action parameters. Validation runs via Data Annotations or FluentValidation.

Binding Sources

  • [FromBody] — JSON request body (one per action)
  • [FromRoute] — URL segment
  • [FromQuery] — query string
  • [FromHeader] — HTTP header
  • [FromForm] — form data (file upload)
  • [FromServices] — pull from DI directly into action
public class CreateUserDto {
    [Required, EmailAddress]
    public string Email { get; set; } = "";

    [Required, StringLength(50, MinimumLength = 8)]
    public string Password { get; set; } = "";

    [Range(18, 120)]
    public int Age { get; set; }
}

[HttpPost]
public IActionResult Register([FromBody] CreateUserDto dto) {
    // [ApiController] auto-returns 400 with ProblemDetails if invalid
    return Ok();
}

// FluentValidation alternative — better for complex rules
public class CreateUserValidator : AbstractValidator<CreateUserDto> {
    public CreateUserValidator() {
        RuleFor(x => x.Email).NotEmpty().EmailAddress();
        RuleFor(x => x.Password).MinimumLength(8).Matches("[A-Z]").Matches("[0-9]");
    }
}

5 DTOs & AutoMapper

Never expose DB entities directly via API — use DTOs (Data Transfer Objects). They decouple wire format from storage and prevent over-posting attacks.

// Entity (DB)
public class User {
    public int Id { get; set; }
    public string Email { get; set; } = "";
    public string PasswordHash { get; set; } = "";   // 🚨 don't expose
    public DateTime CreatedAt { get; set; }
}

// DTO (API)
public record UserDto(int Id, string Email, DateTime CreatedAt);

// AutoMapper profile
public class MappingProfile : Profile {
    public MappingProfile() {
        CreateMap<User, UserDto>();
        CreateMap<CreateUserDto, User>();
    }
}

builder.Services.AddAutoMapper(typeof(MappingProfile));

// usage
var dto = _mapper.Map<UserDto>(user);

🎯 Why DTOs Matter (interview answer)

  • Prevents over-posting (client sets fields they shouldn't, e.g., IsAdmin)
  • Hides sensitive fields (PasswordHash, internal IDs)
  • Decouples API contract from DB schema — can refactor DB without breaking clients
  • Keeps response payload small (fetch what you need)

📅 Week 3 — DI, Middleware, Filters

8 Dependency Injection ⭐

DI is built into ASP.NET Core. Services are registered with a lifetime; the framework supplies them via constructor parameters.

LifetimeCreatedUse For
SingletonOnce per appStateless config / cache / logger
ScopedOnce per HTTP requestDbContext, request-bound services
TransientEvery injectionLightweight stateless helpers
 
builder.Services.AddSingleton<IClock, SystemClock>();
builder.Services.AddScoped<IUserService, UserService>();
builder.Services.AddTransient<IEmailSender, EmailSender>();

// constructor injection
public class UserService(AppDbContext db, IEmailSender mailer, ILogger<UserService> log) {
    // db is Scoped, mailer Transient, log Singleton — all injected automatically
}

🎯 Captive Dependency Bug

  • Never inject Scoped (DbContext) into Singleton — it gets "captured" and lives forever, breaking thread safety
  • Compiler/runtime won't catch this. ValidateScopes in Development helps surface it.
🔗 Angular Parallel: Identical concept!
AddSingleton ≈ providedIn: 'root'
AddScoped ≈ providing per-route module
AddTransient ≈ creating new instance each injection.

9 Middleware Pipeline ⭐

Middleware are components that handle requests in a pipeline. Each can short-circuit or call the next. Order matters — common interview question.

app.UseHttpsRedirection();    // 1. force HTTPS
app.UseStaticFiles();         // 2. serve wwwroot
app.UseRouting();             // 3. match endpoints
app.UseCors("AllowAll");       // 4. CORS
app.UseAuthentication();      // 5. who are you?
app.UseAuthorization();       // 6. are you allowed?
app.UseMiddleware<ExceptionMiddleware>();  // 7. catch unhandled
app.MapControllers();         // 8. dispatch to controllers

Custom Middleware (3 styles)

// 1. Inline
app.Use(async (ctx, next) => {
    Console.WriteLine($"> {ctx.Request.Path}");
    await next();
    Console.WriteLine($"< {ctx.Response.StatusCode}");
});

// 2. Class-based (preferred for production)
public class RequestLoggingMiddleware {
    private readonly RequestDelegate _next;
    public RequestLoggingMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext ctx, ILogger<RequestLoggingMiddleware> log) {
        var sw = Stopwatch.StartNew();
        await _next(ctx);
        log.LogInformation("{Method} {Path} → {Status} in {Ms}ms",
            ctx.Request.Method, ctx.Request.Path, ctx.Response.StatusCode, sw.ElapsedMilliseconds);
    }
}
app.UseMiddleware<RequestLoggingMiddleware>();
⚠️ Gotcha: UseAuthentication MUST come before UseAuthorization. UseRouting must come before both.
🔗 Angular Parallel: Like HTTP interceptors. The pipeline is identical: each interceptor processes request → calls next → processes response.

10 Filters (Action, Exception, Authorization, Resource)

Filters run inside MVC pipeline (after routing). Use them for cross-cutting concerns scoped to controllers.

  • Authorization filter — runs first, checks auth
  • Resource filter — wraps the rest, used for caching
  • Action filter — runs before/after action method
  • Exception filter — catches exceptions in actions
  • Result filter — runs around result execution
public class LogActionFilter : IAsyncActionFilter {
    public async Task OnActionExecutionAsync(ActionExecutingContext ctx, ActionExecutionDelegate next) {
        Console.WriteLine($"> {ctx.ActionDescriptor.DisplayName}");
        var result = await next();
        Console.WriteLine($"< {ctx.ActionDescriptor.DisplayName}");
    }
}

[ServiceFilter(typeof(LogActionFilter))]
[HttpGet] public IActionResult Get() => Ok();

📅 Week 4 — Logging & Errors

11 Logging — ILogger & Serilog

Inject ILogger<T> anywhere. Use structured logging with placeholders, not string concatenation, for searchable logs.

// built-in
public class OrderService(ILogger<OrderService> log) {
    public void Place(Order o) {
        log.LogInformation("Placing order {OrderId} for user {UserId}", o.Id, o.UserId);
        try { ... }
        catch (Exception ex) {
            log.LogError(ex, "Failed to place order {OrderId}", o.Id);
            throw;
        }
    }
}

Log Levels (low → high)

  • Trace, Debug, Information, Warning, Error, Critical

Serilog Setup (recommended)

builder.Host.UseSerilog((ctx, cfg) => cfg
    .ReadFrom.Configuration(ctx.Configuration)
    .WriteTo.Console()
    .WriteTo.File("logs/log-.txt", rollingInterval: RollingInterval.Day)
    .Enrich.FromLogContext()
    .Enrich.WithProperty("App", "Shop.Api"));
⚠️ Gotcha: Use placeholders ("User {UserId}", value), NOT interpolation ($"User {userId}"). Placeholders create searchable, structured fields.

12 Global Exception Handling

Don't try/catch in every controller. Centralize via middleware that returns standardized error responses (RFC 7807 ProblemDetails).

public class ExceptionHandlingMiddleware(RequestDelegate next, ILogger<ExceptionHandlingMiddleware> log) {
    public async Task Invoke(HttpContext ctx) {
        try {
            await next(ctx);
        } catch (NotFoundException ex) {
            await Write(ctx, 404, ex.Message);
        } catch (ValidationException ex) {
            await Write(ctx, 400, ex.Message);
        } catch (Exception ex) {
            log.LogError(ex, "Unhandled");
            await Write(ctx, 500, "Internal Server Error");
        }
    }

    private static Task Write(HttpContext ctx, int status, string msg) {
        ctx.Response.StatusCode = status;
        ctx.Response.ContentType = "application/problem+json";
        return ctx.Response.WriteAsJsonAsync(new ProblemDetails {
            Status = status, Title = msg, Type = $"https://httpstatuses.com/{status}"
        });
    }
}

app.UseMiddleware<ExceptionHandlingMiddleware>();

.NET 8+ alternative

  • Use IExceptionHandler + app.UseExceptionHandler() + AddProblemDetails() — built-in, cleaner

📅 Week 5 — Authentication & Authorization

13 JWT Authentication ⭐

JWT (JSON Web Token) is a signed, base64-encoded token containing claims (user info). Stateless — server doesn't store sessions.

Token Structure

  • Header: algorithm + type
  • Payload: claims (sub, exp, role, custom)
  • Signature: HMAC of header.payload using secret
  • Format: xxx.yyy.zzz
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(opts => {
        opts.TokenValidationParameters = new() {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer = jwt.Issuer,
            ValidAudience = jwt.Audience,
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwt.Key))
        };
    });
builder.Services.AddAuthorization();

// generate token
public string GenerateToken(User user) {
    var claims = new[] {
        new Claim(JwtRegisteredClaimNames.Sub, user.Id.ToString()),
        new Claim(ClaimTypes.Email, user.Email),
        new Claim(ClaimTypes.Role, user.Role)
    };
    var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(_jwt.Key));
    var token = new JwtSecurityToken(_jwt.Issuer, _jwt.Audience, claims,
        expires: DateTime.UtcNow.AddMinutes(_jwt.ExpiryMinutes),
        signingCredentials: new(key, SecurityAlgorithms.HmacSha256));
    return new JwtSecurityTokenHandler().WriteToken(token);
}

🎯 JWT Interview Hot Topics

  • Stateless: server doesn't track sessions — scales horizontally
  • Token sent in Authorization: Bearer xxx header
  • Refresh token pattern: short-lived JWT (15 min) + long-lived refresh token (DB-stored, revocable)
  • Storage: HttpOnly cookie (XSS-safe) vs localStorage (CSRF-safe but XSS-vulnerable)
  • Revocation: pure JWT can't revoke; combine with token blacklist or short expiry + refresh
  • Don't put sensitive data in JWT — it's only base64, anyone can decode
🔗 Angular Parallel: You'll set up a matching HTTP interceptor in Phase 5 to attach the JWT to outgoing requests and refresh on 401.

14 Role & Policy Authorization

Authentication = who you are. Authorization = what you can do. Choose between simple role checks and richer policies.

[Authorize]                                // any authenticated user
[Authorize(Roles = "Admin")]              // role check
[Authorize(Roles = "Admin,Manager")]      // any of
[AllowAnonymous]                           // override on action

// policy-based — flexible
builder.Services.AddAuthorization(opt => {
    opt.AddPolicy("Adult", p => p.RequireClaim("Age").RequireAssertion(c =>
        int.Parse(c.User.FindFirst("Age")!.Value) >= 18));
    opt.AddPolicy("AdminOrSelf", p => p.AddRequirements(new AdminOrSelfRequirement()));
});

[Authorize(Policy = "Adult")]
public IActionResult Buy() => Ok();

15 CORS — Cross-Origin Resource Sharing

Browsers block JS calls across origins by default. CORS is the server saying "this origin is allowed." Critical when Angular (localhost:4200) calls .NET (localhost:5000).

builder.Services.AddCors(o => {
    o.AddPolicy("AllowAngular", p => p
        .WithOrigins("http://localhost:4200", "https://shop.com")
        .AllowAnyHeader()
        .AllowAnyMethod()
        .AllowCredentials());
});

app.UseCors("AllowAngular");   // before UseAuthentication
⚠️ Gotcha: AllowAnyOrigin() + AllowCredentials() is forbidden by spec. Use specific WithOrigins(...) when sending cookies/auth.

📅 Week 6 — Real-World Skills

16 Pagination, Filtering, Sorting

Production APIs never return entire tables. Always page, allow filter/sort. Wrap result in a metadata envelope.

public record PagedQuery(int Page = 1, int Size = 20, string? Sort = null, string? Filter = null);
public record PagedResult<T>(IReadOnlyList<T> Items, int Total, int Page, int Size);

[HttpGet]
public async Task<PagedResult<ProductDto>> List([FromQuery] PagedQuery q) {
    var query = _db.Products.AsQueryable();
    if (!string.IsNullOrWhiteSpace(q.Filter))
        query = query.Where(p => p.Name.Contains(q.Filter));
    var total = await query.CountAsync();
    var items = await query
        .OrderBy(q.Sort ?? "Id")        // System.Linq.Dynamic.Core
        .Skip((q.Page - 1) * q.Size).Take(q.Size)
        .ProjectTo<ProductDto>(_mapper.ConfigurationProvider).ToListAsync();
    return new(items, total, q.Page, q.Size);
}

17 API Versioning & Rate Limiting

Public APIs must be versionable without breaking clients. .NET 8+ ships with built-in RateLimiter middleware.

Versioning Strategies

  • URL segment: /api/v1/products — most common
  • Query string: /api/products?api-version=1.0
  • Header: X-Api-Version: 1.0
  • Media type: Accept: application/json;v=1.0
// Asp.Versioning.Mvc package
builder.Services.AddApiVersioning(o => {
    o.DefaultApiVersion = new(1, 0);
    o.AssumeDefaultVersionWhenUnspecified = true;
    o.ReportApiVersions = true;
});

[ApiVersion("1.0")]
[ApiVersion("2.0")]
[Route("api/v{version:apiVersion}/[controller]")]
public class ProductsController { ... }

Rate Limiting (.NET 8+)

builder.Services.AddRateLimiter(o => {
    o.AddFixedWindowLimiter("perUser", opt => {
        opt.PermitLimit = 100;
        opt.Window = TimeSpan.FromMinutes(1);
    });
});

18 Background Services & Caching

For periodic / queued work, use BackgroundService. For caching, use IMemoryCache (in-process) or IDistributedCache + Redis (multi-node).

public class EmailQueueWorker(IServiceScopeFactory scopes, ILogger<EmailQueueWorker> log) : BackgroundService {
    protected override async Task ExecuteAsync(CancellationToken ct) {
        while (!ct.IsCancellationRequested) {
            using var scope = scopes.CreateScope();
            var svc = scope.ServiceProvider.GetRequiredService<IMailSvc>();
            await svc.FlushQueueAsync(ct);
            await Task.Delay(TimeSpan.FromSeconds(30), ct);
        }
    }
}
builder.Services.AddHostedService<EmailQueueWorker>();

// In-memory cache
builder.Services.AddMemoryCache();
public class ProductService(IMemoryCache cache, ...) {
    public async Task<Product> Get(int id) =>
        await cache.GetOrCreateAsync($"prod-{id}", async e => {
            e.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
            return await _db.Products.FindAsync(id);
        });
}

// Distributed cache (Redis) — multi-server
builder.Services.AddStackExchangeRedisCache(o => o.Configuration = "localhost:6379");

🎤 Interview Q&A — Phase 3

Q. Explain ASP.NET Core middleware pipeline?

Definition:

  • Middleware = chain of small components
  • Every HTTP request pass through this chain

How it works:

  • Each middleware get HttpContext (request + response)
  • Do its work, then call next() to pass to next middleware
  • After next() return — control come back up the chain
  • Like onion layers — go inward, then come back outward

Standard Order (must remember):

  1. UseRouting — endpoint matching first
  2. UseCors — CORS check
  3. UseAuthentication — WHO is the user?
  4. UseAuthorization — WHAT they can do?
  5. MapControllers — call controller

Java parallel:

  • Same like Servlet Filter chain in Java
  • Filter chain.doFilter() = .NET next()

Note: Authentication MUST come before Authorization.

Q. Singleton vs Scoped vs Transient — when to use?

Definition: Three lifetimes for services in DI container.

Compare Table:

LifetimeCreatedUse For
SingletonOnce for appConfig, cache, logger, HttpClient
ScopedOnce per requestDbContext, request-bound services
TransientEvery injectionLightweight stateless helpers

Captive Dependency Bug (interview trap):

  • Don't inject Scoped (DbContext) into Singleton
  • Scoped get captured — live forever as Singleton
  • Break request isolation, not thread-safe
  • Use ValidateScopes in dev to catch this

Java parallel:

  • Singleton = static instance / Singleton pattern
  • Scoped = per-request scope
  • Transient = new object every time
Q. What is [ApiController] attribute?

Definition:

  • Attribute on controller class
  • Enable modern Web API conventions automatically

What it does:

  • Auto model validation — invalid JSON return 400 before action run
  • Binding source inference — complex type = [FromBody], simple = [FromRoute]/[FromQuery]
  • ProblemDetails errors — RFC 7807 standard JSON format
  • Force attribute routing — no conventional routing

Use it:

  • Almost every new ASP.NET Core API controller should have it
  • Make code cleaner and consistent
Q. How does JWT authentication work end-to-end?

Definition:

  • JWT = JSON Web Token
  • Stateless way to authenticate without server session

Token Structure (3 parts separated by dot):

  • Header — algorithm info
  • Payload — claims (userId, email, role, expiry)
  • Signature — hash signed with secret key
  • Format: xxx.yyy.zzz

Flow Steps:

  1. Client send username + password to login endpoint
  2. Server validate credentials against DB
  3. Server create JWT and return
  4. Client store token (localStorage / cookie)
  5. Client send token in Authorization: Bearer <token> header
  6. Middleware verify signature + expiry + issuer + audience
  7. If valid — populate HttpContext.User with claims

Advantages:

  • Stateless — no DB lookup for session
  • Scale horizontally easily
  • Token cannot be tampered (signature check)

Disadvantage:

  • Cannot revoke token easily
  • Use short-lived JWT (15 min) + refresh token (DB stored)
Q. Why use DTOs instead of returning entities?

Definition:

  • DTO = Data Transfer Object
  • Separate class for data sent over wire
  • Different from database entity

4 Big Reasons:

  • Security — User entity has PasswordHash, don't leak it
  • Over-posting protection — client cannot send {"isAdmin": true} sneakily
  • Decoupling — DB schema can change without breaking API
  • Payload size — include only fields client need

Tools:

  • AutoMapper — map Entity to DTO automatically

Java parallel:

  • Same as Java DTO pattern — POJO for transfer
  • Java POJO = .NET POCO (Plain Old CLR Object)
Q. Authentication vs Authorization?

Definition:

  • Authentication — WHO are you? (verify identity)
  • Authorization — WHAT can you do? (check permission)

Compare Table:

AspectAuthenticationAuthorization
QuestionWho are you?What can you do?
MethodUsername/password, JWTRoles, policies, claims
Fail status401 Unauthorized403 Forbidden
MiddlewareUseAuthentication()UseAuthorization()

Real Example — Office:

  • Show ID at gate = authentication
  • Gate check if ID allow 5th floor = authorization

Note: UseAuthentication MUST come BEFORE UseAuthorization.

Q. How to handle exceptions globally in Web API?

Wrong Way:

  • try/catch in every controller action
  • Repetitive, easy to miss, clutter business code

Right Way — Custom Middleware:

  • Write ExceptionHandlingMiddleware
  • Wrap call to next() in try/catch
  • Catch unhandled exceptions
  • Log with full context (path, user, correlation ID)
  • Map exception type to HTTP status code:
    • NotFoundException → 404
    • ValidationException → 400
    • Any other → 500
  • Return ProblemDetails JSON (RFC 7807)

.NET 8 — Cleaner Way:

  • Implement IExceptionHandler
  • Register with AddExceptionHandler<T>()
  • Add app.UseExceptionHandler() in pipeline
  • Call builder.Services.AddProblemDetails()

Java parallel:

  • Same idea like Java exception filter / handler chain
  • One central place to handle all exceptions
Q. IActionResult vs ActionResult<T>?

Definition: Both are return types for controller actions.

IActionResult:

  • Base interface for all action results
  • Can return Ok(), NotFound(), BadRequest(), CreatedAtAction()
  • Flexible — return any of these
  • Downside — Swagger don't know data shape

ActionResult<T>:

  • Generic version (newer .NET)
  • Return either status code OR typed data directly (return user;)
  • Better Swagger documentation
  • Stronger typing for callers

Best Practice:

  • Use ActionResult<T> when action return data
  • Use IActionResult for actions return only status (204 NoContent on delete)
Q. What is CORS and when needed?

Definition:

  • CORS = Cross-Origin Resource Sharing
  • Browser security mechanism
  • Block JS from one origin calling API on different origin

Why exists:

  • Protect user from malicious sites calling their bank app

When needed:

  • Frontend on different origin than backend
  • Common: Angular on localhost:4200 calling .NET on localhost:5000
  • Different port = different origin = CORS check

How to configure (.NET):

  • Configure on SERVER side
  • In services: AddCors with policy
  • Specify WithOrigins, AllowAnyHeader, AllowAnyMethod
  • Optional: AllowCredentials for cookies
  • In pipeline: app.UseCors("PolicyName")

Note: Cannot combine AllowAnyOrigin() + AllowCredentials() — forbidden by spec.

Q. How to secure secrets in ASP.NET Core?

Common Mistake:

  • Storing connection strings, API keys in appsettings.json committed to source control
  • One of biggest security mistakes

Local Development — User Secrets:

  • Run dotnet user-secrets init in project
  • Run dotnet user-secrets set "Jwt:Key" "mysecret"
  • Stored in home folder, outside repo
  • Auto loaded in Development environment

Production — Azure Key Vault + Managed Identity:

  • App Service get Azure AD identity automatically
  • Grant identity Get/List permission on Key Vault
  • Read secrets via DefaultAzureCredential
  • No credentials hardcoded anywhere
  • Azure manage rotation, audit, access policies

Simpler Alternative:

  • Environment variables set on host
  • Docker secrets

Golden Rule: Search repo for "ConnectionString" or "Password" before every commit.

⚠️ Common Mistakes

  • Wrong middleware order (UseAuthorization before UseAuthentication, or both before UseRouting).
  • Returning entities directly — over-posting and sensitive data leaks.
  • Using try/catch in every controller instead of a global exception handler.
  • Storing JWT secrets / DB connection strings in appsettings.json source-controlled.
  • Injecting Scoped DbContext into a Singleton (captive dependency).
  • Calling .Result on async tasks → deadlock.
  • String concat in logs instead of structured placeholders.
  • Returning 200 for everything (errors should be 4xx/5xx).
