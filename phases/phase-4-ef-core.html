Phase 4 — Entity Framework Core

EF Core is the dominant .NET ORM. Learning it well means understanding the SQL it generates, when to use it, and when to drop to Dapper for performance.

⏱ 3 Weeks 14 Topics ⭐ Senior expectation

📋 Prerequisites (READ FIRST!)

  • ✔ Strong SQL fluency — joins, indexing, query plans (Phase 2)
  • ✔ LINQ deep dive (Phase 1) — EF Core uses LINQ to express queries
  • ✔ C# generics, async/await, IQueryable concept
  • ✔ ASP.NET Core DI (Phase 3) — DbContext is registered as Scoped

Why this matters: EF Core is a leaky abstraction. You'll write LINQ and EF will translate to SQL. Without SQL fluency, you can't debug the generated queries or fix N+1 issues.

📅 Week 1 — Basics

1 What is an ORM?

Object-Relational Mapper. Translates C# objects ↔ database rows so you write LINQ instead of raw SQL. EF Core is Microsoft's official ORM; Dapper is the lightweight competitor.

EF Core (full ORM)Dapper (micro-ORM)
SpeedSlower (overhead)Near raw ADO.NET
ProductivityHigh (LINQ, migrations)You write SQL
Learning curveHigherTiny
Use whenCRUD-heavy apps, rapid devRead-heavy hot paths, complex SQL

🎯 Why EF Core Wins for Most Apps

  • Auto-tracks changes — call SaveChanges() once
  • Migrations evolve schema versioned with code
  • Strong typing — compiler catches column rename bugs
  • LINQ unifies syntax with the rest of C#

2 DbContext & DbSet — the Heart of EF

DbContext is your unit-of-work and connection. Each DbSet<T> represents a table. You inherit from DbContext, declare DbSets, register it in DI as Scoped (one per request).

public class AppDbContext(DbContextOptions<AppDbContext> opts) : DbContext(opts) {
    public DbSet<Product> Products => Set<Product>();
    public DbSet<Order> Orders => Set<Order>();
    public DbSet<User> Users => Set<User>();

    protected override void OnModelCreating(ModelBuilder b) {
        b.Entity<Product>().HasIndex(p => p.Sku).IsUnique();
        b.Entity<Order>().Property(o => o.Total).HasPrecision(18, 2);
    }
}

// Program.cs
builder.Services.AddDbContext<AppDbContext>(o =>
    o.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

🎯 DbContext Lifecycle Rules

  • Scoped lifetime in web apps (one per HTTP request)
  • Not thread-safe — never share across threads
  • Implements IDisposable, but DI handles disposal
  • Use DbContextFactory for background services / parallel work
  • Should be light to construct (lazy init internally)

3 Code-First vs Database-First

Two ways to define the model. Code-First is the modern default for new projects.

Source of truthC# entitiesExisting DB
Schema changesMigrationsUpdate via Scaffold-DbContext

// Database-first scaffolding (legacy DB)

4 Migrations — Version-Controlled Schema

A migration captures schema diff as code. Run them in CI/CD to evolve DB safely. Always commit migrations.

dotnet tool install --global dotnet-ef

# create
dotnet ef migrations add InitialCreate
dotnet ef migrations add AddOrdersTable

# apply
dotnet ef database update                # to latest
dotnet ef database update AddOrdersTable # to specific
dotnet ef database update 0              # revert all

# inspect / script
dotnet ef migrations list
dotnet ef migrations script              # SQL for review (CI/CD use)
dotnet ef migrations remove              # undo last (before applied)

🎯 Production Migration Best Practices

  • Never edit applied migrations — create a new one to fix mistakes
  • Generate SQL scripts for prod (migrations script) and have DBA review
  • Use idempotent scripts (--idempotent flag) for safe re-runs
  • Apply migrations in a separate step from app deployment for zero-downtime
  • Watch for breaking changes (drop column, rename) — needs blue/green deploy
⚠️ Gotcha: Calling db.Database.Migrate() at app startup is OK for small apps but NOT for multi-instance prod (race condition).

5 CRUD Operations

// CREATE
var p = new Product { Name = "Pen", Price = 10 };
db.Products.Add(p);
await db.SaveChangesAsync();    // p.Id now populated

// READ
var all = await db.Products.ToListAsync();
var one = await db.Products.FindAsync(5);                 // by PK, uses cache first
var first = await db.Products.FirstOrDefaultAsync(p => p.Sku == "abc");
var exists = await db.Products.AnyAsync(p => p.Price > 100);
var count = await db.Products.CountAsync();

// UPDATE — tracked entity
var p2 = await db.Products.FindAsync(5);
p2.Price = 99;
await db.SaveChangesAsync();    // EF detects change

// UPDATE — bulk (EF Core 7+)
await db.Products
  .Where(p => p.Category == "Old")
  .ExecuteUpdateAsync(s => s.SetProperty(p => p.IsActive, false));

// DELETE
db.Products.Remove(p2);
await db.SaveChangesAsync();

// bulk delete (EF 7+)
await db.Products.Where(p => !p.IsActive).ExecuteDeleteAsync();
⚠️ Gotcha: Pre-EF 7, "bulk delete" loaded all entities, then deleted one-by-one. Always use ExecuteDelete/Update on EF 7+.

6 Data Annotations vs Fluent API

Two ways to configure mappings. Annotations are quick for simple cases; Fluent API is more powerful and keeps entities clean.

// Annotations
public class Product {
    [Key] public int Id { get; set; }
    [Required, MaxLength(100)] public string Name { get; set; } = "";
    [Column(TypeName = "decimal(18,2)")] public decimal Price { get; set; }
    [ForeignKey(nameof(Category))] public int CategoryId { get; set; }
    public Category Category { get; set; } = null!;
}

// Fluent API (preferred for complex setups)
protected override void OnModelCreating(ModelBuilder b) {
    b.Entity<Product>(e => {
        e.HasKey(p => p.Id);
        e.Property(p => p.Name).HasMaxLength(100).IsRequired();
        e.Property(p => p.Price).HasPrecision(18, 2);
        e.HasIndex(p => p.Sku).IsUnique();
        e.HasOne(p => p.Category).WithMany(c => c.Products).HasForeignKey(p => p.CategoryId);
    });
}

🎯 Why Prefer Fluent API in Big Projects

  • Keeps entity classes free of EF dependencies (clean domain layer)
  • Configures things annotations can't (composite keys, table splitting)
  • Easier to apply conventions across many entities

📅 Week 2 — Relationships & Querying

7 Relationships — 1:1, 1:Many, Many:Many

// One-to-Many: Category has many Products
public class Category {
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public ICollection<Product> Products { get; set; } = new List<Product>();
}
public class Product {
    public int Id { get; set; }
    public int CategoryId { get; set; }       // FK
    public Category Category { get; set; } = null!;  // nav
}

// One-to-One: User <-> UserProfile
public class User { public int Id { get; set; } public UserProfile? Profile { get; set; } }
public class UserProfile { public int UserId { get; set; } public User User { get; set; } = null!; }

b.Entity<UserProfile>().HasKey(p => p.UserId);
b.Entity<User>().HasOne(u => u.Profile).WithOne(p => p.User).HasForeignKey<UserProfile>(p => p.UserId);

// Many-to-Many: Student <-> Course (EF 5+ auto join table)
public class Student { public int Id { get; set; } public ICollection<Course> Courses { get; set; } = new List<Course>(); }
public class Course  { public int Id { get; set; } public ICollection<Student> Students { get; set; } = new List<Student>(); }
// EF auto-creates StudentCourse join table

Cascade Delete Behaviors

  • Cascade — delete parent → children deleted (default for required FK)
  • Restrict — block delete if children exist
  • SetNull — set FK to NULL (nullable FK)
  • NoAction — let DB decide

8 Eager / Lazy / Explicit Loading ⭐

By default, navigation properties are NOT loaded. You explicitly choose how to fetch related data — wrong choice = N+1 query disaster.

Three Strategies

  • Eager (Include) — load related entities in same query (JOIN). Recommended.
  • Lazy — load on first access (proxies). Easy but dangerous (N+1).
  • Explicit — manually load via db.Entry(x).Collection(...).LoadAsync()
// EAGER — preferred
var orders = await db.Orders
    .Include(o => o.Customer)
    .Include(o => o.Items).ThenInclude(i => i.Product)
    .Where(o => o.Total > 100)
    .ToListAsync();

// PROJECTION — fastest, fetch only needed columns
var dtos = await db.Orders
    .Where(o => o.Total > 100)
    .Select(o => new { o.Id, o.Total, CustomerName = o.Customer.Name })
    .ToListAsync();

// LAZY (avoid in web apps) — needs Microsoft.EntityFrameworkCore.Proxies
options.UseLazyLoadingProxies();
// nav properties must be virtual
public virtual Customer Customer { get; set; }

// EXPLICIT — load on demand without lazy
await db.Entry(order).Collection(o => o.Items).LoadAsync();
⚠️ Gotcha: Lazy loading + JSON serialization = infinite loops or N+1 queries hidden in API responses. Disable lazy loading by default.

9 LINQ to EF — IQueryable Magic

Same LINQ you learned in Phase 1 — but EF translates the expression tree to SQL. Stay on IQueryable until you really need the data in memory.

var result = await db.Orders
    .Where(o => o.Status == "Paid")         // SQL WHERE
    .GroupBy(o => o.CustomerId)              // SQL GROUP BY
    .Select(g => new {
        CustomerId = g.Key,
        Total = g.Sum(o => o.Total),
        Count = g.Count()
    })
    .OrderByDescending(x => x.Total)
    .Take(10)
    .ToListAsync();   // 1 SQL query — perfect

// inspect generated SQL
var q = db.Orders.Where(o => o.Total > 100);
Console.WriteLine(q.ToQueryString());   // see the SQL!

🎯 IQueryable vs IEnumerable — the EF Trap

  • db.Orders is IQueryable — operations build expression tree, executed at DB
  • Calling .AsEnumerable() or .ToList() too early pulls everything into memory and filters in C# (slow!)
  • Always filter / project on IQueryable, materialize last

10 Raw SQL & Stored Procedures

When LINQ can't express what you need (CTEs, window functions, complex hints), drop to SQL via FromSql.

// Raw SQL with parameters (always parameterize!)
var products = await db.Products
    .FromSql($"SELECT * FROM Products WHERE Price > {minPrice}")
    .ToListAsync();
// FromSql with FormattableString = parameterized SQL automatically

// stored procedure
var top = await db.Customers
    .FromSql($"EXEC sp_GetTopCustomers @Top = {n}", n: 10)
    .ToListAsync();

// non-query SQL
var rows = await db.Database.ExecuteSqlAsync($"DELETE FROM Logs WHERE CreatedAt < {cutoff}");

// SqlQuery for ad-hoc results (EF 7+)
var stats = await db.Database.SqlQuery<StatRow>($"SELECT ...").ToListAsync();
⚠️ Gotcha: NEVER use string concatenation in raw SQL — SQL injection. Always use FromSql with interpolation (it converts to parameters).

📅 Week 3 — Advanced

11 Change Tracking & AsNoTracking ⭐

By default, EF tracks every queried entity to detect modifications. For read-only queries, this is wasted work and memory — use AsNoTracking().

// Tracked (default) — slower, can update
var p = await db.Products.FirstAsync(x => x.Id == 5);
p.Price = 99;
await db.SaveChangesAsync();   // EF knows price changed

// No tracking — faster, read-only
var products = await db.Products
    .AsNoTracking()
    .Where(p => p.IsActive)
    .ToListAsync();

// global default for read-heavy app
optionsBuilder.UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking);

// inspect tracker
foreach (var entry in db.ChangeTracker.Entries()) {
    Console.WriteLine($"{entry.Entity.GetType().Name}: {entry.State}");
    // State: Added, Modified, Deleted, Unchanged, Detached
}

🎯 When to Use Each

  • Tracked: when you'll modify and save the entity
  • NoTracking: API GET endpoints, lists, anything read-only — 20-50% faster
  • AsNoTrackingWithIdentityResolution: NoTracking but de-duplicates entities by key (large joins)

12 Concurrency Control & Transactions

When two users edit the same row, you need a strategy: optimistic (last writer wins, with detection) or pessimistic (locks).

Optimistic Concurrency (recommended)

public class Product {
    public int Id { get; set; }
    public string Name { get; set; } = "";
    [Timestamp] public byte[] RowVersion { get; set; } = [];   // SQL Server
}

try {
    var p = await db.Products.FindAsync(5);
    p.Name = "Updated";
    await db.SaveChangesAsync();
} catch (DbUpdateConcurrencyException ex) {
    // row changed since you read it — handle conflict
    var entry = ex.Entries.Single();
    var dbValues = await entry.GetDatabaseValuesAsync();
    // merge or surface to user
}

Transactions

using var tx = await db.Database.BeginTransactionAsync();
try {
    db.Accounts.Find(1)!.Balance -= 100;
    db.Accounts.Find(2)!.Balance += 100;
    await db.SaveChangesAsync();
    await tx.CommitAsync();
} catch {
    await tx.RollbackAsync(); throw;
}

// SaveChanges is auto-transactional, so explicit tx only needed across multiple SaveChanges

13 Performance — N+1 & Optimization ⭐

The N+1 problem is the #1 EF performance killer and a classic interview question.

// 🚨 N+1: 1 query for orders + N queries for customers
var orders = await db.Orders.ToListAsync();
foreach (var o in orders)
    Console.WriteLine(o.Customer.Name);   // triggers N queries (lazy)

// ✅ FIX 1: eager loading
var orders = await db.Orders.Include(o => o.Customer).ToListAsync();

// ✅ FIX 2: projection (best for reads)
var dtos = await db.Orders
    .Select(o => new { o.Id, CustomerName = o.Customer.Name })
    .ToListAsync();

🎯 EF Performance Checklist

  1. Use AsNoTracking() on read-only queries
  2. Use .Select() projection — fetch only needed columns
  3. Eager-load with .Include(); avoid lazy loading
  4. Use .AsSplitQuery() for large includes (avoids cartesian explosion)
  5. Page large result sets (Skip/Take)
  6. Use ExecuteUpdate/ExecuteDelete for bulk ops (EF 7+)
  7. Use CompiledQuery for hot paths (rare)
  8. Inspect generated SQL with .ToQueryString()
  9. Add indexes on FKs and frequently-filtered columns
  10. Drop to Dapper for super-hot read endpoints
// Split query — avoids cartesian product when including multiple collections
var orders = await db.Orders
    .Include(o => o.Items)
    .Include(o => o.Payments)
    .AsSplitQuery()        // 3 separate queries
    .ToListAsync();
⚠️ Gotcha: Multiple .Include() with collections creates a Cartesian product. Use .AsSplitQuery().

14 Repository, Unit of Work & Dapper

Repository abstracts data access; Unit of Work groups changes. EF Core itself IS a UoW + repo (DbContext + DbSet) — many seniors say don't add another layer.

🎯 To Wrap or Not to Wrap (interview opinion)

  • For: testability (mock IRepository), DB swap-ability, encapsulating complex queries
  • Against: EF Core already IS a Repo + UoW; extra layer is often pure overhead and leaks IQueryable
  • Pragmatic: use Specification pattern or service classes that take DbContext directly; skip generic repos
public interface IProductRepository {
    Task<Product?> GetByIdAsync(int id);
    Task<IReadOnlyList<Product>> GetActiveAsync();
    void Add(Product p);
}

public class ProductRepository(AppDbContext db) : IProductRepository {
    public Task<Product?> GetByIdAsync(int id) => db.Products.FindAsync(id).AsTask();
    public async Task<IReadOnlyList<Product>> GetActiveAsync() =>
        await db.Products.AsNoTracking().Where(p => p.IsActive).ToListAsync();
    public void Add(Product p) => db.Products.Add(p);
}

Dapper — when EF isn't fast enough

using Dapper;
using var conn = new SqlConnection(connectionString);
var products = await conn.QueryAsync<Product>(
    "SELECT Id, Name, Price FROM Products WHERE CategoryId = @cat",
    new { cat = 5 });

Use Dapper for: complex reporting queries, hot read paths with stored procedures, or when the team is more comfortable with raw SQL.

🎤 Interview Q&A — Phase 4

Q. What is EF Core and how it relate to LINQ?

Definition:

  • EF Core = Entity Framework Core
  • Microsoft's official ORM (Object-Relational Mapper) for .NET
  • Bridge between C# objects and database tables

What it does:

  • You define class like Product — EF map it to Products table
  • You query using LINQ — EF generate SQL behind the scene
  • No need to write SQL by hand for normal CRUD

How LINQ + EF work together:

  • EF expose DbSet<T> as IQueryable<T>
  • You write db.Products.Where(p => p.Price > 100)
  • EF capture lambda as expression tree
  • EF walk tree and generate SQL WHERE clause
  • Complex queries (join, group) become single SQL statement

Other features:

  • Migrations — version your schema
  • Change tracking — know what to update
  • Connection management

Java parallel:

  • EF Core is similar to Hibernate / JPA in Java
  • Same idea — write Java/C# code, ORM generate SQL
  • EF Core LINQ = JPA Criteria API but more friendly

Note: EF is "leaky abstraction" — must understand SQL it generates.

Q. Explain N+1 query problem and how to fix?

Definition:

  • N+1 = most common EF performance bug
  • 1 query for parent + N queries for navigation property
  • Total = 1 + N queries

Example:

  • Load 100 orders — 1 query
  • Loop and access order.Customer.Name — 100 separate queries
  • Total = 101 queries
  • 50 requests = 5050 queries — database fall over

Fix 1 — Eager Loading (.Include()):

db.Orders.Include(o => o.Customer).ToListAsync()
  • EF do JOIN in same query
  • One SQL query for everything

Fix 2 — Projection (.Select()) — BEST:

.Select(o => new { o.Id, CustomerName = o.Customer.Name })
  • Shape query directly into DTO
  • Fetch only exact columns needed
  • Fastest option

Java parallel:

  • Same problem in Hibernate / JPA also
  • Fix with fetch join or @EntityGraph

Note: Always check generated SQL with .ToQueryString().

What's the difference between IEnumerable, IQueryable, and how does it matter for EF?

Definition:

  • IEnumerable<T> — in-memory sequence (LINQ to Objects)
  • IQueryable<T> — expression tree run on external source (database)
  • EF Core's DbSet<T> implement IQueryable<T>

Where filtering happens:

  • IQueryable — at database (only matching rows return)
  • IEnumerable — in your app memory (after fetching all)

Performance Example:

  • Good: db.Products.Where(p => p.IsActive).ToList() — SQL WHERE
  • Bad: db.Products.ToList().Where(p => p.IsActive) — load all rows first
  • Bad version on million-row table = disaster

Rule:

  • Stay on IQueryable as long as possible
  • Call .ToListAsync() at very end
  • Don't call .AsEnumerable() too early

Java parallel:

  • IEnumerable = Java Stream API (in-memory)
  • IQueryable = JPA Criteria API (translated to SQL)
Q. Tracking vs No-Tracking — when to use?

Definition:

  • EF tracks every entity returned from query by default
  • Tracking detect changes for UPDATE statements at SaveChanges

Cost of Tracking:

  • Extra memory — EF store snapshot of every entity
  • Extra CPU — EF compare snapshots to detect changes
  • Wasted for read-only queries

NoTracking:

  • Use .AsNoTracking() for read-only queries
  • Skip change tracker entirely
  • 20-50% faster, less memory
  • Best for GET endpoints, list endpoints

Example:

var products = await db.Products.AsNoTracking().ToListAsync();

Rule of thumb:

  • AsNoTracking — every GET / list endpoint
  • Tracking — when you will modify and save

Global default:

  • UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking) for read-heavy apps
Q. Eager vs Lazy vs Explicit loading?

Definition: 3 strategies EF use to fetch related data via navigation properties.

Eager Loading (BEST):

  • Use .Include() to fetch related data in same query
  • EF generate JOIN
  • Example: db.Orders.Include(o => o.Customer)
  • You control exactly what loads

Lazy Loading (AVOID in web):

  • Load related data on first access
  • Read order.Customer — EF fire query at that moment
  • Need Microsoft.EntityFrameworkCore.Proxies package
  • Navigation properties marked virtual
  • Cause N+1 problem in loops
  • Break JSON serialization (infinite recursion)

Explicit Loading:

  • Manual: db.Entry(order).Reference(o => o.Customer).LoadAsync()
  • Use when sometimes need related data, sometimes not
  • You control timing

Java parallel:

  • Same as Hibernate FetchType.EAGER vs LAZY
  • Hibernate also has lazy loading problems

Best Practice: Eager with .Include() OR .Select() projection into DTOs.

Q. How to handle concurrency in EF?

Why needed:

  • Two users edit same row at same time
  • Need to handle conflict

EF Approach — Optimistic Concurrency:

  • Assume conflicts are rare
  • Detect at save time, no DB locks
  • Much more scalable than pessimistic locking

How to setup:

  • Add RowVersion column (byte array with [Timestamp])
  • SQL Server auto-increment on every UPDATE
  • EF include in UPDATE: WHERE Id = @id AND RowVersion = @original
  • If row changed in between — UPDATE affect 0 rows
  • EF throw DbUpdateConcurrencyException

How to Handle Exception:

  • Retry operation
  • Merge conflict by re-fetching
  • Show user "modified by someone else"
  • Pick winner (last-write-wins)

Note: Pessimistic locking — use raw SQL with SELECT ... FOR UPDATE. Rarely needed.

Q. EF Core vs Dapper — when to choose?

Definition:

  • EF Core — full ORM
  • Dapper — micro-ORM, thin layer over ADO.NET

Compare Table:

AspectEF CoreDapper
TypeFull ORMMicro-ORM
Query syntaxLINQYou write SQL
Change trackingYesNo
MigrationsYesNo
SpeedSlightly slowerSame as ADO.NET
ProductivityHighLower (manual SQL)

When to use EF Core:

  • 80% of CRUD work
  • Standard write operations
  • Productivity matters more than 5% perf

When to use Dapper:

  • Hot read paths (millions of hits)
  • Heavy reporting queries
  • Complex queries with CTEs, window functions, hints
  • When LINQ cannot express the query well

Java parallel:

  • EF Core = Hibernate/JPA
  • Dapper = JDBC with light wrapper

Note: Many serious projects use BOTH — EF for writes, Dapper for read-optimized queries.

Q. Code-First vs Database-First?

Definition: 2 ways to setup relationship between C# entities and DB schema.

Code-First (Modern Default):

  • Write C# entity classes first (Product, Order, Category)
  • Define relationships with attributes or Fluent API
  • Run dotnet ef migrations add InitialCreate
  • Run dotnet ef database update
  • EF read classes and create tables
  • Every change = new migration in source control
  • Best for new projects where dev team owns schema

Database-First:

  • Database already exists (cannot change)
  • For legacy enterprise DB owned by DBA team
  • Run dotnet ef dbcontext scaffold "ConnString" Microsoft.EntityFrameworkCore.SqlServer -o Models
  • Schema change first in DB, then re-scaffold C# side

Java parallel:

  • Code-First = Hibernate hbm2ddl.auto=update
  • Database-First = Hibernate reverse engineering

Note: For 90% of new projects, choose Code-First.

Q. What does SaveChanges do internally?

Definition:

  • SaveChanges push in-memory changes to database
  • SaveChangesAsync is async version

Internal Steps:

  1. Walk every entity in DbContext's ChangeTracker
  2. Check entity State — Added, Modified, Deleted, Unchanged
  3. For Added — generate INSERT statement
  4. For Modified — generate UPDATE with only changed columns
  5. For Deleted — generate DELETE statement
  6. Open implicit transaction (whole batch atomic)
  7. Execute all statements in order
  8. Populate auto-generated keys back into entities
  9. Commit transaction
  10. Set all entities back to Unchanged

Return value:

  • Number of affected database rows
  • Check expected rows modified

If anything fails:

  • Database error, concurrency conflict, constraint violation
  • EF rollback transaction
  • Throw exception

Note: Always use SaveChangesAsync in web apps to avoid blocking threads.

Q. Should you use Repository pattern with EF?

Definition:

  • Repository pattern wrap data access in interface like IProductRepository
  • Methods: GetById, Add, Update

Argument FOR:

  • Abstract DB technology — can swap EF for Dapper later
  • Easier to unit-test (mock the interface)
  • Centralize complex queries

Argument AGAINST:

  • EF Core's DbContext IS already Unit of Work
  • EF Core's DbSet IS already a Repository
  • Extra layer duplicate EF features
  • Leak IQueryable everywhere (defeat abstraction)
  • Add boilerplate without real benefit

Modern Senior Advice:

  • Don't add generic Repository<T> just for pattern sake
  • Write service classes that depend on DbContext directly
  • Test with in-memory provider or test containers
  • For complex reusable queries — use Specification pattern

Java parallel:

  • You know POJO + Service + Repository layered architecture from Java
  • In .NET, EF DbContext often replace Repository layer
  • Service can depend on DbContext directly — fine

⚠️ Common Mistakes in EF Core

  • Forgetting AsNoTracking() on read-only queries (perf hit).
  • N+1 queries from missing .Include() or accidental lazy loading.
  • Calling .ToList() too early — pulls everything into memory.
  • Multiple .Include() with collections → Cartesian explosion (use .AsSplitQuery()).
  • String concat in FromSql — SQL injection.
  • Updating without fetching first (when you mean to track changes).
  • Sharing DbContext across threads — not thread-safe.
  • Editing applied migrations instead of creating new ones.
  • Calling db.Database.Migrate() in multi-instance prod (race).
