Phase 2 — SQL & Database Design

Strong SQL is the prerequisite to EF Core, performance tuning, and senior-level architecture. ORMs hide bad queries; you should be able to read what they generate and fix it.

⏱ 3 Weeks 14 Topics ⭐ Prerequisite for EF Core

📋 Prerequisites

  • SQL Server Express + SSMS installed (Phase 0)
  • Sample database (Northwind / AdventureWorks) — download from Microsoft
  • Basic understanding of tables, rows, columns

📅 Week 1 — Core SQL

1 DDL vs DML vs DQL vs DCL vs TCL

SQL commands fall into 5 groups. Knowing the categories helps you discuss permissions, transactions, and execution behavior with senior devs and DBAs.

CategoryCommandsPurpose
DDL Data DefinitionCREATE, ALTER, DROP, TRUNCATE, RENAMESchema (structure)
DML Data ManipulationINSERT, UPDATE, DELETE, MERGEModify data
DQL Data QuerySELECTRead data
DCL Data ControlGRANT, REVOKEPermissions
TCL Transaction ControlBEGIN, COMMIT, ROLLBACK, SAVEPOINTTransactions

🎯 Interview Trap: TRUNCATE vs DELETE vs DROP

  • DELETE — DML, row-by-row, can use WHERE, fires triggers, logged, slow, rollback-able
  • TRUNCATE — DDL, deallocates pages, no WHERE, no triggers, minimally logged, fast, resets identity
  • DROP — DDL, removes table entirely (data + structure)
-- DDL
CREATE TABLE Users (
  Id INT IDENTITY(1,1) PRIMARY KEY,
  Email NVARCHAR(150) NOT NULL UNIQUE,
  CreatedAt DATETIME2 DEFAULT GETUTCDATE()
);

ALTER TABLE Users ADD IsActive BIT DEFAULT 1;

-- DML
INSERT INTO Users (Email) VALUES ('rutik@example.com');
UPDATE Users SET IsActive = 0 WHERE Id = 1;
DELETE FROM Users WHERE Id = 1;

2 SELECT — WHERE, ORDER BY, LIMIT/TOP

SELECT is the workhorse. Learn the order in which clauses logically execute — interviewers love this question.

🎯 Logical Execution Order (NOT the writing order)

  1. FROM + JOIN
  2. WHERE
  3. GROUP BY
  4. HAVING
  5. SELECT (column projection happens here!)
  6. DISTINCT
  7. ORDER BY
  8. TOP / LIMIT / OFFSET

This is why you can't reference a SELECT alias in WHERE but can in ORDER BY.

SELECT TOP 10 Id, Name, Total
FROM Orders
WHERE Total > 100 AND Status = 'Paid'
ORDER BY Total DESC;

-- pagination (modern, SQL Server 2012+)
SELECT Id, Name FROM Products
ORDER BY Id
OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY;

3 Aggregates & GROUP BY / HAVING

Aggregates collapse rows: COUNT, SUM, AVG, MIN, MAX. GROUP BY buckets rows. HAVING filters those buckets (WHERE filters rows).

🎯 WHERE vs HAVING (most-asked SQL question)

  • WHERE filters rows BEFORE aggregation — can't reference aggregate functions
  • HAVING filters AFTER aggregation — used with GROUP BY, can reference aggregates
SELECT CustomerId, COUNT(*) AS Orders, SUM(Total) AS Revenue
FROM Orders
WHERE OrderDate >= '2025-01-01'
GROUP BY CustomerId
HAVING SUM(Total) > 10000
ORDER BY Revenue DESC;

4 Filtering — LIKE, IN, BETWEEN, NULL, CASE

Beyond =, SQL has rich filter operators. NULL handling trips up most devs — NULL ≠ NULL.

-- pattern matching
WHERE Email LIKE '%@gmail.com'
WHERE Name LIKE 'A_%'           -- A then any single char

-- set membership
WHERE Status IN ('Pending', 'Paid', 'Shipped')

-- ranges
WHERE Price BETWEEN 100 AND 500     -- inclusive both ends

-- NULL handling — common bug source
WHERE DeletedAt IS NULL             -- NEVER use = NULL
WHERE COALESCE(Phone, '') = ''     -- treat NULL as empty
SELECT ISNULL(Discount, 0) FROM Orders   -- replace NULL

-- conditional expressions
SELECT
  Name,
  CASE
    WHEN Total >= 1000 THEN 'Premium'
    WHEN Total >= 500  THEN 'Gold'
    ELSE 'Regular'
  END AS Tier
FROM Customers;
⚠️ Gotcha: NULL = NULL is unknown, not true. Always use IS NULL / IS NOT NULL.

📅 Week 2 — Joins, Subqueries, Design

5 Joins — INNER, LEFT, RIGHT, FULL, CROSS, SELF ⭐

Joins are the #1 SQL skill. EF Core can write them for you, but if you can't read the generated SQL you can't tune it.

TypeReturns
INNER JOINRows where match exists in both tables
LEFT (OUTER) JOINAll rows from left + matching from right (or NULL)
RIGHT JOINAll from right + matches from left (rarely used)
FULL OUTER JOINAll rows from both, NULL where no match
CROSS JOINCartesian product — every row × every row
SELF JOINTable joined to itself (e.g., Employee → Manager)
 
-- inner — only customers who placed orders
SELECT c.Name, o.Id, o.Total
FROM Customers c
INNER JOIN Orders o ON o.CustomerId = c.Id;

-- left — all customers, even those with no orders (NULL columns)
SELECT c.Name, COUNT(o.Id) AS OrderCount
FROM Customers c
LEFT JOIN Orders o ON o.CustomerId = c.Id
GROUP BY c.Name;

-- self join — find employees and their managers
SELECT e.Name AS Employee, m.Name AS Manager
FROM Employees e
LEFT JOIN Employees m ON m.Id = e.ManagerId;

🎯 Pro Tips

  • Always alias tables (c, o) — readability + ambiguity prevention
  • Filter in JOIN's ON clause vs WHERE: matters for OUTER joins (changes semantics)
  • EXISTS often beats JOIN+DISTINCT for "does any match exist" checks

6 Subqueries, EXISTS, CTE, Set Ops

Subqueries nest queries. CTE (WITH) makes complex queries readable. EXISTS is faster than IN for large subquery results.

-- subquery in WHERE
SELECT * FROM Products
WHERE Price > (SELECT AVG(Price) FROM Products);

-- EXISTS — semi-join, stops at first match
SELECT * FROM Customers c
WHERE EXISTS (SELECT 1 FROM Orders o WHERE o.CustomerId = c.Id);

-- CTE — readable for complex logic, can be recursive
WITH MonthlyRevenue AS (
  SELECT MONTH(OrderDate) AS M, SUM(Total) AS Rev
  FROM Orders
  GROUP BY MONTH(OrderDate)
)
SELECT * FROM MonthlyRevenue WHERE Rev > 50000;

-- recursive CTE — org hierarchy
WITH OrgTree AS (
  SELECT Id, Name, ManagerId, 0 AS Level FROM Employees WHERE ManagerId IS NULL
  UNION ALL
  SELECT e.Id, e.Name, e.ManagerId, t.Level + 1
  FROM Employees e INNER JOIN OrgTree t ON e.ManagerId = t.Id
)
SELECT * FROM OrgTree;

-- set operators
SELECT Email FROM Users
UNION       -- distinct merge
SELECT Email FROM Subscribers;

SELECT Email FROM A
INTERSECT   -- in both
SELECT Email FROM B;

SELECT Email FROM A
EXCEPT      -- in A but not B (= MINUS in Oracle)
SELECT Email FROM B;
⚠️ Gotcha: UNION removes duplicates (slow). Use UNION ALL if you don't need dedup — much faster.

7 Constraints (PK, FK, UNIQUE, CHECK, DEFAULT, NOT NULL)

Constraints enforce data integrity at the database level — the strongest guarantee, since app code can be bypassed.

  • PRIMARY KEY — uniquely identifies row, NOT NULL implicit
  • FOREIGN KEY — references another table's PK; can have ON DELETE CASCADE / SET NULL / NO ACTION
  • UNIQUE — column values must be unique (allows one NULL in SQL Server)
  • NOT NULL — column requires a value
  • DEFAULT — auto-fills when not specified
  • CHECK — custom predicate (CHECK (Age >= 18))
  • IDENTITY(1,1) — auto-incrementing column
CREATE TABLE Orders (
  Id INT IDENTITY(1,1) PRIMARY KEY,
  CustomerId INT NOT NULL,
  Total DECIMAL(10,2) NOT NULL CHECK (Total >= 0),
  Status NVARCHAR(20) NOT NULL DEFAULT 'Pending',
  CreatedAt DATETIME2 NOT NULL DEFAULT GETUTCDATE(),
  CONSTRAINT FK_Orders_Customers
    FOREIGN KEY (CustomerId) REFERENCES Customers(Id) ON DELETE CASCADE,
  CONSTRAINT UQ_Orders_TrackingNumber UNIQUE (TrackingNumber)
);

8 Normalization (1NF → 3NF) ⭐

Splitting data into related tables to eliminate redundancy and update anomalies. Senior interviews always ask: "explain 1NF/2NF/3NF with an example."

FormRule
1NFAtomic columns (no comma-separated lists). Unique rows.
2NF1NF + every non-key column depends on the WHOLE primary key (matters for composite keys)
3NF2NF + no transitive dependencies (non-key column shouldn't depend on another non-key column)
BCNFStricter 3NF; rare to need this in interviews

🎯 Real-World Tradeoffs

  • OLTP systems (transactions) → highly normalized (3NF) — minimize updates
  • OLAP / reporting → denormalized (star schema) — fast reads
  • Often you'll denormalize on purpose for performance — that's a senior decision
🔗 Angular Parallel: Normalization is to DB what component reuse is to UI — DRY for data, with the same maintainability benefits.

📅 Week 3 — Advanced

9 Stored Procedures & Functions

Stored Procedures (SP) are precompiled SQL routines stored in the DB. Functions return values or tables and can be used in SELECT.

CREATE PROCEDURE sp_GetTopCustomers
  @Top INT = 10,
  @MinTotal DECIMAL(10,2) = 0
AS
BEGIN
  SET NOCOUNT ON;
  SELECT TOP (@Top) c.Id, c.Name, SUM(o.Total) AS Revenue
  FROM Customers c
  INNER JOIN Orders o ON o.CustomerId = c.Id
  GROUP BY c.Id, c.Name
  HAVING SUM(o.Total) >= @MinTotal
  ORDER BY Revenue DESC;
END;

EXEC sp_GetTopCustomers @Top = 5, @MinTotal = 1000;

-- scalar function
CREATE FUNCTION fn_GetAge(@DOB DATE)
RETURNS INT
AS BEGIN
  RETURN DATEDIFF(YEAR, @DOB, GETDATE());
END;

-- table-valued function
CREATE FUNCTION fn_OrdersByCustomer(@CustomerId INT)
RETURNS TABLE
AS RETURN (SELECT * FROM Orders WHERE CustomerId = @CustomerId);

🎯 SP vs Function (interview classic)

  • SP can do INSERT/UPDATE/DELETE; function generally can't (DML restricted)
  • SP can return multiple result sets; function returns one value or table
  • Function can be used in SELECT/WHERE; SP cannot
  • SP can use TRY/CATCH and transactions; functions can't

10 Views & Indexed (Materialized) Views

A View is a saved SELECT — virtual table for abstraction, security (hide columns), and reuse.

CREATE VIEW vw_ActiveCustomers AS
SELECT Id, Name, Email
FROM Customers
WHERE IsActive = 1;

SELECT * FROM vw_ActiveCustomers WHERE Email LIKE '%@gmail.com';

-- indexed view (SQL Server) — physically stored, must be schema-bound
CREATE VIEW vw_OrderTotals
WITH SCHEMABINDING AS
SELECT CustomerId, SUM(Total) AS Total, COUNT_BIG(*) AS Cnt
FROM dbo.Orders
GROUP BY CustomerId;

CREATE UNIQUE CLUSTERED INDEX IX_vw_OrderTotals ON vw_OrderTotals(CustomerId);

11 Indexes — Clustered, Non-Clustered, Covering ⭐

Indexes are the #1 performance lever. Every interview will ask "what's a clustered index?" — know this cold.

TypeBehaviorPer Table
ClusteredDefines physical row order; PK is clustered by defaultOnly ONE
Non-ClusteredSeparate B-tree pointing to row; row data lives elsewhereMany
UniqueAdds uniqueness constraintMany
FilteredIndexes only rows matching predicate (smaller, faster)Many
CompositeMultiple columns; order mattersMany
Covering (INCLUDE)Includes extra cols so query needs only the indexMany
 
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId
ON Orders(CustomerId)
INCLUDE (Total, Status);   -- covering index

-- filtered index — only active
CREATE INDEX IX_Users_ActiveEmail
ON Users(Email) WHERE IsActive = 1;

🎯 Index Tradeoffs

  • Speed up SELECTs but slow down INSERT/UPDATE/DELETE (must update index too)
  • Take disk space
  • Index on columns used in WHERE, JOIN ON, ORDER BY
  • Don't index low-cardinality columns (e.g., gender) — full scan often faster
  • Composite index (A, B) helps queries filtering by A or A+B, NOT B alone
⚠️ Gotcha: Functions on indexed columns (WHERE YEAR(CreatedAt) = 2025) defeat indexes. Rewrite as WHERE CreatedAt >= '2025-01-01' AND CreatedAt < '2026-01-01'.

12 Transactions & ACID ⭐

A transaction = unit of work: all-or-nothing. ACID guarantees (Atomicity, Consistency, Isolation, Durability) are the bedrock of relational databases.

LetterMeaning
AtomicityAll operations succeed or none do (rollback)
ConsistencyDB stays in valid state (constraints upheld)
IsolationConcurrent tx don't interfere (controlled by isolation level)
DurabilityCommitted data survives crashes (written to disk)

🎯 Isolation Levels (low → high)

  • Read Uncommitted — sees uncommitted data (dirty reads)
  • Read Committed — default; only sees committed (avoids dirty reads)
  • Repeatable Read — same query returns same result within tx
  • Serializable — fully isolated; slowest, most locking
  • Snapshot — uses row versioning, no readers block writers
BEGIN TRANSACTION;
BEGIN TRY
  UPDATE Accounts SET Balance = Balance - 1000 WHERE Id = 1;
  UPDATE Accounts SET Balance = Balance + 1000 WHERE Id = 2;
  COMMIT;
END TRY
BEGIN CATCH
  ROLLBACK;
  THROW;
END CATCH;

🎯 Common Anomalies

  • Dirty Read: reading uncommitted change that gets rolled back
  • Non-repeatable Read: same row read twice has different value
  • Phantom Read: same query returns extra rows on re-execute

13 Window Functions ⭐ (senior-level)

Aggregate-like functions that DON'T collapse rows. Game-changer for ranking, running totals, and "top N per group" queries.

-- ROW_NUMBER, RANK, DENSE_RANK
SELECT
  Name,
  Department,
  Salary,
  ROW_NUMBER() OVER (PARTITION BY Department ORDER BY Salary DESC) AS RowNum,
  RANK()       OVER (PARTITION BY Department ORDER BY Salary DESC) AS Rnk,
  DENSE_RANK() OVER (PARTITION BY Department ORDER BY Salary DESC) AS DenseRnk
FROM Employees;

-- running total
SELECT
  OrderDate,
  Total,
  SUM(Total) OVER (ORDER BY OrderDate) AS RunningTotal
FROM Orders;

-- "top 3 per category" — interview classic
WITH Ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY Category ORDER BY Sales DESC) AS rn
  FROM Products
)
SELECT * FROM Ranked WHERE rn <= 3;

-- LAG / LEAD — compare with previous/next row
SELECT
  Month,
  Revenue,
  LAG(Revenue, 1) OVER (ORDER BY Month) AS PrevMonth,
  Revenue - LAG(Revenue, 1) OVER (ORDER BY Month) AS Growth
FROM MonthlyRevenue;

🎯 RANK vs DENSE_RANK vs ROW_NUMBER

  • ROW_NUMBER: 1, 2, 3, 4 — always unique
  • RANK: 1, 2, 2, 4 — ties get same rank, gap after
  • DENSE_RANK: 1, 2, 2, 3 — ties get same rank, no gap

14 Query Optimization & Execution Plans

When queries get slow, you read the execution plan, identify bottlenecks (table scans, key lookups, sorts) and fix with indexes or query rewrites.

🎯 Diagnostic Tools

  • SSMS: Ctrl+M (Include Actual Execution Plan), Ctrl+L (Estimated)
  • SET STATISTICS IO ON; shows reads/writes per operation
  • SET STATISTICS TIME ON; shows CPU/elapsed time
  • SQL Server Profiler / Extended Events for live tracing
  • Azure Data Studio + Query Plan Viewer extension

🎯 Optimization Checklist

  1. Add missing indexes (SSMS suggests them)
  2. Avoid SELECT * — fetch only needed columns
  3. Filter early, narrow before joining
  4. Use EXISTS instead of IN (subquery) for large sets
  5. Avoid functions on indexed columns in WHERE
  6. Replace cursors with set-based operations
  7. Use covering indexes (INCLUDE) for hot reads
  8. Use UNION ALL instead of UNION if duplicates are fine
  9. Watch for parameter sniffing issues — use OPTION (RECOMPILE) if needed
  10. For pagination, use keyset (seek) pagination over OFFSET for large tables
-- bad: function on column — index not used
SELECT * FROM Orders WHERE YEAR(CreatedAt) = 2025;

-- good: range — index used
SELECT * FROM Orders
WHERE CreatedAt >= '2025-01-01' AND CreatedAt < '2026-01-01';

-- bad: SELECT * + no covering index → key lookups
SELECT * FROM Orders WHERE CustomerId = 123;

-- good: project only what you need
SELECT Id, Total FROM Orders WHERE CustomerId = 123;
⚠️ Gotcha: Don't optimize blindly. Measure first — execution plans are your truth, intuition lies.

🎤 Interview Q&A — Phase 2

Q. What is difference between WHERE and HAVING?

Definition:

  • WHERE — filter rows BEFORE grouping
  • HAVING — filter groups AFTER grouping

Key Points:

  • WHERE cannot use aggregate functions (SUM, COUNT) — groups not exist yet
  • HAVING can use aggregate functions — work on groups
  • HAVING used with GROUP BY mostly

Example — Orders table:

  • WHERE Status = 'Paid' — pick only paid rows first
  • GROUP BY CustomerId — bucket them
  • HAVING SUM(Total) > 10000 — keep only customers with spend over 10000

Note: WHERE for row, HAVING for group. Easy to remember.

Q. Clustered vs Non-Clustered Index?

Definition:

  • Index = data structure to find rows fast (like book index)
  • Without index = scan whole table O(n)
  • With index = find in O(log n)

Clustered Index:

  • Define physical order of rows on disk
  • Only ONE per table
  • By default Primary Key become clustered index
  • Rows physically sorted by PK

Non-Clustered Index:

  • Separate structure (B-tree) stored apart from table
  • Contain indexed column + pointer to row
  • Many per table allowed
  • May need "key lookup" to fetch full row (slower)

Covering Index:

  • Add extra columns with INCLUDE keyword
  • Query get everything from index — no table touch

Cost:

  • Take disk space
  • Slow down INSERT/UPDATE/DELETE — index also need update

Rule: Index columns used in WHERE, JOIN ON, ORDER BY.

Q. What is normalization? When to denormalize?

Definition:

  • Normalization = organize tables to remove duplicate data
  • Denormalization = duplicate data on purpose for read speed

Normal Forms:

  • 1NF — each column has atomic value (no comma list)
  • 2NF — every non-key column depend on WHOLE primary key
  • 3NF — no transitive dependency (non-key not depend on non-key)

3NF Example:

  • Don't store customer city in Orders table
  • Join to Customers table to get city

When to Denormalize:

  • Reporting / OLAP system where joins are slow
  • Hot read endpoint joining 6+ tables
  • Star schema in data warehouse

Note: OLTP — aim for 3NF. OLAP — denormalize for speed.

Q. Explain ACID?

Definition: ACID = 4 guarantees databases give for transactions.

A — Atomicity:

  • All-or-nothing
  • Money transfer: both updates happen or none
  • If fail — rollback

C — Consistency:

  • DB stay in valid state
  • All constraints (FK, unique, check) upheld

I — Isolation:

  • Concurrent transactions don't mess each other
  • Read Committed (default) — prevent dirty read
  • Serializable — strongest, slowest

D — Durability:

  • Once committed — data on disk
  • Survive power cut or crash

Note: NoSQL use BASE model — trade ACID for scale.

Q. TRUNCATE vs DELETE?

Definition:

  • DELETE — DML, remove rows one by one
  • TRUNCATE — DDL, deallocate pages

Compare Table:

AspectDELETETRUNCATE
TypeDMLDDL
WHEREYesNo
TriggerYesNo
LoggingFull per rowMinimal
SpeedSlowFast
IDENTITYKeep growingReset to seed

When to use:

  • DELETE — when filter needed or partial delete
  • TRUNCATE — wipe whole table fast (test data, reset)

Note: DROP is different — remove table itself.

Q. What is CTE and when useful?

Definition:

  • CTE = Common Table Expression
  • Named temporary result set with WITH keyword
  • Like temporary view for next statement only

Why use CTE:

  • Readability — complex query become like recipe
  • Chain multiple CTEs with comma
  • Recursion — CTE can reference itself
  • Standard for org chart, category tree, folder hierarchy

vs Temporary Table:

  • CTE not write to disk
  • CTE auto cleaned — scope to one statement
  • No cleanup needed
Q. RANK vs DENSE_RANK vs ROW_NUMBER?

Definition:

  • All 3 are window functions in OVER (ORDER BY ...)
  • Difference is how they handle ties

Output Compare (with ties at value 2):

FunctionOutputBehavior
ROW_NUMBER1, 2, 3, 4, 5Always unique number
RANK1, 2, 2, 4, 5Same for ties, SKIP next
DENSE_RANK1, 2, 2, 3, 4Same for ties, NO skip

When to use:

  • ROW_NUMBER — pagination, "first" record per group
  • RANK — competition rank where ties skip
  • DENSE_RANK — "top 3 salaries" — give all tied at top
Q. How to find second-highest salary?

Method 1 — Subquery:

SELECT MAX(Salary) FROM Employees
WHERE Salary < (SELECT MAX(Salary) FROM Employees);
  • Find max of salaries less than highest
  • Simple and works fine

Method 2 — Window Function (Modern):

SELECT Salary FROM (
  SELECT Salary, DENSE_RANK() OVER (ORDER BY Salary DESC) AS rk
  FROM Employees
) t WHERE rk = 2;
  • DENSE_RANK assign rank 1 to highest, rank 2 to next distinct
  • Filter where rk = 2
  • Use DENSE_RANK so ties at top correctly handled

Wrong way — Don't use:

  • ORDER BY Salary DESC OFFSET 1 LIMIT 1
  • Give second ROW not second distinct salary
  • Wrong if multiple people share top salary
Q. What is deadlock in SQL Server?

Definition:

  • Two transactions each hold locks the other need
  • Neither can move forward

Example:

  • Transaction A — lock Orders, want Customers
  • Transaction B — lock Customers, want Orders
  • Both wait forever

How SQL Server handle:

  • Deadlock monitor run periodically
  • Pick cheaper transaction to kill (deadlock victim)
  • Rollback victim, let other proceed
  • App get error code 1205 — should retry

How to Prevent:

  • Always access tables in same order in all code
  • Keep transactions short — no user think time inside
  • Use Read Committed Snapshot Isolation
  • Add proper indexes — queries lock fewer rows
  • Add retry logic with exponential backoff

Java parallel:

  • Same as Java thread deadlock — same prevention rule
  • Take locks in same order everywhere
Q. How does indexing work and what is cost?

Definition: Index = separate B-tree to find rows fast.

How it Help:

  • Without index — scan every row O(n)
  • With index — find in O(log n)
  • On 1 million rows: half second vs few microseconds

Cost 1 — Disk Space:

  • Each index is own structure on disk
  • Big tables can have indexes bigger than table itself

Cost 2 — Slower Writes:

  • Every INSERT/UPDATE/DELETE update table AND every index
  • 5 indexes on write-heavy table = much slower inserts

Rules:

  • Index columns used in WHERE, JOIN ON, ORDER BY
  • Don't index columns that change a lot
  • Don't index low cardinality columns (Gender — only 2-3 values)
  • Check execution plan to see which indexes used

⚠️ Common Mistakes in SQL

  • Using = NULL instead of IS NULL
  • SELECT * in production code (defeats covering indexes, fetches unused data)
  • Functions on indexed columns in WHERE clause
  • UNION when UNION ALL would do (slower needlessly)
  • Putting filter in WHERE instead of JOIN's ON for outer joins (changes semantics)
  • Cursors / loops where set-based operations work
  • Trusting "the ORM will handle it" — always inspect generated SQL
  • Missing transactions on multi-step operations (data corruption risk)
